## [001.Introducing Prompt Optimization in Preview in Amazon Bedrock](https://aws.amazon.com/about-aws/whats-new/2024/11/prompt-optimization-preview-amazon-bedrock/)

- 機能概要: Amazon BedrockにおけるPrompt Optimization機能がプレビューとして発表されました。この機能は、基盤モデルからより高品質な応答を得るために、プロンプトを再構成することを目的としています。開発者は、プロンプトの最適化を行い、異なる基盤モデルにおけるパフォーマンスを比較することができます。また、最適化されたプロンプトは「Prompt Builder」として保存され、生成AIアプリケーションで利用可能です。
- アップデート日付: 2024/11/21
- 特徴:
  - プロンプトの最適化により、より高品質な応答を生成。
  - 最適化前後のパフォーマンスを簡単に比較できる。
  - 対応モデル: Claude Sonnet 3.5、Claude Sonnet、Claude Opus、Claude Haiku、Llama 3 70B、Llama 3.1 70B、Mistral Large 2、Titan Text Premier。
  - 最適化されたプロンプトはPrompt Builder内で保存され、再利用可能。
- 利用ケース:
  - 開発者が異なる基盤モデルで生成される応答の品質を向上させるためにプロンプトの最適化を行うケース。
  - 生成AIアプリケーションにおいて、プロンプトのパフォーマンスを比較・最適化する場面。
- 注意事項:
  - プロンプト最適化はプレビュー段階であり、今後のアップデートで機能の改善が行われる可能性がある。
  - 現時点では、デプロイなしで最適化されたプロンプトを比較することができる。
- 対応リージョン: 東京リージョン対応なし
- 簡単な利用手順の概要:
  - 手順1: Amazon Bedrockのコンソールにアクセス。
  - 手順2: Prompt Builder内でプロンプトを作成または選択。
  - 手順3: 最適化機能を使用してプロンプトを最適化し、パフォーマンスを比較。

## [002.Replicate changes from databases to Apache Iceberg tables using Amazon Data Firehose (in preview)](https://aws.amazon.com/jp/blogs/aws/replicate-changes-from-databases-to-apache-iceberg-tables-using-amazon-data-firehose/)

- 機能概要: Amazon Data Firehoseがデータベース（PostgreSQLやMySQL）の変更データをキャプチャし、これをApache Iceberg形式でAmazon S3にレプリケートする機能がプレビュー提供されています。この機能により、データベースのトランザクション性能に影響を与えることなく、データの変更をリアルタイムでストリーミングし、大規模な分析や機械学習に利用することができます。
- アップデート日付: 2024/11/15
- 特徴:
  - Amazon Data Firehoseが、データベースの変更をApache Icebergテーブルに自動でレプリケートします。
  - 従来のETL手法に比べ、リアルタイムでデータの更新を反映できるため、データ分析やMLアプリケーションで最新データを活用可能。
  - MySQLやPostgreSQLなどのデータベースに対応。今後、SQL ServerやMongoDBにも対応予定。
  - 自動でApache Icebergテーブルのスキーマを作成および進化させる機能が含まれています。
- 利用ケース:
  - データベースの変更をリアルタイムでストリーミングし、Amazon S3上のIcebergテーブルに反映させることで、最新データを用いた大規模な分析や機械学習の実行。
  - 複数のデータベースのデータを集約し、分析基盤で使用する際に最適。
- 注意事項:
  - 現在のプレビュー版では、全てのAWSリージョンで利用可能ではなく、アジア太平洋（マレーシア）など一部リージョンでは利用不可。
  - プレビュー期間中は無料で提供されており、今後利用量に基づく課金が行われる予定。
- 対応リージョン: 東京リージョンを含む、全てのAWSリージョンに対応予定（中国リージョン、AWS GovCloud（US）リージョン、アジア太平洋（マレーシア）リージョン除く）。
- 簡単な利用手順の概要
  - 手順1: AWS Management Consoleを開き、Data Firehoseセクションで新しいストリームを作成。
  - 手順2: データベース（MySQLやPostgreSQL）の情報を入力し、接続設定を行う。
  - 手順3: 対象となるデータベースのテーブルと列を指定し、S3バケットでIcebergテーブルを作成。
  - 手順4: ストリームがデータのキャプチャを開始し、変更が反映されることを確認。

## [003.Build cost-effective RAG applications with Binary Embeddings in Amazon Titan Text Embeddings V2, Amazon OpenSearch Serverless, and Amazon Bedrock Knowledge Bases](https://aws.amazon.com/blogs/ai/build-cost-effective-rag-applications-with-binary-embeddings-in-amazon-titan-text-embeddings-v2-amazon-opensearch-serverless-and-amazon-bedrock-knowledge-bases/)

- 機能概要: Amazon Titan Text Embeddings V2では、バイナリ埋め込みをサポートし、これをAmazon Bedrock Knowledge BasesおよびAmazon OpenSearch Serverlessと統合することで、コスト効率の良いRAG（Retrieval Augmented Generation）アプリケーションの構築が可能になります。バイナリ埋め込みは、ストレージと計算の効率性を高め、メモリ使用量とコストを削減します。  
- アップデート日付: 2024/11/18  
- 特徴:
  - バイナリ埋め込みによって、Amazon Titan Text Embeddings V2が高次元データを効率的に格納および計算できるフォーマットで表現します。
  - フロート32（FP32）埋め込みと比較して、バイナリ埋め込みはストレージの削減と検索パフォーマンスの向上を実現します。
  - サポートされているサービスには、Amazon Bedrock Knowledge BasesおよびAmazon OpenSearch Serverlessがあります。
  - オプションで、バイナリ埋め込みだけでなく、フロート埋め込み（FP16/FP32）を同時にリクエストできます。  
- 利用ケース:
  - 大規模なテキストデータを効率的に検索・解析し、RAGを利用した生成的AIアプリケーションを構築するケース。特に、低コストで高効率な検索とデータ取得を実現する用途。  
- 注意事項:
  - バイナリ埋め込みを使用すると、精度は若干低下する可能性がありますが、検索速度とコストの削減に対する影響は最小限です。
- 対応リージョン: 東京リージョンでの対応あり。  
- 簡単な利用手順の概要  
  - 手順1: Amazon Bedrockコンソールで知識ベースを作成し、S3データソースを指定。
  - 手順2: Titan Text Embedding v2を選択し、バイナリベクトル埋め込みを指定。
  - 手順3: Amazon OpenSearch Serverlessを設定し、バイナリデータタイプをサポートする新しいベクトルストアを作成。

### 専門用語:

| カテゴリー                     | 専門用語               | 説明                                                         |
|------------------------------|----------------------|------------------------------------------------------------|
| 埋め込み技術                   | バイナリ埋め込み          | データをバイナリ形式（0と1）で表現し、効率的な格納と計算を可能にする手法。              |
| 検索アルゴリズム               | HNSW                   | ヒエラルキー型ナビゲート可能小世界（HNSW）アルゴリズム、近似最近傍検索に利用される。  |

## [004.Split cost allocation data for Amazon EKS now supports metrics from Amazon CloudWatch Container Insights](https://aws.amazon.com/about-aws/whats-new/2024/11/split-cost-allocation-data-amazon-eks-metrics-amazon-cloudwatch-container-insights/?nc1=h_ls)

- 機能概要: 本機能では、Amazon Elastic Kubernetes Service (EKS) クラスターで使用されているコンテナのコストを、Amazon CloudWatch Container Insightsによって収集されたCPUおよびメモリのメトリクスを基に、より詳細なKubernetesポッド単位で割り当てられるようになります。この機能により、EC2インスタンスを共有する複数のアプリケーションコンテナが動作するEKSクラスターのコスト可視化が向上し、コスト配分の精度が改善されます。
- アップデート日付: 2024/11/15
- 特徴:
  - Amazon CloudWatch Container Insightsで収集されたCPUおよびメモリのメトリクスを基に、EKSクラスターのコストをポッド単位で割り当てることができます。
  - 複数のアプリケーションコンテナをEC2インスタンスで共有しているEKSクラスターのコスト可視化が向上します。
  - AWS Cost and Usage Reports (CUR)で、EKSクラスターのポッド単位の使用データを確認することが可能になります。
  - オプション機能として、CloudWatch Observability EKSアドオンまたはHelmチャートを利用してCloudWatchエージェントおよびFluent-bitエージェントをEKSクラスターにインストールすることができます。
- 利用ケース:
  - EKSクラスターにおける複数のアプリケーションコンテナのコストを、より詳細な単位で把握したい場合。
  - EC2インスタンスを共有するEKSクラスターのコスト配分を精緻化したい場合。
- 注意事項:
  - この機能を利用するには、CloudWatch Container InsightsとEnhanced Observabilityの有効化が必要です。
  - ポッドレベルの使用データがCURに反映されるまでに最大24時間かかることがあります。
- 対応リージョン: 東京リージョンを含む全AWSリージョンで利用可能です。
- 簡単な利用手順の概要
  - 手順1: Amazon CloudWatch Observability EKSアドオンまたはHelmチャートを利用して、CloudWatchエージェントとFluent-bitエージェントをEKSクラスターにインストールする。
  - 手順2: AWS Billing and Cost Managementコンソールで、EKSの分割コスト配分データを有効にし、Amazon CloudWatchをメトリクスのソースとして選択する。

## [005.Amazon OpenSearch Serverless Includes SQL API Support](https://aws.amazon.com/blogs/analytics/amazon-opensearch-serverless-includes-sql-api-support/)

- 機能概要: Amazon OpenSearch Serverlessは、インフラ管理を気にすることなく検索と分析のワークロードを実行できるサーバーレスオプションで、SQLとPPL (Piped Processing Language) APIをサポートするようになりました。これにより、ユーザーはREST API、JDBC、CLIを通じてデータのクエリを実行でき、従来のリレーショナルデータベースからOpenSearch Serverlessに移行する企業にとって便利な機能となります。
- アップデート日付: 2024/11/20
- 特徴:
  - OpenSearch SQLとPPLのクエリをサポートし、既存のSQLスキルやツールを活用できる。
  - AWS CLIを使ってSQLクエリを直接ターミナルから実行でき、JDBCドライバーを用いてBIツールに接続可能。
  - SQLおよびPPLクエリをJavaアプリケーションに統合でき、従来のリレーショナルデータベースからの移行をサポート。
  - サーバーレス環境での検索・分析ワークロードにおける簡便さを提供。
- 利用ケース:
  - 従来のリレーショナルデータベースからOpenSearch Serverlessへの移行を進める企業や、SQLを用いて既存のデータを分析するデータアナリスト、開発者にとって便利な機能。
  - 分析ワークフローの効率化やビジネスインテリジェンスツールとの連携を強化する用途に最適。
- 注意事項:
  - 現在サポートされているリージョンに制限があり、使用する前にリージョンの確認が必要。
- 対応リージョン: 東京リージョン (Asia Pacific Tokyo) を含む15のリージョンで利用可能。
- 簡単な利用手順の概要
  - 手順1: AWS CLIをインストールし、SQLクエリを実行する。
  - 手順2: ビジネスインテリジェンスツールにJDBCドライバーを接続して、OpenSearchデータにアクセスする。
  - 手順3: JavaアプリケーションにSQL/PPLクエリを組み込んでデータ分析を行う。

## [006.Amazon OpenSearch Serverless now supports point in time (PIT) search](https://aws.amazon.com/jp/about-aws/whats-new/2024/11/amazon-opensearch-serverless-pit-search/)

- 機能概要: Amazon OpenSearch Serverlessは、データセットの特定の瞬間に基づいた複数のクエリ実行をサポートする「Point in Time (PIT) 検索」機能を追加しました。この機能により、データが更新されても一貫した検索結果を維持できるようになります。特に、深いページネーションを必要とするアプリケーションや、複数のクエリに渡って安定したデータビューを保持する必要があるアプリケーションに有用です。
- アップデート日付: 2024/11/19
- 特徴:
  - Point in Time (PIT) 検索により、データが変化し続けている間でも一貫性のある検索結果を保持できます。
  - PIT検索では、検索結果を前後にナビゲートすることができ、データのインジェストが進行中でも整合性を保ちます。
  - eコマースアプリケーション、コンテンツ管理システム、分析プラットフォームに最適です。
  - Amazon OpenSearch Serverlessは、現在15のリージョンで利用可能です。
- 利用ケース:
  - eコマースアプリケーションやコンテンツ管理システム、分析プラットフォームなど、安定した検索結果を求める大規模なデータセットを扱うシステムでの利用。
- 注意事項:
  - PIT検索は、データのインジェスト中でも一貫した結果を提供しますが、データの変更が反映されない点を留意する必要があります。
- 対応リージョン: 東京リージョン（Asia Pacific (Tokyo)）に対応。
- 簡単な利用手順の概要:
  - 手順1: OpenSearch ServerlessのPIT検索機能を有効にする。
  - 手順2: PIT検索を実行したいデータセットを選択する。
  - 手順3: クエリをPIT検索モードで実行する。

- 機能概要: Amazon SageMakerが、NVIDIAのL40S Tensor Core GPUを搭載したG6eインスタンスのサポートを開始しました。これにより、開発者はコスト効率の良い高性能なGPUインスタンスを利用して、さまざまな生成AIモデルを推論することが可能になりました。
- アップデート日付: 2024/11/22
- 特徴:
  - G6eインスタンスは、最大384GBのGPUメモリを搭載し、大規模言語モデルを単一GPUノードで処理する能力を提供します。
  - G5インスタンスと比較して、G6eインスタンスは高いパフォーマンスを提供し、コスト効率が向上しています。
  - サポートするモデルには、Llama 3.2 11B VisionやLlama 2 13B、Qwen 2.5 14Bなどが含まれます。
  - 最大8GPUを持つインスタンスで、モデルサイズが最大90Bパラメータまで対応可能です。
- 利用ケース:
  - チャットボットや会話型AI
  - テキスト生成や要約
  - 画像生成や視覚モデル
- 注意事項:
  - G6eインスタンスは、コスト効率とパフォーマンスを重視する高スループットな推論ワークロード向けに最適化されていますが、大規模なモデルを扱う場合はインスタンスの選定に注意が必要です。
- 対応リージョン: 東京リージョンに対応
- 簡単な利用手順の概要:
  - 手順1: AWSアカウントとIAMロールを準備し、Amazon SageMaker StudioでG6eインスタンスを使ってモデルをデプロイします。
