## この記事について

以前、「[Amazon Bedrock 専用 LLM Proxy を作った](https://zenn.dev/tosshi/articles/21df66f8c90e9f)」という記事で、自作の OSS **Stratoclave**（ストラトクレイブ）を紹介しました。テナント分離・2 階層クレジット・RBAC・Claude Code 統合あたりを解説した記事です。

あれから、Stratoclave は「Bedrock 課金を管理する Proxy」から一歩進んで、**汎用の AI ゲートウェイ**に育ちました。この記事では、その後に入った新機能を、**前提知識なしで読める粒度**で噛み砕いて解説します。

前回すでに解説した機能（テナント分離、クレジット管理の基本、認証まわり、STS Vouch など）は繰り返しません。今回の主役は次の5つです。

- **原子的な予算執行** — 予算超過を「あとで気づく」のではなく「その場で止める」
- **ハイブリッドサービング** — Bedrock でも自前 GPU でも、同じ台帳に載せる
- **判断・実行・課金の信頼境界** — 「どのモデルを使うか」の判断を、実行・課金から切り離す。判断そのものは、既製の [vLLM Semantic Router](https://github.com/vllm-project/semantic-router) に委ねる方針です
- **セッション対応ルーティング（SAAR）** — エージェントの会話を「同じモデルに貼り付ける」。これも本来は vLLM Semantic Router が内包する考え方です
- **Savings Certificate（節約証明書）** — 「いくら安くできたか」を課金台帳と突き合わせて正直に出す

全体像はこんな構成です。クライアント（Claude Code / CLI / 各種 API）からのリクエストが、CloudFront → ALB を通って単一の ECS Fargate バックエンドに入り、RBAC・クレジット台帳・ルーティング・監査を通ってから Bedrock（あるいは自前サービング）に流れます。データストアは DynamoDB 1本です。

![Stratoclave アーキテクチャ](/images/stratoclave-ai-gateway-update/architecture.png)

## なぜ「AI ゲートウェイ」が要るのか

機能の話に入る前に、「そもそも誰が嬉しいのか」を先に共有させてください。LLM をアプリに組み込むと、ある共通の悩みが必ず出てきます。それは:

> **「誰が・どのくらい・いくら分」使ったのかを、後から「使っていない」と言い逃れできない形で正確に記録し、使いすぎを未然に止めたい**

これは1つの業種の話ではありません。代表的なものを3つ挙げます。

**1. 顧客単位のコスト分離と請求根拠（SaaS・受託）**
複数の顧客（テナント）に AI 機能を提供すると、推論コストは「変動費」としてそのまま原価にのしかかります。従来型 SaaS の高い粗利率と違い、AI 機能は使われるほどコストが増える。だからこそ「テナントごとに、いくら使ったか」を正確に測り、上限を設け、請求の根拠（チャージバック）にできる仕組みが要ります。受託・エージェンシーで「案件ごとにコストを分けて請求根拠を残したい」のも、同じ構図です。

**2. 社内プラットフォームとしての予算配布**
「部署ごと・チームごとに Bedrock の予算を配り、暴走を止め、あとで費用を按分したい」。情シスやプラットフォームチームの定番ニーズです。誰かのスクリプトが暴走して 1 時間で予算を溶かす、を仕組みとして防ぎたい。

**3. 台帳がそのまま実験計測基盤になる**
これは他ツールにない使い方です。**テナントを「実験の単位」として切れば**、Stratoclave の台帳がそのまま「実験ごとのコスト・スループット・精度を突き合わせる計測基盤」になります。モデル A と B を別テナントで走らせれば、後から「どっちが安くて速くて良かったか」を実測値で比較できる。プロンプト改善の効果測定にも使えます。

（なお、個人利用でも「気づいたら Bedrock の請求が跳ねていた」を防げる、規模を問わない話です。）

この3つに共通する核心を、私は次のように捉えています。

> **「金になるイベントを、信頼境界を越えて、一度だけ・正しい相手に・否認できない形で記録する」**

（「否認できない形」＝あとから「そんなの使ってない」と言い逃れできない記録、という意味です。会計や監査でよく求められる性質です。）

これは実は、分散システムで最も難しい問題のひとつです。LLM・ツール・エージェントを跨いで処理が流れる中で、課金という「お金が動くイベント」を、二重にも取りこぼしもせず、正確な相手に紐づけて記録しなければいけない。Stratoclave の新機能は、すべてこの一点を軸に設計されています。

## 1. 原子的な予算執行 — 「あとで気づく」を「その場で止める」に

### よくある予算管理の落とし穴

多くの LLM Proxy は、コストを「カウンター」で管理します。リクエストが来るたびに使った分を足し込み、上限を超えたら止める。一見よさそうですが、落とし穴があります。

**リクエストが同時に大量に来ると、上限を突き抜けてしまう。**

たとえば残り予算が「あと 100 円分」のとき、同時に 50 本のリクエストが飛んでくると、全部が「まだ 100 円ある」と判断して通ってしまう。カウンターを足すのは処理が終わったあと（＝事後）なので、超過に気づいたときには手遅れです。これは競合状態（レースコンディション）と呼ばれる、古典的なバグです。

### Stratoclave のやり方: 予約してから使う

Stratoclave は「事後に足す」のではなく、**リクエストを受けた瞬間に、必要な分を予約（reserve）してから実行**します。これはクレジットカードの **オーソリ（与信枠の確保）→ キャプチャ（確定）** とまったく同じ考え方で、それを LLM の課金に持ち込んだものです。使う前に枠を押さえ、実際に使った額で確定させる。決済の世界で確立された定番パターンです。

この予約に、DynamoDB の **[TransactWriteItems](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/transaction-apis.html)**（複数の書き込みを「全部成功か、全部失敗か」のどちらかにする仕組み＝トランザクション）を使っています。予約は原子的（分割不可能）に行われるので、同時に 50 本来ても枠は正確に取り合いになり、**同時アクセスによる予算の突き抜けが構造的に起きません**。

ポイントは、予約する金額を「起こり得る最大」（`max_tokens` などに基づく上限見積り）で取ることです。実際の使用量は LLM の出力が終わるまで分かりませんが、上限で先に押さえておくので、精算は常に「予約しすぎた分を返す（返金）」方向になります。この「予約 → 実行 → 精算」の流れが、次に説明する台帳の背骨になります。

**予算超過は「起きてから気づく」ものではなく、「起きる前に止める」もの**になります。暴走スクリプトが 1 時間で予算を溶かす、という事故が、仕組みとして発生しません。

（トレードオフも正直に書くと、安全側に倒すために上限見積りで予約するので、残額ギリギリのときは「実際には足りたはずのリクエスト」が保守的に拒否されることがあります。「使いすぎて事故る」より「厳しめに止まる」を選んだ設計です。）

## 2. ハイブリッドサービング — Bedrock でも自前 GPU でも、同じ台帳に

予算の話（予約 → 実行 → 精算）が出たところで、その「台帳」の話を先に片付けます。

「provider をたくさんサポートしています！」というゲートウェイは世の中にたくさんあります。100 以上の provider に対応、みたいな。Stratoclave はそこでは戦いません。**代わりに1本の「差し替え点（seam）」＝バックエンドを差し込む共通の窓口を用意しました。**

`served_by="vllm"` という設定を付けたモデルは、Bedrock ではなく**指定した OpenAI 互換エンドポイント**（＝OpenAI の API と同じ形式でやり取りできる窓口。自前の GPU で動かす vLLM など）に流れます。しかも重要なのは:

> **Bedrock でも自前サービングでも、まったく同じ「予約 → 実行 → 精算」の台帳に載る。**

![予約→実行→精算の統合台帳](/images/stratoclave-ai-gateway-update/unified-ledger.png)

つまり、自前 GPU で動かしたモデルの利用も、Bedrock とまったく同じ**仕組み**（予約 → 実行 → 精算の台帳）でコスト管理・予算執行・課金記録がされます。「自前サービングは別勘定でどんぶり」にはなりません。

ただし正直に言うと、Bedrock には「公表料金」という外部の真実がありますが、自前サービングには請求書が存在しません。台帳に載る金額は**運用者が設定した単価**であり、その単価が GPU の実コスト（償却・電気代・稼働率）を正しく反映しているかは運用者の責任です。「金額の"精度"まで Bedrock と同じ」ではなく、「載せる"仕組み"が同じ」というのが正確な表現です。

そして今の推論サービングの世界は、**API がほぼ OpenAI 互換に収束**しています（vLLM、TGI、Ollama、Together、Fireworks……）。だから「provider ごとにアダプタを 100 個書く」のではなく、「OpenAI 互換の差し替え点を1つ用意する」だけで、実質的に多くのバックエンドを同じ台帳に載せられる。**広さを"アダプタの数"ではなく"1つの差し替え点"で確保する**、という戦略です。ハイブリッドサービングは現在フィーチャーフラグで OFF のまま出荷しており（＝デフォルトでは動かない）、有効化は運用者の明示的な操作が必要です。

## 3. 信頼境界 — 「どのモデルを使うか」を、実行と課金から切り離す

### 判断・実行・課金を混ぜると危ない

LLM ゲートウェイには「賢くルーティングしたい」という欲求があります。「簡単な質問は安いモデル、難しい質問は高いモデルに振り分けたい」といった具合です。

でも、この「どのモデルを使うか」という**判断ロジック**を、実際に Bedrock を叩く**実行**や、お金を動かす**課金**と同じ場所に混ぜてしまうと、危険です。判断ロジックのバグが、そのままお金の事故や、権限の逸脱につながる。

### 判断は「隔離された助言者」に任せる

Stratoclave の設計の背骨は、この「判断」を明確な**信頼境界**（ここから先は信用しない、という線引きのこと）の向こう側に切り離すことです。判断役は「このリクエストはこのモデルがいいと思うよ」と**助言する**だけで、実行も課金も一切触りません。

大事なルールは2つ。

1. **助言は、必ずテナントの許可リスト（allowlist）を通る。** 助言者が「このモデルを使え」と言っても、そのテナントが使えないモデルなら無視される。助言者がアクセス権を勝手に広げることは、構造的にできません。
2. **助言者はバージョンで固定（version-pin）される。** 外部サービスと通信するとき、想定と違うバージョンなら**通信を拒否して安全側に倒す**（fail-closed）。「知らないうちに助言者がすり替わっていた」を防ぎます。

つまり判断役は「賢い助言者」でありながら、**間違っても・乗っ取られても、お金とアクセス権には手が届かない**。

![判断・実行・課金を分ける信頼境界](/images/stratoclave-ai-gateway-update/vsr-trust-boundary.png)

### 判断そのものは、既製の vLLM Semantic Router に委ねる

ここが今回いちばん正直に書きたいところです。「賢い判断ロジック」を Stratoclave が自前で作り込むべきか——答えは「作らない」です。

モデル選択の判断は、すでに優れた OSS が存在します。[vLLM Semantic Router](https://github.com/vllm-project/semantic-router)（以下 SR）です。SR は 16 種類のシグナル（ドメイン、PII、脱獄検知、複雑さ、ツール利用、セッション状況……）でプロンプトを分類し、「どのモデルに振るべきか」を判断してくれる、ルーティング専門のレイヤーです。

Stratoclave の方針は「**SR が判断し、Stratoclave が勘定する**」です。具体的な設計はこうです。SR の **判断だけを取り出せるエンドポイント**（`/api/v1/eval`。推論を実行せず、判断結果だけを返す）に問い合わせ、返ってきた「このモデル」を Stratoclave が**自前のトランスポートで予約・実行し、自前の実測値で精算する**——という形にします。判断は SR に委ね、お金は自分で握る。この形なら、SR がどれだけ賢くなっても（あるいはバグっても）、課金の正しさは Stratoclave 側で完結します。ルーティングは fail-open（SR が落ちたら通常のルーティングに戻る）、課金は fail-closed（SR を経由する新しい支払い経路は作らない）という設計です。

:::message
正直な現状: この「実 SR 連携」は**実装済みで3つの API すべてに配線済み、ただし既定では OFF**です。`SEMANTIC_ROUTER_ENABLED` とテナントごとの `sr_mode` を有効にすると、Stratoclave が実際に SR の `/api/v1/eval` に問い合わせ、返ってきたモデルを（クライアント指定のピンと同じ許可リスト・予約経路を通して）選びます。既定では両方 OFF なので、通常のデプロイは従来とバイト単位で同一です。実機の SR ルータを立てて `/api/v1/eval` が実際に判断を返すところまで検証済みで、お金の不変条件（予約なしに実行しない／請求は予約額を超えない／精算は一度きり）は疑似 SR を使った形式検証（Z3）とプロパティテストで証明しています。本番で有効化する前の前提条件（判断エンドポイントの権限スコープ、レイテンシ SLO、判断→モデル対応表の CI ゲート、プロンプトの取り扱い）は、リポジトリ内の設計ドキュメント（`backend/mvp/sr/CONTRACT.md`）に明記しています。
:::

なお、この方針に至る前は「VSR（Value/Session Router）」という名前で判断層を自前実装していました。次章の SAAR も、もとは自前実装です。現在はどちらも「既製の SR に委ねる」方向へ舵を切り、自前実装は SR 連携が有効になるまでの**つなぎ**として残っています。この記事では設計の考え方（信頼境界）を主役にし、実装の置き換えは進行中、という前提で読んでください。

## 4. セッション対応ルーティング — エージェントの会話を「同じモデルに貼り付ける」

今回いちばん紹介したいのは、機能そのものよりも「**セッション対応ルーティングという潮流**」です。これは私の思いつきではなく、LLM 推論エンジンの世界で今まさにホットなテーマで、Stratoclave はそれをゲートウェイ側に持ち込もうとしています（後述のとおり、その実装は現状 vLLM Semantic Router 側に寄せていく途中で、自前分は既定 OFF のつなぎです）。

### 問題: エージェントは「1回のプロンプト」ではなく「セッション」

まず、なぜセッションを意識したルーティングが要るのか。ここは vLLM（高速 LLM 推論エンジン）の公式ブログの説明が的確なので引用します。

> A coding or research agent is not one prompt. It is a session. It plans, calls tools, receives tool outputs, edits files, runs tests, recovers from errors...
> （コーディングやリサーチのエージェントは、1回のプロンプトではない。セッションだ。計画を立て、ツールを呼び、ツールの出力を受け取り、ファイルを編集し、テストを走らせ、エラーから復帰する……）
>
> — vLLM Blog, [Session-Aware Agentic Routing](https://vllm.ai/blog/2026-06-02-session-aware-agentic-routing)（2026-06-02）

Claude Code のようなエージェントは、1つのタスクの中で何度も LLM とやり取りします。このとき、リクエストを毎回バラバラに「今いちばん空いてるモデル」へ振り分けると、何が起きるか。vLLM ブログはここで2つの問題を挙げています。1つは「ツールの結果が、そのツール呼び出しをしていないモデルに届いてしまう」こと（会話の一貫性が崩れ、挙動が不安定になり得ます）。もう1つは「長く暖まったセッションがプレフィックスの局所性を失い、無駄に高くつく」こと——**同じ会話を別のワーカーに散らすと、せっかくのキャッシュが無駄になってしまう**という問題です。

### なぜキャッシュが効くのか（prefix caching）

「キャッシュが無駄になる」の意味を補足します。vLLM には **automatic prefix caching** という機能があります。

> caches the KV cache of existing queries, so that a new query can directly reuse the KV cache if it shares the same prefix
> （既存クエリの KV キャッシュを保存し、同じ前置き（prefix）を共有する新しいクエリがその計算済み KV を直接再利用できるようにする）
>
> — vLLM Docs, [Automatic Prefix Caching](https://docs.vllm.ai/en/latest/features/automatic_prefix_caching/)

かみ砕くと: マルチターンの会話では、毎回「これまでの会話全部」を LLM に読ませ直します。でも前半は前回と同じなので、その計算結果（KV キャッシュ）を使い回せば、**最初のトークンが出るまでの時間（TTFT）が大きく縮む**。会話が長いほど効きます。

ただし——ここが肝心です——このキャッシュは、**「同じワーカー」に当たったときだけ**再利用できます。会話を毎回バラバラのワーカーに散らすと、キャッシュはミスし、全部を計算し直すことになる。vLLM の公式ドキュメント自体は「どのワーカーに当てるか（ルーティング）」までは扱っていません。**その"当て方"を担うのがセッション対応ルーティングです。**

### SAAR がやること

Stratoclave の **SAAR（Session-Aware Agentic Routing）** は、まさにこれをゲートウェイ側で実現します。位置づけとしては、前章の信頼境界（判断を実行・課金から切り離す）と並ぶ「ルーティングのセッション軸」です。

（前章と同じ注記です。このセッション対応ルーティングは、本来は vLLM Semantic Router が内包する考え方であり、Stratoclave では自前実装が**つなぎ**として動いています。既定では OFF で、最終的には SR 側のセッション対応ポリシーに寄せていく方針です。）

- **セッションに貼り付ける（sticky）**: 同じセッションのリクエストは、原則として**同じモデル選択を維持**する。会話の途中でモデルがコロコロ変わらない。
- **ツールループのハードロック**: エージェントがツールを呼んでいる最中は、**そのモデルに強制的にロック**する。ツール結果が別のモデルに飛んで会話の一貫性が崩れる事故を防ぐ。vLLM ブログのいう「ツール呼び出しをしたのと同じモデルに結果を返す」を実装したものです。
- **アイドルタイムアウト・ドリフトのリセット**: 会話が途切れて時間が経ったり、話題が大きく変わったら、貼り付けを解除して選び直す。
- **プロバイダ状態のロック**: 「会話の続き」を表す ID（continuation id）など、特定のバックエンドでしか通用しない状態を持つリクエストは、その提供元に貼り付けて、別のバックエンドに飛ばさない。
- **切り替えコストのゲート**: モデルを乗り換えると（キャッシュが飛ぶので）コストがかかる。だから乗り換えの是非を、あの原子的な予算予約の時点で判断する。予算を超える乗り換えは、他の予算超過と同じように止まります。

vLLM ブログが報告しているセッション対応ルーティングの効果は、なかなか強烈です。

> Full SAAR cuts model switches by 79.29%, eliminates 3,836 unsafe switches, and reduces estimated physical-model cost by 78.71%.
> （完全な SAAR は、モデル切り替えを 79.29% 削減し、3,836 件の危険な切り替えをゼロにし、**推定**物理モデルコストを 78.71% 削減する）

この数値は、vLLM が**自分たちの実装・自分たちのベンチマーク**（21,600 ターンの決定論的な評価。ツール中心・フロンティア中心・アイドル中心など複数のセッション型を含む）で、セッションを見ずに毎ターン選び直す方式（single-turn routing）を基準に測った**推定**コストです。なお引用文中の「SAAR」は vLLM 側の実装名で、Stratoclave は同じ名前を借用しているだけです。上記の数値は vLLM の実装のものであり、Stratoclave の数値ではありません。

Stratoclave の SAAR は、**この「セッションに貼り付ける」という考え方**を、Bedrock を含むゲートウェイのレイヤーに持ち込んだものです（vLLM の 79% という数値がそのままゲートウェイ層で出る話ではなく、**当方環境での効果は未測定**です）。強調したいのは数値そのものではなく、「エージェント時代には、セッションを意識したルーティングにこれだけの伸びしろがある」という業界の潮流に、Stratoclave が乗れているという点です。しかも Stratoclave は切り替えの判断を「お金の予約」と統合しているので、「**一貫していて安全で、しかも予算内**」を1か所で決められます。セッション対応ルーティングは現在フィーチャーフラグで OFF のまま出荷しており（＝デフォルトでは動かない）、必要なテナントだけ有効化できます。

### Bedrock でも「貼り付け」が効く理由

ここは補足ではなく、この機能の論理の要です。vLLM の SAAR は「複数の物理モデル（GPU 上のサービング）を切り替える」文脈で、キャッシュは「同じワーカーに当たること」で効きます。一方 Bedrock ではワーカーを利用者が選べません。では Bedrock で貼り付けが効く理由は何かというと、別の2つの理由です。

1. Bedrock の [prompt caching](https://docs.aws.amazon.com/bedrock/latest/userguide/prompt-caching.html)（対応モデルは限定されます）はモデルに紐づくため、モデルを切り替えるとキャッシュが無効になる
2. 会話の途中でモデルの"性格"が変わると、出力が不安定になる

Stratoclave は vLLM の発想を、Bedrock のモデル選択・自前サービングの選択を含むゲートウェイのルーティングに一般化して持ち込んでいます。すでに紹介したハイブリッドサービングと組み合わされば、Bedrock も自前 vLLM も同じセッション対応ルーティングの対象になります。

## 5. Savings Certificate — 「いくら安くできたか」を正直に出す

### ルーティングの"うさんくささ"

「賢くルーティングしてコストを削減！」という宣伝はよく見ます。でも、たいてい根拠が曖昧です。「安いモデルに振ったから、たぶんこれくらい浮いたはず」という**推計**で、実際の請求と突き合わされていない。

Stratoclave はここに正直であろうとしました。それが **Savings Certificate（節約証明書）** です。

### 何をしているか

Stratoclave には **shadow judge**（シャドー判定器）という、実行には一切影響しない"影の判定器"があります。これはリクエストごとに「このリクエストなら、もっと安い同系統のモデルでも足りたはず」という判断を記録します。**実際のモデル選択は一切変えません**（だから安全）。記録するだけです。

そして毎日、この記録を **span_id（リクエストごとの一意な ID）で実際の課金台帳と突き合わせて**、「もし助言に従っていたら、実際の請求ベースでいくら浮いていたか」を計算し、証明書として発行します。ここが肝心で:

- 計算の土台に、"それっぽい概算" ではなく **実際に請求されたトークン数 × 検証可能な料金レート**を使う（Bedrock は公表料金、自前サービング分は運用者が設定した単価です。また「もし安いモデルなら」という反実仮想である以上、これも本質的には見積りなので、後述の通り "潜在的な節約" として明示します）
- **料金レートのバージョンを刻印**する（あとで再現・検証できる）
- **もし高いモデルにエスカレーションしていたら、その損失も差し引く**（都合のいい数字だけ見せない）

![span_id による節約証明の突合](/images/stratoclave-ai-gateway-update/savings-join.png)

さらに正直さの徹底として、Stratoclave は、**「潜在的な節約（potential）」と「実証済みの節約（realized）」を厳密に分けます**。shadow judge は「安いモデルで足りたはず」と言っているだけで、実際に安いモデルを走らせて品質を確認したわけではない。だから証明書には常に「品質は未測定」という但し書きが付き、この数字が確定値に化けることはありません。

「実際に安いモデルを走らせて品質まで測る」のは、使う人が自分のユースケースで判断すること——ゲートウェイの役割は、**その判断ができる正直なデータを、監査できる形で出すところまで**。ここが「うさんくさい削減アピール」との決定的な違いです。

### テナント単位で、安全に有効化できる理由

この shadow judge は**テナント単位で ON / OFF / 未設定（システムのデフォルトに従う）の3状態で制御できるトグル**になっていて、管理画面から操作できます。新しく作ったテナントはデフォルト ON なので、導入した週から節約証明書が貯まり始めます。

先ほどの SAAR がデフォルト OFF なのに対して、shadow judge をデフォルト ON にできるのには理由があります。shadow judge は**実行に一切影響しない**（記録するだけ）ので、ON にしても安全なのです。ここにも「判断は記録するが、実行・課金には触れない」という信頼境界の思想が効いています。もちろん、運用者が全体を一発で止める緊急スイッチ（kill-switch）も用意しています。

## まとめ: Stratoclave の"強み"を整理する

ここまでの機能を踏まえて、Stratoclave が何者なのかを整理します。この記事はアピール記事なので、遠慮なく言い切ります。

Stratoclave は「たくさんの provider に対応した便利ツール」ではありません。「**お金を預かるゲートウェイ**」です。その一点に振り切ったからこそ出せる価値があります。

**強み1: 同時アクセスによる予算の突き抜けが構造的に起きない**
事後カウンター方式の予算管理では、同時に大量のリクエストが来ると上限を突き抜けるレースコンディションが構造的に残ります。Stratoclave はリクエスト時点で原子的に予約するので、その場で止まる。暴走スクリプトが同時アクセスで予算を溶かす事故が起きません。

**強み2: コストの数字を、推計ではなく検証可能な形で出す**
ルーティングの判断・実行・課金が 1本の台帳で span_id によって繋がっています。だから節約額を、実際の請求トークン数と突き合わせ、料金レートのバージョンを刻印し、エスカレーション損失を差し引いて出せます。「削減できました（推計）」で終わらせず、検証可能なデータを出す。この構造は、コスト管理を別コンポーネントで後付けしたシステムでは再現が困難です。

**強み3: 判断が、お金とアクセス権に触れられない**
賢いルーティングを載せても、その判断ロジックはお金と権限から隔離される設計です。バグっても乗っ取られても、事故が課金・アクセスに波及しない。だからこそ判断そのものは既製の vLLM Semantic Router に安心して委ねられる——賢さを足しても、お金の正しさは Stratoclave 側で完結します（この実 SR 連携は前述のとおり実装済み・既定 OFF の段階です）。

**強み4: エージェント時代のルーティング（セッション対応）を見据える**
vLLM が自社ベンチマークで大きな効果（モデル切り替え約 79% 減）を示した「セッションに貼り付けるルーティング」という設計を、Bedrock を含むゲートウェイ層に取り込む方針です。しかも切り替えの判断が予算予約と統合されているので、「一貫していて安全で、しかも予算内」を1か所で決められます。

**強み5: Bedrock も自前 GPU も、同じ台帳・同じ執行の仕組みに載せる**
1つの OpenAI 互換の差し替え点で、自前サービングを Bedrock と同じ予約 → 精算のフローに載せられます。「広さ」を諦めたのではなく、「アダプタを書く戦い」を「差し替え点に載せる戦い」に変えただけです。

---

冒頭に挙げた「金になるイベントを、信頼境界を越えて、一度だけ・正しい相手に・否認できない形で記録する」——この難しい問題を、予約型台帳・判断と課金を分ける信頼境界・span_id による突合という3点で解いているのが Stratoclave です。SaaS の AI 課金でも、社内プラットフォームの予算配布でも、実験ごとのコスト計測でも、個人の使いすぎ防止でも、根っこは同じ。だからこそ、幅広く使えると思っています。
