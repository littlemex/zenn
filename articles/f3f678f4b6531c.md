## 1. Introduction

AWS recently published [`aws-neuron/neuron-agentic-development`](https://github.com/aws-neuron/neuron-agentic-development), a pip-installable package that ships a curated set of AI agents and skills for AWS Trainium and Inferentia development straight into an agentic coding assistant such as Claude Code. Once the package is installed and its `deploy-neuron-agentic-development-to-claude` command is run, the assistant gains slash-invocable skills for writing NKI kernels from PyTorch or natural language, debugging compilation errors, walking a NEFF profile, and answering documentation questions grounded in the bundled NKI references, together with matching sub-agents that take ownership of each of those responsibilities. Because every skill carries its own markdown reference tree under `~/.claude/skills`, the assistant grounds its answers in source-level citations rather than drifting into hallucinations, and the workflows that normally live in a team's internal runbooks become one-command building blocks available to any developer logged into the instance.

This article introduces a small, reproducible environment that pairs the new package with a Trainium workstation so that you can try the skills and agents end to end within minutes of cloning a repository. The environment is driven by the public [`littlemex/aws-neuron-samples`](https://github.com/littlemex/aws-neuron-samples) tooling: a single `deploy.sh` invocation provisions a Neuron DLAMI instance via CDK, attaches persistent EFS storage, exposes code-server only through an AWS Systems Manager tunnel, installs the Claude Code CLI, wires it to Amazon Bedrock through a narrow instance-role policy, and runs `deploy-neuron-agentic-development-to-claude` so that every agent and skill is ready the moment you log in. The sections that follow walk through the whole flow, including an optional detour that pulls a Neuron Deep Learning Container from ECR so you can compare the DLAMI-native path against a container-based one.

![](/images/f3f678f4b6531c-arch.png)

Everything in this article was validated end to end on a Trainium 2 instance (`trn2.3xlarge`) deployed into `sa-east-1`, with Bedrock calls routed to `us-east-1` via a US inference profile. All command snippets are written so that you can copy them into a shell unchanged: every value that you might want to customise is presented as an `export` with a working default, so you only edit the ones you actually need to change.

## 2. Workshop: from zero to a Neuron-aware Claude Code session

This section follows the flow a new user would take on a freshly bootstrapped AWS account. The goal is to end with a workstation where (a) code-server is reachable from your laptop through an SSM tunnel, (b) a Neuron DLC image can be pulled from ECR and inspected, and (c) Claude Code invokes Amazon Bedrock using the instance role and exposes the `neuron-agentic-development` agents and skills as `/slash` commands. Every subsection opens with a short `:::message` block explaining what the step accomplishes, followed by the commands and the observed output.

### 2.1 Prerequisites and shared environment variables

:::message
The values below are not placeholders: they are the concrete defaults used throughout the rest of the article. Override any of them in your shell before running the subsequent commands if you need a different region, stack name, or EFS subpath.
:::

```bash
# AWS region where the Trainium instance will run and where the
# Capacity Block is reserved. sa-east-1 is used below because that
# is one of the regions that offers trn2.3xlarge Capacity Blocks.
export AWS_REGION=sa-east-1

# Region where the Neuron Deep Learning Container is pulled from on
# ECR. Kept separate from AWS_REGION because the Bedrock environment
# block that --install-claude-code writes into ~/.bashrc points
# AWS_REGION at us-east-1 (the region of the Claude inference
# profile), whereas docker pulls should stay in the instance region.
export ECR_REGION=sa-east-1

# AWS profile to use locally. Leave unset to fall back to the default
# credential chain on your laptop.
export AWS_PROFILE=default

# CloudFormation stack name for the workstation.
export STACK_NAME=neuron-ws

# Pre-existing EFS file system ID used for persistent state.
# Create one ahead of time (see below) and paste its ID here.
export EFS_ID=fs-00000000000000000

# EFS subpath, so multiple stacks can share the same file system
# without trampling each other's /home/coder.
export EFS_SUBPATH=/neuron-workspace/main

# Port forward mapping used when connecting from the laptop.
export LOCAL_PORT=8080
export REMOTE_PORT=80
```

Set these variables once at the top of the shell session. Every later command re-uses them. If you prefer a different stack name for a second instance, export `STACK_NAME=neuron-ws-b`, pick a different `EFS_SUBPATH` such as `/neuron-workspace/b`, and rerun the deploy step.

### 2.2 Provision an EFS file system and mount target

:::message
The CDK stack deliberately does not create or own the EFS resources, because a team typically wants one long-lived EFS shared across many stacks rather than a per-stack file system that disappears on `cdk destroy`. You create the file system and its mount target once using the AWS CLI, capture their ids, and hand the file system id to `deploy.sh` through `--efs-id`. Subsequent deploys that point at the same `EFS_ID` but a different `EFS_SUBPATH` simply carve out a separate subtree under the same file system, so one EFS is enough for an arbitrary number of stacks.
:::

On the security group side, you still do not need to edit anything manually: `deploy.sh` inspects the EFS mount target and adds a scoped NFS (TCP/2049) ingress rule from the new instance's security group to the mount target's security group for you, and the matching `--destroy` path revokes that rule when you tear the stack down. The commands below therefore focus on the two resources you do have to own, the file system and its mount target, and leave the security group wiring to the deploy step.

```bash
# Default VPC subnet in the target region. Replace with a specific
# subnet id if you do not use the default VPC.
export SUBNET_ID=$(aws ec2 describe-subnets --region "$AWS_REGION" \
  --filters Name=default-for-az,Values=true \
  --query 'Subnets[0].SubnetId' --output text)

# Create the file system and capture its id.
export EFS_ID=$(aws efs create-file-system \
  --region "$AWS_REGION" \
  --encrypted \
  --tags Key=Name,Value=neuron-workspace \
  --query FileSystemId --output text)

# Wait until the file system reaches the `available` state.
aws efs describe-file-systems --region "$AWS_REGION" \
  --file-system-id "$EFS_ID" \
  --query 'FileSystems[0].LifeCycleState' --output text

# Pick a security group to attach to the mount target. It does not have
# to allow anything yet; deploy.sh will add NFS ingress from the new
# instance SG automatically at deploy time.
export EFS_MT_SG=$(aws ec2 describe-security-groups --region "$AWS_REGION" \
  --filters Name=group-name,Values=default \
  --query 'SecurityGroups[0].GroupId' --output text)

# Create one mount target in the subnet you plan to launch in.
aws efs create-mount-target \
  --region "$AWS_REGION" \
  --file-system-id "$EFS_ID" \
  --subnet-id "$SUBNET_ID" \
  --security-groups "$EFS_MT_SG"
```

If you already have an EFS file system and a mount target in the target AZ, skip the `create-*` commands and simply set `EFS_ID` to the existing id. The next deploy will reuse it.

### 2.3 Clone the repository and deploy the workstation

:::message
A single `deploy.sh` invocation builds the CDK app, creates the stack, waits for the SSM agent, and then runs the SSM Run Command tasks that install code-server, nginx, persistent storage, and VS Code extensions. The optional `--install-claude-code` flag attaches a scoped Bedrock inline policy, installs the Claude Code CLI, and deploys the `neuron-agentic-development` package under `~/.claude`. The optional `--full` flag, in turn, layers a Cognito Hosted UI + CloudFront frontend in front of code-server so the workbench is reachable from a browser without ever opening a port on the instance.
:::

```bash
git clone https://github.com/littlemex/aws-neuron-samples.git
cd aws-neuron-samples/setup/single-node

# One-time CDK bootstrap for the target account and region.
npx cdk bootstrap aws://$(aws sts get-caller-identity --query Account --output text)/"$AWS_REGION"

# Install CDK dependencies.
(cd cdk && npm install)
```

You then pick one of two deploy shapes. The first is the original SSM-only workstation, useful when you only need code-server through `aws ssm start-session` and do not want any Cognito or CloudFront moving parts. The second is the one-shot browser-ready workstation introduced in the most recent revision: `--full` adds the EFS, internal ALB, Cognito User Pool, and CloudFront stacks on top of the EC2 stack, and the `--operator-email` / `--operator-password` pair bootstraps an operator user inside Cognito at deploy time so you can log in the moment the stacks reach `CREATE_COMPLETE`.

```bash
# Shape 1: SSM-only workstation (no CloudFront, no Cognito).
# Best for solo developers who just want a port forward to code-server.
bash scripts/deploy.sh \
  --stack-name "$STACK_NAME" \
  --efs-id "$EFS_ID" \
  --efs-subpath "$EFS_SUBPATH" \
  --install-claude-code \
  -r "$AWS_REGION"

# Shape 2: full browser-ready workstation. EFS + internal ALB +
# Cognito Hosted UI + CloudFront, with an operator user pre-created.
# The password is staged in Secrets Manager and only its ARN flows
# through CDK context, so the plaintext never lands in CFN templates,
# cdk.out, or drift diffs.
bash scripts/deploy.sh \
  --stack-name "$STACK_NAME" \
  --full \
  --efs-subpath "$EFS_SUBPATH" \
  --install-claude-code \
  --operator-email ops@example.com \
  --operator-password 'StrongPwd123!' \
  -r "$AWS_REGION"
```

When the script completes it prints the instance id, the Secrets Manager ARN holding the code-server password (kept for backwards compatibility, no longer used as a login password in the `--full` shape), the CloudFront distribution domain, and an exact `aws ssm start-session` command that forwards a port of your choice from `localhost` to `:80` on the instance. In my runs the CDK phase completes in a little over three minutes for the SSM-only shape and seven to nine minutes for `--full`, while the code-server setup tasks add another fifteen to twenty minutes, dominated by `apt install` over the EFS-backed `/home/coder`.

The invocation above runs the instance on on-demand pricing, which is the simplest default. The same `deploy.sh` supports two alternative capacity models, and you select between them at deploy time. Spot is the cheapest path and is well suited to interactive development as long as you are prepared for the interruption; `--spot-interruption-behavior stop` is usually what you want because the instance comes back in place rather than being torn down. Capacity Block is the right answer when you have reserved a specific time window for guaranteed access to `trn2.3xlarge` in a given AZ; `manage-capacity-block.sh` captures the reservation id and subnet id in SSM Parameter Store so that `deploy.sh --use-capacity-block` resolves them automatically on the next run, which keeps the single-reservation case down to one flag.

When you hold multiple Capacity Block reservations simultaneously, give each reservation its own named slot in SSM Parameter Store with `manage-capacity-block.sh save-params --slot NAME` and then point each stack at the slot it should consume using `deploy.sh --use-capacity-block --slot NAME`. Slots are cheap to create, easy to list with `manage-capacity-block.sh list-params`, and the default slot stays compatible with the simpler single-reservation flow above. If instead you hold a single reservation that already covers several instances (`TotalInstanceCount > 1`), reuse the same slot across several `deploy.sh` runs with different `--stack-name` and `--efs-subpath` values; AWS consumes one instance slot per stack until the reservation is full.

```bash
# Spot, with stop-on-interruption so the instance survives reclaim events.
bash scripts/deploy.sh \
  --stack-name "$STACK_NAME" \
  --efs-id "$EFS_ID" --efs-subpath "$EFS_SUBPATH" \
  --install-claude-code \
  --use-spot --spot-interruption-behavior stop \
  -r "$AWS_REGION"

# Capacity Block, resolving the reservation id from SSM Parameter Store.
bash scripts/deploy.sh \
  --stack-name "$STACK_NAME" \
  --efs-id "$EFS_ID" --efs-subpath "$EFS_SUBPATH" \
  --install-claude-code \
  --use-capacity-block \
  -r "$AWS_REGION"

# Two Capacity Blocks in parallel, one stack per named slot. Put each
# reservation into its own SSM slot once, then deploy stacks by name.
export CR_A=cr-AAAAAAAAAAAAAAAA
export CR_B=cr-BBBBBBBBBBBBBBBB
export SUBNET_A=subnet-AAAAAAAAAAAAAAAA
export SUBNET_B=subnet-BBBBBBBBBBBBBBBB

bash scripts/manage-capacity-block.sh save-params --slot cb-a \
  --reservation-id "$CR_A" --subnet-id "$SUBNET_A" -r "$AWS_REGION"
bash scripts/manage-capacity-block.sh save-params --slot cb-b \
  --reservation-id "$CR_B" --subnet-id "$SUBNET_B" -r "$AWS_REGION"

bash scripts/deploy.sh --use-capacity-block --slot cb-a \
  --stack-name neuron-cb-a \
  --efs-id "$EFS_ID" --efs-subpath /neuron-workspace/cb-a \
  --install-claude-code -r "$AWS_REGION"

bash scripts/deploy.sh --use-capacity-block --slot cb-b \
  --stack-name neuron-cb-b \
  --efs-id "$EFS_ID" --efs-subpath /neuron-workspace/cb-b \
  --install-claude-code -r "$AWS_REGION"

# List every slot in the current region (useful when rotating reservations).
bash scripts/manage-capacity-block.sh list-params -r "$AWS_REGION"
```

### 2.4 Connect code-server (browser via CloudFront, or SSM port forward)

:::message
The instance security group has no ingress rules and the ALB is `internal`, so neither the EC2 instance nor the load balancer is ever reachable from the public internet directly. There are two supported ways in: a CloudFront + Cognito browser login (when you deployed with `--full`), or an SSM port forward (always available).
:::

#### 2.4.1 Browser login via CloudFront + Cognito (recommended for `--full`)

When you deployed with `--full`, every browser request hits CloudFront first. CloudFront forwards to an internal ALB through a VPC Origin, the ALB stamps an `X-Origin-Verify` shared secret on every accepted request, and a CloudFront Function rejects any browser session that does not carry a valid HMAC-signed `cf_session` cookie. A request without that cookie is bounced to `/oauth/login`, which sends the user through the Cognito Hosted UI and returns to `/oauth/callback` to mint the cookie. Once the cookie is set, code-server itself runs in `auth: none` mode because the front door has already authenticated the user; running a second password prompt at the editor would conflict with the SSO flow and is intentionally disabled.

```bash
# Resolve the CloudFront distribution domain emitted by the
# <stackName>-frontend stack.
export CLOUDFRONT_DOMAIN=$(aws cloudformation describe-stacks \
  --region "$AWS_REGION" \
  --stack-name "${STACK_NAME}-frontend" \
  --query "Stacks[0].Outputs[?OutputKey=='CloudFrontDomain'].OutputValue" \
  --output text)

echo "Open https://${CLOUDFRONT_DOMAIN}/ in your browser."
```

Visiting `https://${CLOUDFRONT_DOMAIN}/` triggers the redirect to the Cognito Hosted UI on `*.amazoncognito.com`. Sign in with the email you passed to `--operator-email` and the password you passed to `--operator-password`. Cognito returns to `/oauth/callback?code=...`, the OAuth Lambda exchanges the code for the user identity, sets the `cf_session` cookie, and redirects to `/`. From that point on every request from the same browser is admitted by CloudFront and forwarded to code-server, so the editor opens without any further prompts.

If you ever need to invalidate a session manually, delete the `cf_session` cookie in your browser. The CloudFront Function will see a request with no cookie and bounce you back through Cognito, which is the simplest "log out" path because the cookie is the only durable session state on the client.

#### 2.4.2 SSM port forward (always available)

The SSM-only path remains useful for the workstation deployed without `--full`, for emergency access when CloudFront is misconfigured, and for forwarding application ports beyond code-server.

Open a second terminal on your laptop and run the helper that the repository ships with. It wraps `aws ssm start-session --document-name AWS-StartPortForwardingSession` so that you can forward one or more ports at once without re-typing the incantation.

```bash
bash scripts/deploy.sh \
  --port-forward --stack-name "$STACK_NAME" \
  -p "${LOCAL_PORT}:${REMOTE_PORT}" \
  -r "$AWS_REGION"
```

In the SSM-only shape (no `--full`), code-server still uses a generated password kept in Secrets Manager. Fetch it and open the URL:

```bash
# Resolve the Secrets Manager ARN for this specific stack.
export SECRET_ARN=$(aws secretsmanager list-secrets --region "$AWS_REGION" \
  --query "SecretList[?contains(Name, 'CodeServerPassword') && \
           Tags[?Key=='aws:cloudformation:stack-name' && Value=='$STACK_NAME']].ARN | [0]" \
  --output text)

aws secretsmanager get-secret-value --region "$AWS_REGION" \
  --secret-id "$SECRET_ARN" --query SecretString --output text

echo "Open http://localhost:${LOCAL_PORT} in your browser."
```

The login form renders at `http://localhost:8080/login`, paste the password, and you land on the code-server workbench. I have seen occasional WebSocket 1006 errors when the reverse proxy forwards `Host` instead of `$http_host`; the nginx configuration shipped in this repository uses `$http_host` so that the upstream Origin check in code-server is satisfied and the editor connects without reload loops. In the `--full` shape this password no longer gates the editor (code-server runs with `auth: none`); CloudFront + Cognito has already authenticated the user, so the SSM port forward, when used, drops you straight into the workbench.

### 2.5 Survive Spot interruption and Capacity Block churn

:::message
Persistent state lives on EFS, not on the instance root volume, so a Spot reclaim or a Capacity Block expiry never throws away `/home/coder`, `/work`, or the Neuron NEFF compile cache. A companion `recover.sh` turns that durable state back into a working workstation in one command, without repulling the DLAMI or reinstalling code-server from scratch.
:::

In day-to-day use the EC2 instance is treated as disposable: Spot can reclaim it at any time and Capacity Blocks expire on a schedule. What makes that acceptable is that everything a developer actually cares about, including the `/home/coder` dotfiles, the `/work` checkout, the Claude Code state under `~/.claude`, and the Neuron NEFF cache warmed by previous compilations, is bind-mounted through EFS in the `EFS_SUBPATH` subtree. The instance's root volume and the NVMe instance store cache are intentionally throwaway tiers in that design, which means a replacement instance can be brought online without touching any of the developer state.

```bash
# After a Spot reclaim or a CB renewal, point deploy.sh at the same
# EFS_SUBPATH as before. The CDK stack is recreated, the instance boots
# from the same DLAMI, setup-code-server.sh re-attaches /home/coder and
# /work to the EFS subtree, and the editor opens on the same tree of
# files you left behind. Nothing in your working directory is lost.
bash scripts/deploy.sh \
  --stack-name "$STACK_NAME" \
  --efs-id "$EFS_ID" --efs-subpath "$EFS_SUBPATH" \
  --install-claude-code \
  -r "$AWS_REGION"

# If the existing stack is still around (for example after an SSM-only
# transient issue or a stopped Spot instance), recover.sh is the faster
# path: it re-reads the CloudFormation outputs, starts the instance if
# it is stopped, re-authorises the cross-SG NFS rule if it went missing,
# and re-runs the idempotent setup tasks so the editor comes back.
bash scripts/recover.sh --stack-name "$STACK_NAME" -r "$AWS_REGION"
```

The combination keeps the mental model simple. If the instance is gone entirely, run `deploy.sh` again and you get an identical workstation pointing at the same EFS subtree within a few minutes. If the instance is there but something about it is unhealthy, `recover.sh` brings it back without a full redeploy. Either way the only thing that changes across these events is the underlying EC2 identity; the editor opens on the exact state you were working on before.

### 2.6 Run the Neuron Deep Learning Container

:::message
The helper `setup-neuron-dlc.sh` pulls a Neuron DLC image from ECR and leaves it available locally for `docker run`. Defaults resolve to the public AWS Neuron DLC, so no private credentials are required to reach this step. The quickest way to prove that the DLC actually talks to the Trainium devices is to run `neuron-ls` inside the container and confirm that the expected NeuronCores are listed.
:::

Before running Docker you need a working daemon and a user that can talk to it without `sudo`. The task runner's `17-install-docker` step takes care of both: it installs `docker.io` when the DLAMI has not already shipped it, starts the systemd service, and adds the code-server user (`coder`) to the `docker` group. The step is idempotent, so re-running the full `deploy.sh --install-claude-code` flow is always safe, and on a fresh instance it lands you with a working `docker` CLI for `coder` before you ever touch `setup-neuron-dlc.sh`.

From a terminal inside code-server (or from an SSM shell on the instance), run the helper to authenticate and pull the image. The defaults point at the public `763104351884.dkr.ecr.<region>.amazonaws.com/pytorch-training-neuronx` repository. Setting `NEURON_DLC_IMAGE_URI` overrides everything, and is how you would point the same script at a private or pre-release image without committing any URI into source control.

```bash
# Optional: pin an exact image URI. Leave unset to use the public default.
# export NEURON_DLC_IMAGE_URI=<your full ECR URI including tag>

# Authenticate Docker against ECR using the instance role.
bash scripts/setup-neuron-dlc.sh login

# Pull the image. The first pull is a multi-GB download; subsequent
# runs reuse the cached layers.
bash scripts/setup-neuron-dlc.sh pull
```

Once the image is local, run `neuron-ls` inside a throwaway container with the NeuronCore devices and the Neuron runtime sockets mounted through. The container exits as soon as `neuron-ls` finishes, so there is no long-lived process to clean up.

```bash
# The image URI is the same one setup-neuron-dlc.sh just pulled. Use
# ECR_REGION rather than AWS_REGION so the tag resolves to the region
# where the image actually lives, even when the shell's AWS_REGION has
# been overridden for Bedrock in us-east-1.
export NEURON_DLC_IMAGE="763104351884.dkr.ecr.${ECR_REGION}.amazonaws.com/pytorch-training-neuronx:2.9.0-neuronx-py312-sdk2.29.1-ubuntu24.04"

# Run neuron-ls from inside the DLC and exit.
docker run --rm \
  --device=/dev/neuron0 \
  -v /run/neuron/:/run/neuron/ \
  "$NEURON_DLC_IMAGE" \
  neuron-ls
```

On a `trn2.3xlarge` the output is a table that enumerates one NeuronCore v3 device with four NeuronCores and the associated HBM size, which confirms that the DLC has visibility into the accelerator that the DLAMI itself exposes at `/dev/neuron0`. From here you can swap `neuron-ls` for any training script by adding bind mounts for your code and data, or drop into an interactive shell with `bash scripts/setup-neuron-dlc.sh shell` when you want to explore the image layout.

If you want to inspect the image's on-disk layout without starting the container, run `bash scripts/setup-neuron-dlc.sh extract`. The helper creates a temporary container from the pulled image (without running it), copies a subtree out to `~/neuron-dlc-workspace`, and removes the container. With no arguments the helper only looks for `/workspace` at the image root, which is the convention used by AWS's pre-release and workshop DLCs; if the image has no `/workspace`, as is the case for the public `pytorch-training-neuronx` DLC, extract warns and exits cleanly so the rest of your flow is unaffected. Pass `--extract-path PATH` to pull an explicit subtree, for example `bash scripts/setup-neuron-dlc.sh extract --extract-path /opt/aws_neuronx_venv_pytorch_2_9` once you have identified an interesting directory by poking around with `bash scripts/setup-neuron-dlc.sh shell`.

### 2.7 Verify that Claude Code talks to Amazon Bedrock

:::message
The `--install-claude-code` flag that was passed to `deploy.sh` above already installed the Claude Code CLI under `~/.npm-global`, populated `~/.bashrc` with the Bedrock environment variables, and deployed the `neuron-agentic-development` agents and skills under `~/.claude`. The step below is a smoke test.
:::

From an SSM shell on the instance (or from a terminal in code-server), verify the environment and issue a one-word prompt.

```bash
# Switch to the code-server user. `sudo -i` picks up ~/.bashrc.
sudo -i -u coder

# Confirm that the Bedrock environment is active.
env | grep -E '^(CLAUDE_CODE_USE_BEDROCK|AWS_REGION|ANTHROPIC_DEFAULT_OPUS_MODEL)='

# Route the smoke test through Opus 4.7 on Bedrock.
echo "Reply with just: ok" | claude -p --output-format text
```

The expected output is the single word `ok`. The model identifier comes from `ANTHROPIC_DEFAULT_OPUS_MODEL=us.anthropic.claude-opus-4-7`, which the managed block in `~/.bashrc` exports for every login shell. The IAM inline policy attached by the CDK stack grants `bedrock:InvokeModel`, `bedrock:InvokeModelWithResponseStream`, `bedrock:ListInferenceProfiles`, and `bedrock:GetInferenceProfile`, scoped to foundation models and application inference profiles in any region, which is enough for the CLI to use a `us.` cross-region inference profile even though the instance itself runs in `sa-east-1`.

### 2.8 Exercise a Neuron NKI skill end to end

:::message
The `neuron-agentic-development` package ships four agents and five skills into `~/.claude`. The skills are exposed as slash commands such as `/neuron-nki-docs`, and they carry their own references (NKI API documentation, debugging guides, profiling cookbooks) in markdown form, so Claude Code can answer questions without reaching out to any external documentation host.
:::

Still as the `coder` user, ask the documentation skill about a specific NKI API. The `--permission-mode bypassPermissions` flag is a pragmatic choice for an interactive workstation where `~/.claude` is already trusted; keep the default mode for shared or untrusted environments.

```bash
cd ~

echo "/neuron-nki-docs nisa.tensor_tensor" \
  | claude -p --output-format text \
           --permission-mode bypassPermissions \
           --add-dir "$HOME/.claude/skills" \
           --add-dir "$HOME/.claude/agents"
```

In my runs this produced a structured summary of `nisa.tensor_tensor` assembled from `~/.claude/skills/neuron-nki-docs/references/programming/api/api-nki-isa-tensor.md`. The response included the function signature, the complete list of supported operators, the data-type coercion rules, the memory constraints between SBUF and PSUM, and a performance hint suggesting `nki.isa.tensor_scalar` when broadcasting is needed. Every fact in the response cited a specific file path inside `~/.claude/skills`, which is how the skills bundle ensures that Claude grounds its answers in the shipped documentation rather than the base model's training data.

### 2.9 Tear down when you are finished

:::message
Destroying the stack removes the EC2 instance and every resource the CDK created. The EFS file system and its data are deliberately preserved so you can relaunch a new stack against the same `EFS_SUBPATH` and resume exactly where you left off.
:::

```bash
bash scripts/deploy.sh \
  --destroy --stack-name "$STACK_NAME" \
  -r "$AWS_REGION"
```

The script revokes the cross-security-group NFS rule that it created on the EFS mount target before it calls `cdk destroy`. If you also want to remove the EFS file system, delete its mount targets first and then `aws efs delete-file-system --file-system-id "$EFS_ID"`.
