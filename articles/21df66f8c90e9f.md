## はじめに

**自分が使うために Amazon Bedrock 専用の LLM Proxy Gateway を作った。**

一番作りたかったのは Tenant / User 単位のクレジット管理機能で、組織利用用途だと部門ごとに利用者ユーザーにクレジットを割り当ててその上限まで使える、部門単位でクレジットを設定、ユーザー単位でも設定できる。個人的には個人プロジェクトごとに Tenant 単位で分離してクレジット管理したかったので作った。名前は `Stratoclave` で Strato が岩盤の上の地層、Conclave が秘密集会でテナント分離をイメージした。

**主な機能**

- Tenant / User 単位のトークン使用量把握、クレジット管理
- AWS ネイティブなインフラ(DynamoDB、ECS Fargate、CloudFront、ALB..)
- Admin / Team Lead / User の **3 段 RBAC** (独自拡張可能)
- `stratoclave claude` で Claude Code を Stratoclave 越しに起動
- `stratoclave ui open` で Web UI 起動
- Anthropic SDK からは `ANTHROPIC_BASE_URL` 差し替えだけ
- **SSO (AWS SSO / STS vouch) / API Key / Cognito User Pool** の 3 経路認証
- Claude Desktop の **Cowork (Gateway mode)** 対応

:::message
LiteLLM じゃ、、ダメなん、、？( ˘•ω•˘ ) → LiteLLM でいいんじゃ？ワイはこれを使うんや。

強いていうなら、RDBMS 不要、、SSO ログインがデフォで使える、、デプロイが CDK で簡潔、、Apache 2.0、、
:::

## 機能

### 1. AWS 完全ネイティブ

Stratoclave は以下のサーバレス系の AWS サービスだけで完結する。

| レイヤー | サービス |
|---|---|
| Auth | Amazon Cognito User Pool |
| Data | DynamoDB |
| Compute | ECS Fargate |
| CDN | CloudFront + CloudFront Function |
| Edge LB | ALB (HTTP、CloudFront 終端 HTTPS) |
| IaC | AWS CDK v2 |

```bash
export AWS_PROFILE=your-profile
export STRATOCLAVE_PREFIX=stratoclave
cd iac && npm install
./scripts/deploy-all.sh
```

これでインフラが全部デプロイされる。CI/CD 未整備。｡ﾟ(ﾟ＾ω＾ﾟ)ﾟ｡

### 2. Admin / Team Lead / User の 3 段 RBAC と Tenant 隔離

Role は DynamoDB `Permissions` テーブルに正規化されている (`permissions.json` がバージョン付きのソース)。

| Role | 主な権限 |
|---|---|
| admin | ユーザー / テナント / SSO / 使用量の全権操作 |
| team_lead | **自分が所有する Tenant のみ**可視、配下メンバーの credit と使用量を管理 |
| user | Messages API 呼び出しと自分の使用量閲覧、自分の API Key 発行 |

**Tenant 隔離は UserTenants テーブルの `(user_id, tenant_id)` 複合キー + `status=active` フィルタ**で実装している。他の Tenant は URL 直打ちでも 404。

![ユーザー詳細画面 (admin ロール)](https://github.com/littlemex/stratoclave/raw/main/docs/screenshots/images/06_admin_user_detail.png)

### 3. Tenant × User の 2 階層クレジット管理

- **Tenant ごとにデフォルトクレジット** (例: 新規メンバーは 100,000 tokens)
- **ユーザーごとに個別上書き** (`user_override`) が可能
- `POST /v1/messages` 受付時に `max_tokens + 入力見積もり`を DynamoDB の `ConditionExpression` で atomic に先取り予約 (pessimistic reservation)、Bedrock 呼び出し後に実消費との差額を refund。並列 N 本でも TOCTOU なしで残高超過をブロックしつつ、`UsageLogs` には実消費で 1 行記録する ( ง ˙o˙)ว

![所属 Tenant のメンバー一覧とクレジット](https://github.com/littlemex/stratoclave/raw/main/docs/screenshots/images/08_admin_tenant_detail.png)

自分の使用履歴はモデル別・日別で参照できる。

![自分の使用履歴](https://github.com/littlemex/stratoclave/raw/main/docs/screenshots/images/03_me_usage.png)

### 4. CLI から Claude Code、Web UI、Messages API すべてを叩ける

CLI は、`~/.stratoclave/config.toml` と `~/.stratoclave/mvp_tokens.json` (0600) だけに状態を持つ。

```bash
# Admin から配られた CloudFront URL を使って bootstrap
stratoclave setup https://<your-cloudfront>.cloudfront.net

# サインイン
stratoclave auth login --email you@example.com

# Claude Code を Stratoclave 越しに起動 (Claude Code 本体は別途インストール)
stratoclave claude -- "Summarize this repo in one sentence"

# Web UI を開く (認証済みタブでそのまま開く)
stratoclave ui open
```

:::message alert
Claude Code 本体はバンドルしていません。
:::

### 5. SSO / API Key / Cognito User Pool、3 通りの認証

#### Cognito User Pool (email + password)

管理者が email+password を発行、初回ログイン時にユーザーが新パスワードを設定する。

![ログイン画面](https://github.com/littlemex/stratoclave/raw/main/docs/screenshots/images/01_login.png)

#### AWS 短期 credentials 経由 (Vouch by STS)

`~/.aws/credentials` に AWS 短期 credentials がある状態で **パスワード不要**ログイン可能。credentials の入手方法は問わず、以下すべて同じ経路で通る。

- `aws sso login --profile xxx` (IAM Identity Center / AWS SSO)
- `saml2aws login --profile xxx` (Okta / ADFS / Entra ID SAML / OneLogin / Shibboleth 等、SAML 対応 IdP 全般)
- `aws configure sso` 後の通常プロファイル
- IAM user の長期 access key (default DENY、trusted account 側で `allow_iam_user: true` を明示した場合のみ)

```bash
# AWS SSO を使っている場合
aws sso login --profile your-sso-profile
stratoclave auth sso --profile your-sso-profile

# 社内 SAML IdP (Okta / ADFS / Entra ID 等) を使っている場合
saml2aws login --profile your-saml-profile
stratoclave auth sso --profile your-saml-profile
```

内部では CLI が `sts:GetCallerIdentity` を SigV4 で署名 → Backend が署名済みリクエストを STS にそのまま転送して身元確認 → Arn から identity_type 分類 (`sso_user` / `federated_role` / `iam_user` / `instance_profile`) → Trusted Account の policy に従い Cognito user を resolve / provision、という流れである。本稿ではこの方式を便宜上 **Vouch by STS** と呼ぶ (Stratoclave 固有の通称)。考え方は [HashiCorp Vault の AWS auth method (iam)](https://developer.hashicorp.com/vault/docs/auth/aws) が原典で、Vault は署名済み `GetCallerIdentity` リクエストを client が作り server 側が STS に転送する同じ仕組みを採用している。

Backend は「AWS 短期 credentials を持っている = AWS がその人を認証した」という事実だけを信用するので、**IdP が何であっても同じコードパスで通る**。saml2aws を使う組織も、AWS SSO を使う組織も、運用上の差分は `stratoclave auth sso --profile xxx` の profile 名だけ。

**EC2 Instance Profile は default DENY** (複数ユーザー共有は追跡不能なので意図的に塞いでいる)、社内 SAML で session_name が email でない環境も **招待** 経由で通過可能である。

#### 長期 API Key (`sk-stratoclave-*`)

IAM user を作らずに発行できる、headless クライアント向けの長期 API キー。

![API キー一覧](https://github.com/littlemex/stratoclave/raw/main/docs/screenshots/images/12_me_api_keys.png)

- **プレーンテキストは一度だけ表示**、サーバには **SHA-256 ハッシュのみ**保存
- スコープ narrowing: `messages:send` / `usage:read-self` (owner の roles と AND)
- 1 ユーザーあたり active 5 個まで
- revoke は即時 (キャッシュなし、毎リクエストで DynamoDB 参照)

### 6. Claude Desktop Cowork 対応

Claude Desktop 内蔵の **Cowork (Gateway mode)** に、長期 API キーで接続できる。

```
Gateway base URL   : https://<your-stratoclave>.cloudfront.net
Gateway auth scheme: Bearer
Gateway API key    : sk-stratoclave-...
```

> :::message alert
> Gateway base URL には `/v1` を含めないでください。
> cowork は内部で `/v1/models` / `/v1/messages` を自動 prefix するため、
> `/v1` を入れると `/v1/v1/...` にリクエストが飛んで 404 になります。
> :::

コマンドでも発行できる。

```bash
stratoclave api-key create \
  --name "cowork on macbook" \
  --scope messages:send --scope usage:read-self \
  --expires-days 30
```

### Step 6. CLI を bootstrap する

ローカル (macOS / Linux) で Stratoclave CLI をビルドして、先ほどの URL に向ける。

### ワーストケースで何が起こるか

一応先に結論を書いておく。Backend が RCE 等で完全に乗っ取られた場合に起こりうること、起こらないことは以下の通り。

**起こりうる (Stratoclave という箱の中に閉じる)**

- Stratoclave の全ユーザに成り代わられる
- Bedrock を過剰に呼ばれて課金が跳ねる
- DynamoDB 上のクレジット残量や UsageLogs を改竄される

**起こらない (構造上たどり着けない)**

- 組織の **IAM Identity Center / AWS SSO / Okta / Entra ID** の乗っ取り
  → Stratoclave は IdP の refresh token や API key を一切保持していない
- 組織の **IAM ユーザ / ロール / 他の AWS アカウント** への侵入
  → Backend の Task Role は `iam:*` も `sts:AssumeRole` も持たず、自分の Cognito / DynamoDB / Bedrock 以外に手が届かない
- 他の Cognito User Pool や SaaS サービスへの横移動
  → 保持しているのは自分が立てた User Pool 1 つだけ

**つまり最悪でも Bedrock 課金事故で止まる**、という設計方針。「SSO ごと乗っ取られる」は構造的に起こらない。これが Vouch by STS 方式 (JWT ではなく AWS 署名に乗っかる方式) の本質的な利点で、普通の OIDC proxy だと IdP の refresh token を Backend が保持するので同じ RCE でも IdP アカウントまで到達しうる。

詳しい攻撃モデルの分解は以下の details にまとめた。

::::details Vouch by STS 経路のセキュリティについて

「SSO login 済みのシェルから叩いた署名を Backend が STS に転送して身元確認」という仕組みは、聞くと「それ本当に大丈夫なん？( ；´Д｀)」となるので、何を前提にどこで守っているのかを整理しておく。

### そもそも何を信用しているか

Backend は JWT も password も見ない。**「このリクエストの主は Arn:xxxx である」と保証しているのは AWS STS 自身**で、STS の署名検証を借りてきているだけである。これは HashiCorp Vault の AWS auth method が 10 年以上使ってきた確立パターンで、署名鍵の rotation 問題も起きない(AWS 側で勝手にやってくれる)。

参考: [HashiCorp Vault - AWS Auth Method](https://developer.hashicorp.com/vault/docs/auth/aws)

なお「Vouch by STS」は本稿の便宜上の呼称であり、Vault 公式の用語ではない。

### 攻撃モデルと実装の防御

| 攻撃 | 実装の防御 |
|---|---|
| SSRF (Backend を踏み台に内部を叩く) | `https://sts.*.amazonaws.com` allowlist、`scheme=https` 強制 |
| 古い署名の再利用 (replay) | `X-Amz-Date` が ±5 分以内のみ受理 (nonce は未実装、今後の宿題) |
| 任意 action の差し込み | `Action=GetCallerIdentity` 以外は 400 |
| GET 署名による偽装 | `method=POST` 強制 |
| 他アカウントユーザの紛れ込み | `TrustedAccounts` allowlist、未登録アカウントは 403 |
| EC2 Instance Profile 乱用 | デフォルト DENY、明示的に `allow_instance_profile: true` を設定しないと通らない |
| 任意 role で入り込む | `allowed_role_patterns` (fnmatch) で絞る |
| session_name 偽装で別人なりすまし | session_name は STS が返す値 = AWS が署名保証するので任意には書けない |
| iam_user の無秩序 provision | iam_user は常に招待必須、auto_provision 対象外 |

### 残っているリスクと運用上の注意

1. **Replay の 5 分窓** — 同じ署名済みリクエストを 5 分以内に複数回送ると通る。盗聴できる環境（TLS 終端プロキシ、マルウェア、ログ誤流出）では代理ログインが成立する。Vault は nonce オプションを提供しているが、Stratoclave は未実装。今後の TODO。
2. **Cognito に permanent password が残る設計** — SSO ユーザでも `admin_initiate_auth` を通すため permanent password を設定している。24 文字ランダム全文字種混在なので推測は困難だが、bruteforce / password spray を気にするなら Cognito Account Takeover Protection を併用するのがよい。
3. **auto_provision モードでの session_name 信用** — `provisioning_policy=auto_provision` にすると session_name から email を抽出してそのまま Cognito user にする。IdP 側で session_name を email 相当に強制していないと、`x@evil.com` のような文字列を仕込まれて別人になりすませる余地がある。**デフォルトの `invite_only` なら email は招待側の値を使うのでこの攻撃は不成立**。
4. **Backend compromise 時の影響範囲** — Backend が Cognito `admin_*` 権限を持つので、最悪ケースでは Stratoclave 全ユーザ成り代わり + Bedrock 過剰利用 + DynamoDB 改竄が起こりうる。**ただし影響は Stratoclave という箱の中に閉じる**。Backend の Task Role は `cognito-idp:Admin*` を自分の User Pool だけ、`dynamodb:*` を自分のテーブルだけ、`secretsmanager:*` と `ssm:*` を `${prefix}/*` 配下だけに限定しており、`iam:*` も `sts:AssumeRole` も `organizations:*` も一切持たない。Backend が RCE 等で完全に掌握されても、**組織の IAM Identity Center / 他の AWS サービス / 他の Cognito Pool には到達しない**。Stratoclave は IdP の refresh token を一切保持しないので、SSO 側への横移動経路が構造的に存在しない(これが Vouch by STS 方式の本質的な利点)。ECS task role の最小権限化、IMDSv2 強制、WAF、自前ビルドのイメージで押さえる。
5. **SSO 経路の DoS 耐性** — `/auth/sso-exchange` は無認証で、10 秒 timeout の STS 呼び出しが走る。CloudFront + WAF の rate limit で守るのが無難。

### まとめ

**「設計は Vault と同じで妥当。nonce と rate limit の強化が入れば普通の SaaS proxy と同等以上」** だと思う。多分。

::::details LiteLLM Proxy との比較

冒頭で「LiteLLM でええんちゃう？」とボケたのでちゃんと比較しておく。結論: **LiteLLM は汎用・エンタープライズ寄り / Stratoclave は Bedrock 特化・AWS ネイティブ**。

以下は 2026-04-27 時点の [LiteLLM 公式ドキュメント](https://docs.litellm.ai/docs/) 準拠。

### 対照表

| 観点 | LiteLLM Proxy | Stratoclave |
|---|---|---|
| 対応 provider | 100+ (OpenAI / Anthropic / Bedrock / Vertex / Azure / Gemini / Ollama 等) | **Amazon Bedrock のみ** |
| データストア | **Postgres 必須** + Redis 推奨 | **DynamoDB のみ** (サーバレス) |
| RBAC | 3 階層 (Proxy / Internal User / Team) + Global/Team/User/Key/Model 別 budget | 3 階層 (Admin / Team Lead / User) + Tenant スコープ |
| Virtual / API Key | `expires / max_budget / budget_duration / rpm_limit / tpm_limit / models` 指定可 | `expires` 指定可、スコープ narrowing、active 5 個上限、revoke 即時 |
| SSO / STS | **Enterprise ライセンス** (Okta / Entra ID / OIDC / SAML) | **OSS 本体に同梱** (Vouch by STS で `aws sso login` / `saml2aws login` / IAM Identity Center すべて) |
| デプロイ | Docker / Helm (Beta) / ECS / EKS / Cloud Run 等、推奨 4 CPU / 8 GB | AWS CDK v2 一発、Fargate |
| ライセンス | **デュアル** (MIT + Commercial)、SSO・監査ログ・モデル別 budget は Commercial | **Apache 2.0 単一**、全機能 OSS |

### Stratoclave を選ぶケース

- **AWS 中心で Bedrock しか使わない** / Claude 系中心の組織
- Postgres を足したくない、IAM / Cognito / CDK の範囲で収めたい
- **AWS SSO / saml2aws で認証したい** (LiteLLM では SSO は Enterprise)
- Claude Code / Claude Desktop Cowork のユースケースが主
- OSS ライセンスが単一 (Apache 2.0) で全機能利用できる必要がある

### LiteLLM を選ぶケース

- 複数 provider (OpenAI / Bedrock / Gemini 等) を同じ proxy 越しに使う
- 既に Postgres 基盤がある / RDBMS を許容できる
- `rpm / tpm` / 月次 budget rotate など細かい rate/budget 制御が必須
- Langfuse / Datadog など外部観測基盤に統合済
- コミュニティ規模と実績を重視
