## はじめに

**AI Gateway を作った。名前は `Stratoclave` で、汎用 AI Gateway として [LiteLLM](https://github.com/BerriAI/litellm) に正面から挑む OSS である。**

最初は「自分が使うための Amazon Bedrock 専用 proxy」として始めた。組織利用だと部門ごとにクレジットを割り当てて上限まで使わせたい、個人的にはプロジェクト単位で Tenant を分離して課金管理したい、というのが動機だった。名前は Strato が岩盤の上の地層、Conclave が秘密集会で、テナント分離をイメージしている。

作っているうちに立ち位置が変わった。Bedrock 専用のニッチ proxy に閉じこもるのではなく、**LiteLLM が構造的に持てないものを 3 つ持つ汎用 Gateway**として真正面から比較されにいく、という方針に振り切った。その 3 つは次の通りである。

- **原子的 hard-budget 課金台帳** — 単一の `TransactWriteItems` による pre-flight 予約で予算超過を受付時点で構造的に排除し、その台帳の二重計上ゼロという不変条件を Z3 で形式証明した。超過が起きないことは原子予約の設計保証、二重計上ゼロは Z3 の証明対象という線引きは以降も守る
- **正直な counterfactual Savings Certificate** — 「ルーティング助言に従っていたら幾ら安く（あるいは高く）なったか」を、同一 rate・同一トークンで recompute した監査可能な節約証明書
- **信頼境界** — 「どのモデルを選ぶかの判断」は外部の [vLLM Semantic Router](https://github.com/vllm-project/semantic-router) に委譲し、「実行と記録」だけを Gateway が握る、という判断と実行の分離

**主な機能**

- Tenant / User 単位のトークン使用量把握、クレジット管理（dollar pool + per-user token + per-model quota を原子的に pre-flight 予約）
- AWS ネイティブなインフラ（DynamoDB、ECS Fargate、CloudFront、ALB。Postgres も Redis も無し）
- Admin / Team Lead / User の **3 段 RBAC** と Tenant 隔離
- `stratoclave claude` で Claude Code を、`stratoclave codex` で codex を Stratoclave 越しに起動
- Anthropic SDK / OpenAI SDK からは base URL 差し替えだけ
- **SSO (AWS SSO / STS vouch) / API Key / Cognito User Pool** の 3 経路認証
- Claude Desktop の **Cowork (Gateway mode)** 対応

:::message
LiteLLM じゃダメなん、、？ → 多くの場合 LiteLLM で良い。ただし「その課金数字は正しいと証明できるか」が要件になる相手には、構造で勝てる領域がある。この記事はその領域と、**今どこまで実証できていて何がまだ言えないか**を誇張なしで書く。最後の「実証到達度」の節が肝である。
:::

## 3 つの武器

### 武器 1: 形式証明された原子的 hard-budget 課金台帳

AI を SaaS に組み込むと、推論は**変動費**になる。誰がその変動費を払い、どう上限を切るかが P&L の核心であり、これは本質的に分散システムの問題である。信頼境界をまたいで、課金対象イベントを、ちょうど一度だけ、正しいテナントに、否認不可能な形で記録する、という問題になる。

Stratoclave は `POST /v1/messages` の受付時に、テナントの dollar pool と per-user token quota と per-model micro-USD quota を、**単一の DynamoDB `TransactWriteItems` でモデル呼び出し前に予約する**。`ConditionExpression` 付きの optimistic concurrency control なので、並列 N 本が同時に上限を突き抜けようとしても、超過した書き込みは `TransactionCanceledException`（`ConditionalCheckFailed` 起因）で弾かれ、それを捕捉して `402` に変換する。クライアントがこれをバイパスする経路は存在しない。**hard limit** である。

これに対し LiteLLM の予算は best-effort である。予算チェックは非同期に更新される spend 値をインメモリキャッシュから読むので、並行実行下では check-then-act (TOCTOU) の窓が残りうる（LiteLLM v1.x 系のドキュメントおよび実装に基づく。挙動はバージョンで変わりうる）。実用上は多くのワークロードで十分だが、これは **soft ceiling** であって、超過が金銭事故になる要件には構造的に足りない。counter 前提で組まれた全モジュールを書き直さない限り、原子的 pre-flight 予約は後付けできない。

さらに Stratoclave の課金台帳は event-sourced で、reserve / settle / release / reclaim / late-settle の各 money move が、それを動かす counter と同じ `TransactWriteItems` の中で 1 件の immutable event として書かれる。この台帳の二重計上ゼロという不変条件は **Z3 で形式証明済み** である。加えて、内部で進めている PENDING protocol への移行（恒久的な二経路フラグを消すリファクタリング）は、旧経路と新経路が金を同一に動かすことを **Z3 の joint-transition 等価証明** で示してから旧経路を削除する、という手順を踏んでいる。動いてそうだから消すのではなく、等価だと証明できたから消す、という順序である。

クラッシュ耐性も台帳に組み込んである。pooled な予約は同じ原子的書き込みの中に sibling の hold 行を書き、settle / release がそれを削除する。reserve と settle の間でタスクが落ちても（OOM、デプロイ drain）、後続リクエストの reserve 経路で走る inline lazy sweep が、TTL を過ぎた孤児 hold を expiry 順に数件だけ回収してから予算判定を行うので、孤児 hold が pool を埋めたことが原因で 402 が返り続けるデッドロックにはならない。別プロセスの reaper もサーバ側タイマーも無く、DynamoDB 1 テーブルだけで完結する。ただし sweep は後続リクエストが駆動するので、以降トラフィックが来ないテナントでは期限切れ hold が残留し続ける（fail-closed 側なので金銭事故ではないが、その分の pool は解放が遅れる）。この一点は正直に書いておく。

:::message
形式証明はデプロイ環境に依存しない。だから「Z3 で money invariant が証明された台帳設計」は今すぐ言える。一方で「実 DynamoDB・実デプロイでの production-grade な動作」はまだ証明できていない（後述の実証到達度の節を参照）。この線引きを曖昧にした瞬間、正直さを売る製品として自己矛盾する。
:::

![所属 Tenant のメンバー一覧とクレジット](https://github.com/littlemex/stratoclave/raw/main/docs/screenshots/images/08_admin_tenant_detail.png)

到達度バッジ: 二重計上ゼロ不変条件は Z3 証明済み。charge-of-record は実 Bedrock トラフィックで通したがストレージは moto・ネットワークは in-process。台帳の hot-path 計測のみ実 DynamoDB/EC2 で、そこは p99 < 50ms 目標が未達。詳細は末尾の実証到達度の節にある。

### 武器 2: 正直な counterfactual Savings Certificate

passthrough な AI Gateway（LiteLLM 等）は「**何が課金されたか**」を忠実に記録する。モデル別 spend、トークン数、リクエストログ。これは本物の価値である。しかし構造的に出せない数字が 1 つある。コスト責任者が実際に問う次の反実仮想である。

> 「ルーターが別のモデルを助言したリクエストについて、もしその助言に**従っていたら**、この同じワークロードは幾ら安く、あるいは幾ら**高く**なっていたのか」

先に前提を一つ。本証明書の入力となる実トラフィックのルーティング助言は、現状 dark である（武器 3 末尾で詳述する）。以下はエンジンの設計と、checked-in ワークロードでの offline demo で検証済みの動作であって、本番トラフィックでの数字ではない。

その上で、Stratoclave の Savings Certificate はこの問いに答える。per-`(tenant, day)` で、billed モデルと advised モデルの両方を、**同一 rate snapshot**・**同一トークン**で recompute する。rate のずれもキャッシュの非対称も相殺されるので apples-to-apples になる。バイアスは必ず VSR に不利な保守側へ倒す。エスカレーション損失（ルーターがより高いモデルを勧め、従っていたら高くついたケース）は**隠さず net から減算する**ので、`net` は負にもなる。損失を見せられる証明書だからこそ、利益も信じられる。rate の version を刻印してあるので、同じ (tenant, day) は同じ数字に再現される。realized（実際に enact されたルーティング）だけが headline に入り、shadow の助言は別の potential 上限として分離され、決して合算されない。

そして重要なのは、この証明書が `quality measured: False` を出したまま**放置する**という設計である。Stratoclave は**コスト**の反実仮想は証明するが、ルーティングされたトラフィックが元のモデルと同じくらい**良かった**とは主張しない。測っていないからである。安くて質が落ちたモデルは節約ではなく、請求額が小さくなっただけの品質劣化である。品質の line は、**テナント自身の eval** が埋めるまで false のままにする。検証していないことを告げる Gateway は、コスト数字を信頼に足るものにしているのと同じ正直さを実演している。

LiteLLM はリクエストごとの spend はログするが、「どの安いモデルで足りたか」を「その同じリクエストの実 billed トークン」で価格付けして監査可能にする、decision と charge の join 契約を持たない。これは機能追加ではなくアーキテクチャの差である。

到達度バッジ: 証明書エンジンと offline demo は動く。ただし入力となる実トラフィックのルーティング助言は現状 dark で、実テナントの数字はまだ出ていない（武器 3 参照）。

### 武器 3: 信頼境界 — 判断は外、実行と記録は内

「どのモデルを選ぶか」というルーティングの**判断**は、意味理解の問題である。ここは自前で分類器を書くのではなく、外部の [vLLM Semantic Router](https://github.com/vllm-project/semantic-router)（複数の signal を束ねた分類器。signal 数はバージョンで変わるので参照先で確認してほしい）に委譲すると決めた。Stratoclave はその decision-only な `/api/v1/eval` に助言を問い合わせ、**予算と認可の制約のもとで**選ばれたモデルを自分の transport で予約・実行し、first-party の usage から settle する。SR が判断し、Gateway が課金する。原子的な台帳が唯一の charge of record であり続ける。

これを一文にすると「**判断は VSR のもの、実行と記録は Stratoclave のもの**」である。この分離があるからこそ、「ルーティング判断が本当に安くなったか」を**課金台帳に対して**検証できる。武器 1（証明された台帳）と武器 3（判断/実行の分離）が結線して初めて、武器 2（Savings Certificate）が成立する。LiteLLM にも stand-alone な VSR にも、この閉ループは作れない。

:::message alert
正直に書くと、この VSR 連携は **まだ dark である**。実装済みだが `/api/v1/eval` client が未配線で、有効化しても現状は何も起きない。だから武器 2 の実効性は、この seam の実証が済むまでは留保が付く。Savings Certificate のエンジン自体は動いていて offline demo も回るが、その入力であるルーティング助言を実トラフィックで受けるところは、これからである。
:::

## breadth（接続の広さ）をどう扱うか

LiteLLM の最大の強みは 100+ provider への対応である。ここは正直に、2 つの軸に分けて評価する。混同すると評価を誤る。

**接続の広さは LiteLLM が勝つ**。100+ adapter は URL のリストではなく、provider ごとの streaming の癖、tool-call スキーマの差、retry semantics、認証を何年もかけて吸収した資産である。「OpenAI 互換だから何でも繋がる」は 8 割正しいが、残り 2 割（Anthropic ネイティブ機能、Gemini マルチモーダル、provider 固有の tool 形状）が LiteLLM の本当の moat で、VSR seam はこれを自動では埋めない。ここは追いかけない。adapter を自前で量産するのは LiteLLM の再発明であり最悪手である。

**統合の深さは Stratoclave が勝つ**。Amazon Bedrock（Claude を converse で、GPT-5.x を bedrock-mantle 経由で）、自前 GPU（`served_by="vllm"` の transport seam で任意の OpenAI 互換 endpoint）、そのすべてが**同じ reserve / rating / settle と同じ Z3 証明済み台帳**を流れる。LiteLLM は多くの backend に**接続**するが、その billing と冪等性は best-effort である。任意の backend を「形式証明された charge of record」の下に束ねるのは、別カテゴリの能力である。

接続の広さでは負ける。その上で、VSR + OpenAI 互換 seam により Bedrock も自前サービングも任意の OpenAI 互換 backend も同一台帳に載せる、という別の軸で戦う。

## 移行コストの非対称性（ただし条件付き）

方向によって移行コストが非対称である、という主張がある。これは敵対的に検証すると**半分妥当、半分は我田引水**である。

妥当な部分。「best-effort 集計に原子的 hard-budget を後付けできない」は正しい。LiteLLM の budget は集計であり、`TransactWriteItems` 相当の pre-flight 原子予約と bypass 不可を足すには課金経路の再設計が要る。方向の非対称性自体は構造的事実である。

我田引水な部分。「LiteLLM → Stratoclave の吸収は薄い shim で数週」は過小見積もりである。LiteLLM ユーザーが依存しているのは API 形状ではなく、100+ provider・Langfuse/Datadog callback・guardrails hooks・prompt caching であり、Stratoclave はそのどれも持たない。shim で吸収できるのは「LiteLLM の使い方が Bedrock 中心で機能を薄く使っている」場合だけである。だから正直に絞ると、こう言える。**Bedrock 中心で LiteLLM を薄く使っている利用者なら shim で移行可能だが、逆方向は LiteLLM が課金経路を全面的に書き直さない限り不可能である**。ここで言う逆方向とは、Stratoclave の証明済み台帳を LiteLLM に足すことを指す。

## 誰がどれを選ぶべきか

3 者（Stratoclave / LiteLLM / AWS の credential broker）は、本質的に別のものを売っている。

- **Stratoclave** は「その課金数字は正しいと証明できる」を売る。data path に入る AWS ネイティブ Gateway で、売り物は**正しさの保証**である。
- **LiteLLM** はあらゆる provider を 1 つの API に束ねる汎用 proxy で、売り物は**接続の広さと機能量**である。
- **AWS credential broker** すなわち Guidance for Claude Code with Amazon Bedrock は proxy を排し、IdP → STS 連携でクライアントに Bedrock を直接叩かせる参照実装で、売り物は**経路に居ないこと自体**である。ゼロレイテンシかつゼロ運用面という価値になる。

セグメント別の第一候補はこうなる。

- **社内で Claude Code を安全に配りたいだけ** なら credential broker。data path に何も足さず、MDM 配布・IdP 連携・IAM allow/deny で十分である。予算が事後に見えれば良いなら proxy を運用する理由がない。
- **テナントに従量課金する SaaS / 予算超過が金銭事故になる組織** なら Stratoclave。best-effort 集計でも事後カウンタでもなく、pre-flight 原子予約が要件そのものだからである。ここは構造の差で、機能追加では埋まらない。
- **多 provider を束ねたい・OpenAI/Vertex/Azure 混在** なら LiteLLM。議論の余地はない。

つまり 3 武器が本当に効くのは、**課金数字が charge-of-record であること自体が価値を持つ相手**である。課金を顧客に転嫁する SaaS、財務監査を受ける enterprise、FinOps が推定値を拒否する組織がそれにあたる。逆に言えば、数字を信用する必要がない人には効かない。

## 動かしてみる — Admin ログインから使い始めるまで

ここからは **IaC デプロイは完了している前提**で、Admin が管理画面を使い始めるところからの雰囲気を示す。全手順の網羅ではない。

まずブラウザで CloudFront URL を開くと Cognito の Hosted UI 経由でサインイン画面が出る。

![サインイン画面](https://github.com/littlemex/stratoclave/raw/main/docs/screenshots/images/01_login.png)

Admin でサインインするとダッシュボードに遷移し、上部ナビに「ユーザー管理」「テナント管理」「全体使用量」「信頼アカウント」が出る。ユーザー管理から email と role と初期クレジットを指定すればユーザーが払い出される。

![Admin ダッシュボード](https://github.com/littlemex/stratoclave/raw/main/docs/screenshots/images/02_dashboard.png)

ローカルで CLI をビルドして URL に向けると使い始められる。

```bash
# clone してビルド (Rust stable)
git clone https://github.com/littlemex/stratoclave.git
cd stratoclave/cli && cargo build --release
export PATH="$PWD/target/release:$PATH"

# デプロイ先を教える (well-known から設定を自動取得)
stratoclave setup https://xxxxxxx.cloudfront.net

# admin ログイン (Cognito password / AWS SSO どちらでも)
stratoclave auth login --email admin@stratoclave.local

# Claude Code を Stratoclave 越しに呼ぶ (Claude Code 本体は別途インストール)
stratoclave claude -- "このリポジトリを一言で要約して"
```

内部で `ANTHROPIC_BASE_URL` + `ANTHROPIC_API_KEY` が注入され、Claude Code が Stratoclave を通して Bedrock を呼ぶ。呼び出しごとにトークン数が記録され、クレジット残量が減る。使用履歴はモデル別・日別で参照できる。

![自分の使用履歴](https://github.com/littlemex/stratoclave/raw/main/docs/screenshots/images/03_me_usage.png)

API Key を発行すれば Anthropic SDK / OpenAI SDK からそのまま叩ける。プレーンテキストは一度だけ表示し、サーバには SHA-256 ハッシュのみ保存する。

![API キー一覧](https://github.com/littlemex/stratoclave/raw/main/docs/screenshots/images/12_me_api_keys.png)

Claude Desktop の Cowork（Gateway mode）にもこの長期 API キーで接続できる。base URL には `/v1` を含めない点だけ注意する（cowork が内部で `/v1/` を自動 prefix するため、入れると `/v1/v1/...` に飛んで 404 になる。CloudFront Function 側でも guard している）。

## セキュリティ — Vouch by STS

認証は 3 経路ある。Cognito email/password、長期 API Key（`sk-stratoclave-*`）、そして **Vouch by STS** である。

Vouch by STS は、AWS 短期 credentials を持っていることを身元の証拠として使う方式で、[HashiCorp Vault の AWS auth method](https://developer.hashicorp.com/vault/docs/auth/aws) と同じパターンである。CLI が `sts:GetCallerIdentity` を SigV4 で署名し、Backend がそれを STS に転送して身元確認する。`aws sso login` でも `saml2aws login` でも IAM Identity Center でも、`~/.aws/credentials` を populate する方式なら同じコードパスで通る。つまり IdP 連携で得た AWS credentials をそのまま身元証拠にできるので、AWS 環境の組織なら IdP 統合を別途実装せずに SSO 相当のログイン体験になる。これは LiteLLM の Enterprise SSO（admin UI への OIDC/SAML 統合）とは実現方式が異なる別物である点は正直に断っておく。

この方式の本質的な利点は blast radius にある。Backend は JWT も IdP の refresh token も一切保持しないので、Backend が RCE 等で完全に乗っ取られても、**箱の外への横移動は構造的に遮断される**。具体的には次が起こらない。

- 組織の IAM Identity Center / AWS SSO / Okta / Entra ID の乗っ取り（IdP の token を保持していない）
- 組織の IAM ユーザ / ロール / 他の AWS アカウントへの侵入（Task Role は `iam:*` も `sts:AssumeRole` も持たない）
- 他の Cognito User Pool や SaaS への横移動（保持しているのは自分の User Pool 1 つだけ）

正直に書くと、遮断されるのはあくまで横移動であって、箱の中の被害は閉じ込め対象ではない。完全 RCE を前提にすれば、Task Role の Bedrock 権限上限までの課金、台帳そのものの改ざん、自分の User Pool 内の全テナントの侵害は起こりうる。これが Stratoclave の blast radius の定義である。「その課金数字は正しいと証明できる」という主張も、台帳を書くコードが乗っ取られた瞬間からは無効になる。ここを「課金事故が箱に閉じる」と丸めるのは、この記事が最も嫌うべき矮小化なので、そうは書かない。台帳側の防御としては各 money move が immutable event として追記される event-sourced 設計なので、改ざんは検知の対象にはなる（ただし RCE 下での完全な否認防止は別問題である）。

比較として、普通の OIDC proxy だと IdP の refresh token を Backend が保持するので、同じ RCE でも IdP アカウントまで到達しうる。横移動が構造的に止まるという一点が、Vouch by STS 方式（JWT ではなく AWS 署名に乗っかる方式）の差である。

残っているリスクも正直に書いておく。replay の 5 分窓（`sso-nonces` テーブルで nonce 消費して塞いでいるが、Vault 相当の nonce オプションは元々の設計課題だった）、`auto_provision` モードでの session_name 信用（デフォルトの `invite_only` なら不成立）、無認証の SSO exchange endpoint の DoS 耐性（CloudFront + WAF の rate-based rule は IP 単位なので分散攻撃には緩和にとどまる）、署名済み `GetCallerIdentity` の転送先 STS エンドポイント検証（偽 STS への誘導対策。Vault AWS auth と同型の既知の攻撃面）、origin 直撃の遮断可否。これらの詳細は repo の `SECURITY.md` と `docs/ARCHITECTURE.md` にある。

## 実証到達度 — 今どこまで言えて、何がまだ言えないか

ここが本記事で一番大事な節である。それぞれの主張について、**今日存在する最も強い証拠**と、**同じ可視性でその証拠が何をカバーしていないか**を並べる。この線引きを正直に出すこと自体を、信頼の武器にしたい。

証拠のレベルを次のように分けている。

「gateway-live」のような一次元のラベルだと「live なのに moto とは何だ」と壊れるので、証拠は 3 つの直交する軸で示す。トラフィックが実在するか、ストレージが実在するか、ネットワーク境界が実在するか、である。

| 軸 | 意味 |
|---|---|
| トラフィック実在性 | リクエストが実モデル（実 Bedrock）に届いて実コード経路を通ったか、それとも合成入力か |
| ストレージ実在性 | 台帳が実 DynamoDB か、moto モックか |
| ネットワーク実在性 | uvicorn + ALB + TLS を越えたか、in-process ASGI dispatch か |

証明系（Z3 / property test / 決定的 offline fold）はこの 3 軸のいずれにも乗らない別カテゴリで、ネットワークにもストレージにも依存しない。未着手のものは正直に unverified と書く。

主要な主張と到達度は次の通りである。

- **課金台帳が正しい** こと、すなわち原子的 reserve/settle と二重計上ゼロ: Z3 形式証明 + Hypothesis stateful でカバー。証明の形は、状態遷移系に対する帰納的不変条件として、非有界の全対称状態にわたり反例が存在しないこと（Z3 が `unsat` を返すこと）で示している。有界モデル検査ではない。ただし証明対象は実行可能モデルであって、実 DynamoDB そのものではない。3 軸のいずれにも乗らない証明系。
- **PENDING 移行が安全である** こと、すなわち golden ↔ pending が等価: Z3 の joint-transition 等価証明（両経路を同時に走らせ、金の動きが常に等しいことを帰納的に証明）+ Hypothesis 差分テスト。ただし live/soak トラフィックはこれから。証明系。
- **Gateway を経由した charge-of-record**: `/v1/messages` を auth → reserve → 実 Bedrock → settle → 台帳 と通し、Gateway が **$0.000492** を settle した。クライアント側推計は $0.000562 で、約 14% ずれる。これは gateway が実 usage（Bedrock が返す実 billed トークン）で settle するのに対し、クライアント側は送信前のトークナイザ推計で見積もるためで、乖離の向き自体が「gateway 側が正しい実測、クライアント側が推計」を示している。トラフィック実在（実 Bedrock）だが、ストレージは moto、ネットワークは in-process ASGI である。
- **Gateway の TTFT / overhead が測れる**: gateway 経路 TTFT p50 **2384ms** vs direct p50 **2089ms**、paired overhead 中央値 **248.7ms**。ただし N=10 の点推定で分布も CI も主張しない。この overhead は auth + reserve + ASGI dispatch だけを測っており、ネットワーク/ALB/TLS は含まない。トラフィック実在だが in-process。
- **台帳が hot path に載るか** という論点、目標は p99 < 50ms: これは達成できていない。実 DynamoDB/EC2 での実測が end-to-end authorize p50 **57ms**、`TransactWriteItems` p50 **20ms** で、p50 の時点で既に 50ms を超えているので p99 < 50ms は原理的に届かず（参考まで p99 は 225ms）、チューニングでは埋まらない。single-item CAS / pool sharding という設計変更が前提になる。ここだけはストレージもネットワークも実在する direct-baseline（実 DynamoDB/EC2）だが、gateway 全体を通した数字ではなく台帳単体の計測である。
- **Savings Certificate**: 実エンジンを checked-in ワークロードで回した offline demo。ただし synthetic ワークロードで、実テナントの数字にはそのテナントのトラフィックが要る。offline の証明系。

正直な国境線をもう一度だけ、はっきり書く。台帳の hot-path 計測だけは実 DynamoDB/EC2 だが、それ以外の live 数字（charge-of-record、TTFT、overhead）は**トラフィックは実 Bedrock、ストレージは moto、ネットワークは in-process ASGI** である。Gateway の**コード経路と課金精度**は実トラフィックで動いたが、**実 DynamoDB** 台帳を**デプロイ済み**経路（uvicorn + ALB + TLS）で通した end-to-end は未検証である。だから overhead の 248.7ms は auth + reserve + ASGI dispatch を測っているのであって、production の latency SLO ではない。これらの live 数字は N=10〜30 の点推定で、p50 より上の percentile は名指ししない。唯一 p99 を出しているのは実 DynamoDB/EC2 の台帳計測（別系列）で、そこは目標未達を隠さないために敢えて tail まで出している。

**charge-of-record を武器として売り出すのは今のエビデンスで妥当か**。主張の構成次第である。「Z3 で money invariant が証明された台帳設計」は今すぐ言える。「production-grade の charge-of-record gateway」はまだ言えない。前者で売り、後者を約束しない。この線を守れば、証明済みと未検証の線引きそのものが信頼の武器になる。

## まとめ — 一番効く 1 メッセージ

誇張なしで一番効く言い方はこうなる。

**LiteLLM の予算は事後集計の soft ceiling、Stratoclave の予算は pre-flight の原子予約による hard limit。超過してから気づく数字か、超過が受付で拒否される数字か、どちらが要件かで選べ。**

多くの人にとっては LiteLLM で良い。だが「その課金数字は正しいと証明できるか」が契約要件になる相手には、Stratoclave は構造で戦える。そして今できていることと、まだできていないことを、この記事の実証到達度の節のように正直に出し続けることが、この製品の一貫性そのものである。
