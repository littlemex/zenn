## はじめに

MoE (Mixture-of-Experts) モデルを複数 GPU に分散させると、各トークンを「担当の expert が載っている GPU」へ送り、計算後にまた元へ戻す **all-to-all 通信**が発生します。この送り (dispatch) と戻し (combine) は、MoE の分散実行で最大の通信ボトルネックになります。

この dispatch/combine を高速化するライブラリとして [DeepEP](https://github.com/deepseek-ai/DeepEP) が広く使われていますが、DeepEP は **NVIDIA GPU + NVIDIA NIC** の組み合わせに強く依存しており、**AWS EFA (Elastic Fabric Adapter) では動きません**。

本記事で扱う [UCCL-EP](https://github.com/uccl-project/uccl/tree/main/ep) は、この dispatch/combine を **EFA を含む多様な NIC で動かせるようにした**ライブラリです。本記事では、

- なぜ MoE の all-to-all が EFA で問題になるのか
- DeepEP と UCCL-EP は何が違うのか (設計の肝)
- EFA で動かすために何を工夫しているのか

を初学者向けに整理し、最後に **AWS の p5en (H200) 2 ノード = 16 GPU で UCCL-EP のマイクロベンチを実際に動かした結果**を示します。

:::message
本記事の実機検証は **EP16 (16 GPU) のマイクロベンチのみ**です。UCCL 公式ブログの主要な数値は EP32 (32 GPU) で測られており、規模が異なるため直接比較はしません。また slime/SGLang などの実ワークロードへの組み込みは本記事では検証していません。詳しくは「[検証の範囲と限界](#検証の範囲と限界)」を参照してください。
:::

## 用語の整理: 「GPU 起点」には 2 種類ある

本記事の理解でいちばん大事なのが、「GPU 起点 (GPU-initiated) の通信」という言葉に **2 つの異なる意味がある**ことです。ここを最初に区別しておかないと、DeepEP と UCCL-EP の違いが分からなくなります。

- **真の GPU-initiated (IBGDA 方式)**: GPU のスレッドが、NIC に送信を指示する作業 (WQE = Work Queue Element の書き込みと、doorbell というレジスタ書き込み) まで**自分で**行います。CPU は一切介在しません。DeepEP が NVSHMEM/IBGDA を通じて使うのがこれです。
- **GPU-initiated / CPU-driven (UCCL-EP 方式)**: GPU は「このトークンを送れ」という**小さなコマンドを積むだけ**で、実際に NIC を叩く作業は **CPU 側のプロキシスレッド**が肩代わりします。

どちらも「通信のきっかけを GPU が作る」点は同じですが、**NIC を最終的に誰が叩くか**が決定的に違います。この違いが、そのまま「どの NIC で動くか」を決めます。

:::message
CPU が NIC を代行する「プロキシ方式」自体は新しいものではありません。NVSHMEM にも古くから CPU プロキシを使う IBRC 転送があります。UCCL-EP の新しさは、プロキシを使うこと自体ではなく、**トークン単位の細かいコマンドを大量にさばけるよう、ロックフリーの FIFO・マルチスレッド・pacing で高速化した**点にあります (後述)。
:::

## MoE の dispatch / combine とは

前提記事「[MoE 推論を理解する](https://zenn.dev/tosshi/articles/431977b09a79bd)」で見たように、MoE では Router が各トークンに対して「どの expert に送るか」を Top-K 個選びます。expert parallelism (EP) では、この expert 群を複数 GPU に分散配置します。すると、

1. **dispatch**: 各 GPU が持つトークンを、Router の選択に従って「担当 expert が載っている GPU」へ送る (all-to-all)
2. 各 expert がトークンを処理する
3. **combine**: 処理結果を元のトークンの持ち主 GPU へ戻す (all-to-all)

という流れになります。dispatch と combine はどちらも「全 GPU が全 GPU へ、それぞれ違う量のデータを送る」典型的な all-to-all で、EP の規模 (EP8, EP16, EP32...) が大きいほど通信相手が増え、レイテンシと帯域の両方が効いてきます。

UCCL-EP には、この all-to-all に **2 つのモード**があります。本記事の実測でも両方を測るので、先に押さえておきます。

- **normal モード**: 大きなバッチをまとめて送る、**スループット重視**の経路。学習や prefill 的な大量トークン処理に向く。
- **low-latency モード**: 少数トークンを**とにかく低レイテンシ**で送る経路。推論の decode のように 1 ステップあたりのトークンが少ない状況に向く。

## DeepEP のアプローチと限界

DeepEP は、dispatch/combine を **トークン単位の GPU 起点 RDMA** で実装します。GPU のスレッドが、NVSHMEM を経由して IBGDA (InfiniBand GPUDirect Async) で NIC を直接叩き、トークンを相手 GPU のメモリへ RDMA write します。CPU を介さないため計算と通信のオーバーラップが効き、非常に高速です。

問題は、この **IBGDA が NVIDIA GPU + NVIDIA (Mellanox/ConnectX) NIC 専用**である点です。IBGDA は GPU から NIC のドアベル (UAR) を直接叩き、WQE バッファも GPU メモリに置きます。この実装は NVSHMEM が mlx5 の低レベル API (DEVX / `mlx5dv`) を前提に作られているため、事実上 ConnectX 系 InfiniBand/RoCE でしか動きません。EFA は QP の処理が Nitro カード側にあり、こうした低レベル操作をユーザー空間に露出しないため、IBGDA は使えません。

その結果、DeepEP は次のような環境では動きません。

- **AWS EFA** (パブリッククラウドの多くが採用)
- AMD GPU
- Broadcom など他ベンダの NIC

そして、GPU ベンダが m 種類、NIC ベンダが n 種類あると、素朴には **m × n 通りの移植**が必要になってしまいます。

## UCCL-EP のアーキテクチャ

UCCL-EP は「**関心の分離**」でこの問題を解きます。

- **GPU 側**: トークン単位の細かい通信の**起動だけ**を担当します。具体的には 128-bit の `TransferCmd` (「どのトークンをどこへ」を表す小さなコマンド) を、ロックフリーの FIFO キューに積みます。GPU はここまでで、あとの計算にすぐ戻れます。
- **CPU 側**: マルチスレッドの**プロキシ**が FIFO からコマンドを取り出し、キュー管理・フロー制御・順序制御をこなしながら、**`libibverbs` (verbs) という移植性の高い API で NIC を叩きます**。

この分離のポイントは、NIC ごとの差異を「verbs を喋る CPU プロキシ」で吸収できることです。NIC を変えても GPU カーネルは変えなくてよいので、移植の労力が **m × n から m + n に**下がります (GPU ベンダ対応と NIC ベンダ対応が掛け算でなく足し算になる)。

:::message
「移植労力が m + n になる」は、NIC 側の対応が完全に不要という意味ではありません。実際、後述するように EFA 対応のためには「SRD の順序保証がない」問題への追加実装が必要でした。verbs で NIC 差の大半を吸収しつつ、NIC 固有の癖には個別対応が要る、というのが正確な理解です。
:::

### なぜ CPU を挟んでも速いのか

CPU プロキシを挟むと、素朴には「GPU→CPU の受け渡しが遅くなるのでは」「CPU コアを食うのでは」という懸念があります。実際、プロキシスレッドは FIFO をビジーポーリングするので CPU コアを消費します (UCCL-EP のデフォルトは GPU あたり 4 スレッド)。

それでも実用的な性能が出る理由は、**FIFO を流れるのは 128-bit の小さなコマンドだけ**で、**データ本体 (トークンの中身) は GPU メモリから直接 RDMA される**からです。公式ブログによれば FIFO は毎秒 800 万オペレーション以上、平均レイテンシ約 3 us・P99 8 us 未満をさばけるとされ、これはネットワーク自体のレイテンシ (EFA で 20〜50 us 以上) に比べれば無視できる、という設計になっています。

### EFA で verbs が使えるのはなぜか

前提記事「[AWS EFA と Nitro System の仕組みを理解する](https://zenn.dev/tosshi/articles/0eeb53ca63f8b2)」では、EFA を主に **libfabric (OFI)** の下のプロバイダとして説明しました。ここで「UCCL-EP は verbs を使う」と聞くと、「EFA は libfabric では?」と混乱するかもしれません。

答えは、**EFA には rdma-core の `efa` プロバイダがあり、SRD の Queue Pair を verbs (`libibverbs` / `efadv`) で直接叩ける**、です。libfabric は verbs のさらに上の抽象化で、EFA は「libfabric 経由」でも「verbs 直叩き」でも使えます。NCCL は前者 (aws-ofi-nccl → libfabric → EFA) を、UCCL-EP は後者 (verbs → EFA) を使う、という違いです。

### DeepEP と UCCL-EP の対応関係

3 つのライブラリの関係を 1 枚の表にすると、こうなります。

| 層 | DeepEP (NVIDIA 専用) | UCCL-EP (EFA など) |
|----|---------------------|-------------------|
| API | `deep_ep` | `deep_ep` 互換 |
| 通信の起動 | GPU が WQE + doorbell まで (NVSHMEM/IBGDA) | GPU は FIFO にコマンド、CPU プロキシが verbs で post |
| NIC | ConnectX 系 (IBGDA 必須) | EFA / verbs が動く NIC |

:::message
UCCL-EP は **NVSHMEM の汎用的な代替ではありません**。NVSHMEM は PGAS 全般を担う汎用ライブラリですが、UCCL-EP は「DeepEP の下回り (NVSHMEM/IBGDA の部分) を差し替えて、MoE の dispatch/combine を portable にした EP 特化の実装」です。API も DeepEP (`deep_ep`) 互換に作られています。
:::

## EFA で動かすための工夫

UCCL-EP を EFA で動かすうえで、いちばんの壁が **SRD の順序保証がない**ことです。前提記事で見たように、EFA の SRD (Scalable Reliable Datagram) は到達を保証しますが、**パケットの到着順序は保証しません**。all-to-all で受信側が「どのトークンがどの位置か」を正しく組み立てるには、順序の情報が要ります。

UCCL-EP はこれを、**RDMA write の immediate data (write に相乗りさせられる 32-bit の付加データ) に、チャネルごとのシーケンス番号を埋め込む**ことで解決します。受信側は immediate data の番号を見て、GPU に確定させる前に正しい順序へ並べ替えます。完了通知も別メッセージにせず、通常の RDMA write に相乗り (piggyback) させます。

もう 1 点、DeepEP は完了フラグやカウンタの更新に **NIC の atomics** を使いますが、**EFA は RDMA atomics に対応していません**。UCCL-EP は CPU プロキシがこれをソフトウェアで代替 (エミュレート) することで、atomics の無い NIC でも同じ意味論を成立させています。

加えて CPU プロキシが、送信中のリクエスト数を追跡して pacing (流量制御) を行い、複数の NIC / Queue Pair に負荷を分散します。これにより、順序保証や atomics の弱い NIC でも性能が崩れないようにしています (Broadcom Thor-2 や AMD Pollara などでの崩壊回避。詳細は [論文](https://arxiv.org/abs/2512.19849))。

:::message
EFA の「RDMA write with immediate」対応は世代・ファームウェアに依存します。本記事で検証する p5en (EFA v3 世代) では利用可能です。
:::

## 使い方 (DeepEP の drop-in 代替)

UCCL-EP は DeepEP と同じ `deep_ep` インターフェースを提供するため、SGLang や vLLM のような既存フレームワークからは、import を差し替えるだけで使える設計になっています。

:::message
「drop-in」とはいえ、normal / low-latency 両モード、FP8 dispatch、hook ベースのオーバーラップなど、全機能が完全互換かは各自の検証が必要です。本記事では主要 API でマイクロベンチが動くことまでを確認します。
:::

ビルドについては、次の実機検証の節で、EFA + GPUDirect を有効にするための実際のハマりどころとあわせて説明します。

### ハマりどころ 1: prebuilt wheel は EFA + GPUDirect で動かない

UCCL は PyPI に prebuilt wheel (`pip install uccl`) を公開しています。NGC の PyTorch イメージ (Python 3.12 / CUDA 12.8) にそのまま入り、`from uccl import ep` も通ります。ところが、この wheel で 2 ノードの `test_internode.py` を走らせると、UCCL の CPU プロキシ初期化 (`per_thread_rdma_init`) の段階で次のエラーを出してクラッシュしました。

```text
RDMA buffer MR registration failed: Operation not supported
[RDMA] main RDMA buffer probe failed on NIC rdmap135s0 (bytes=67108864, errno=95: Operation not supported)
```

`errno=95` は `EOPNOTSUPP` です。これは「**GPU メモリを RDMA の Memory Region (MR) として登録できない**」ことを意味します。切り分けると、原因は GPUDirect RDMA の GPU メモリ登録経路にありました。

前提記事「[GPUDirect RDMA 入門](https://zenn.dev/tosshi/articles/42f0ee03b328a4)」で触れたように、NIC が GPU メモリを直接読み書きするには、GPU メモリを RDMA に登録する仕組みが要ります。verbs から見た登録方法は大きく 2 つあります。

- **従来の `ibv_reg_mr` で GPU メモリを登録する経路**: GPU メモリの物理ページを NIC に見せるために、**カーネル側の「peer-memory 機構」**が必要です。代表が NVIDIA の `nvidia_peermem` で、これは GPU ベンダ固有の追加モジュールです。
- **dma-buf 経路 (`ibv_reg_dmabuf_mr`)**: GPU メモリを dma-buf としてエクスポートし、その fd を NIC に渡します。**GPU ベンダ固有の peer-memory 機構に依存しません**。

検証ノード (Amazon Linux 2023 + NVIDIA Open Kernel Module 580 + EFA 3.1) を調べると、**GPU メモリの `ibv_reg_mr` 登録を仲介する peer-memory ブリッジがロードされていませんでした** (`nvidia_peermem` も、efa の nv-peermem 連携も見えない)。一方で GPU の dma-buf エクスポート自体は有効 (`CU_DEVICE_ATTRIBUTE_DMA_BUF_SUPPORTED = 1`) でした。同居する NCCL が EFA で GPUDirect を効かせられているのは、NCCL (aws-ofi-nccl → libfabric) が **dma-buf 経路**を使うためと考えられます。ところが **prebuilt wheel の UCCL は `ibv_reg_mr` 経路のみ**でビルドされているため、ブリッジの無いこのノードでは登録に失敗していました。

**なぜ prebuilt wheel は動かない経路を選ぶのか?** 「wheel がわざと悪い経路を選んだ」わけではなく、**wheel をビルドした環境の制約**です。dma-buf 経路 (`ibv_reg_dmabuf_mr`) をコンパイルするには、ビルド環境の rdma-core が比較的新しい必要があります (加えて実行時も CUDA 11.7+ / 新しめの kernel が要る)。それが揃っていないビルド環境では、dma-buf 経路が有効化されず、伝統的な `ibv_reg_mr` 経路だけが焼き込まれます。`ibv_reg_mr` で GPU メモリを掴めるかは**実行するノード側のカーネル機構次第**なので、機構が載っている環境なら動き、載っていない今回のような EFA ノードでは動かない、という差が出ます。

:::message
「peermem が無いから失敗する」と単純化すると誤解を生みます。正確には「`ibv_reg_mr` で GPU メモリを掴むにはカーネル側の peer-memory 機構 (mlx5 なら `nvidia_peermem`) が要り、この環境ではそれが無効だった」であり、`nvidia_peermem` をロードすれば直るとは限りません (そもそも EFA は `nvidia_peermem` を使いません)。**根本解決は、この機構に依存しない dma-buf 経路に切り替えること**です。
:::

補足すると、両者の違いは「GPU メモリを NIC が触れるようにする**仲介の仕組み**」にあります。`ibv_reg_mr` で GPU メモリを登録する経路は、カーネルの **peer-memory 機構** (代表が NVIDIA の `nvidia_peermem`) に頼ります。これは **GPU ベンダ固有**で、多くの環境では標準カーネルに含まれない追加モジュールです。この「標準外の追加モジュールへの依存」が移植性を下げる本質です。一方 dma-buf は、GPU が自分のメモリを共有可能な `fd` (ファイルディスクリプタ) としてエクスポートし、NIC ドライバがその `fd` を取り込む、という **Linux カーネル標準の仕組み**に乗ります (NIC ドライバ側にも取り込みの実装は要りますが、mlx5 も EFA も対応済み)。標準 API に乗る分だけ、dma-buf は移植性が高いのです。

実際、wheel の判定 API はこの状況を正しく検出します。

```python
from uccl import ep
ep.can_register_rdma_gpu_buffer(0, 64 << 20)   # => False (GPU メモリ登録不可)
```

### ハマりどころ 2: USE_DMABUF=1 でソースビルドする

UCCL のソース (`ep/src/rdma.cpp`) には、まさにこのための **dma-buf 経路** (`reg_mr_gpu_dmabuf`) が用意されています。ただしこの経路は **ビルド時フラグ `USE_DMABUF=1` を付けたときだけ**有効になり (`ep/setup.py` / `ep/Makefile`)、**PyPI の prebuilt wheel には含まれていません** (wheel のタグ `v0.1.1` の時点では `USE_DMABUF` の仕組み自体が無く、`main` で追加されています)。

したがって、peer-memory ブリッジの無い EFA ノードで UCCL-EP を GPUDirect で動かすには、`USE_DMABUF=1` を付けて**ソースからビルド**する必要があります。ビルドは GPU ノード上の NGC PyTorch イメージ内で完結できます。

:::message
dma-buf 経路は **NVIDIA Open Kernel Module + 比較的新しい kernel** を前提とします (GPU メモリの dma-buf エクスポートがこの組み合わせで提供されるため)。proprietary driver のノードでは `USE_DMABUF=1` でビルドしても、この経路は使えないことがあります。
:::

```bash
git clone https://github.com/uccl-project/uccl.git
cd uccl/ep
# H200 (sm_90) 単独に絞る。TORCH_CUDA_ARCH_LIST を明示しないと NGC の
# 多アーキ設定 (sm_70/75 など古い世代を含む) が使われ、sm_90 以降の TMA 命令で ptxas が失敗する。
USE_DMABUF=1 TORCH_CUDA_ARCH_LIST="9.0" python setup.py bdist_wheel
pip install dist/uccl-*.whl
```

ビルド時の主なハマりどころは 2 つでした。どちらも「なぜ起きるか」が分かると対処は単純です。

**(1) rdma-core が古いとコンパイルが通らない**

こういうエラーが出ます。

```text
error: 'struct efadv_qp_init_attr' has no member named 'sl'
error: 'EFADV_QP_FLAGS_UNSOLICITED_WRITE_RECV' was not declared in this scope
```

これは「UCCL のソースが使おうとしている EFA の新機能を、システムの rdma-core (EFA を verbs で叩くためのヘッダとライブラリを提供するパッケージ) がまだ知らない」ために起きます。`efadv.h` に現れる `UNSOLICITED_WRITE_RECV` (RDMA write with immediate の受信で recv WR を消費しない最適化) や、efadv の QP 属性に追加された `sl` (service level) メンバは **rdma-core v52 以降**で入ったものです。ところが Ubuntu 24.04 ベースの NGC イメージは **rdma-core 50** で、ヘッダもライブラリも揃って古い状態でした。

対処は、**rdma-core をヘッダとライブラリまとめて新しくする**ことです。ここが要注意で、`efadv.h` (ヘッダ) だけ差し替えるとコンパイルは通っても、リンク・実行時の `libefa.so` / `libibverbs.so` が古いままだと新機能が動きません (シンボルが無い、あるいは新しい属性が無効扱いされる)。今回は rdma-core v56 をビルドし、ヘッダとライブラリの両方を差し替えたうえで、実行するノードでも `LD_LIBRARY_PATH` で新しい `libibverbs`/`libefa` を優先させました。新しい [aws-efa-installer](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/efa-start.html) を使う場合も、EFA のユーザ空間一式が新しくなる点は同じです (カーネル側の EFA ドライバのバージョンも合わせて確認します)。

**(2) `TORCH_CUDA_ARCH_LIST` を絞らないと ptxas が落ちる**

アーキを指定しないと、こんなエラーが大量に出ます。

```text
ptxas ... error : Feature 'cp.async.bulk' requires .target sm_90 or higher
```

理由は「**ビルド対象の GPU 世代を広げすぎている**」ことです。CUDA のビルドは「どの GPU 世代 (compute capability) 向けにコードを生成するか」をリストで指定でき、NGC イメージはデフォルトで古い世代 (Volta/Turing など sm_70 / sm_75) から新しい世代まで幅広く含めています。ところが UCCL-EP の dispatch/combine カーネルは **sm_90 以降 (Hopper で導入)** の命令 (TMA = Tensor Memory Accelerator による一括転送 `cp.async.bulk` など) を使っているため、**古い世代向けにコンパイルしようとした瞬間に「その命令は sm_90 以上でないと使えない」と ptxas (PTX アセンブラ) が拒否**します。

対処は、実際に使う H200 (sm_90) だけを対象に絞ることです。`TORCH_CUDA_ARCH_LIST="9.0"` を明示すれば、古い世代向けの生成が止まり、エラーは消えます。

ビルド後、判定 API が `True` を返せば GPUDirect (dma-buf 経路) が使える状態です。

```python
from uccl import ep
ep.can_register_rdma_gpu_buffer(0, 64 << 20)   # => True
```

### 結果: dispatch / combine の帯域とレイテンシ

まず何より、**`nvidia_peermem` の無い EFA ノードで、トークン単位の dispatch/combine の all-to-all が正しく完走しました**。ベンチ起動時のチェックでも、GPU メモリの RDMA 登録が有効 (dma-buf 経路) であることを確認できています。

```text
[init] uccl.ep sanity (expect can_reg=True, host_allocated=False)
can_reg: True
```

#### normal モード (`test_internode.py`, 4096 tokens / hidden 7168 / top-8 / 256 experts)

`RDMA` 列がノード間 (EFA) の帯域、`NVL` 列がノード内 (NVLink) の帯域です。本記事で見たいのは EFA を流れる **RDMA 列**です。

| 操作 | RDMA (ノード間 EFA) | NVLink (ノード内) |
|------|---------------------|-------------------|
| dispatch (BF16) | **41.3 GB/s** | 134.7 GB/s |
| dispatch (FP8) | 38.9 GB/s | 127.1 GB/s |
| combine (BF16) | **18.0 GB/s** | 58.9 GB/s |

per-GPU の EFA は片方向 400 Gbps ≒ 50 GB/s なので、dispatch (BF16) の 41.3 GB/s は **NIC ラインレートの約 8 割**に達しています (ここでの帯域は「フェーズの経過時間で転送バイト量を割った実効値」で、ノード内 NVLink 転送とオーバーラップした上での値です)。EFA 上のトークン単位 all-to-all として十分実用的な水準です。

combine が dispatch より低い理由は、この検証だけでは切り分けていません (受信側での reduction、EFA が RDMA atomics 非対応のためソフトウェアで代替する完了管理、並べ替えコストなどが候補)。UCCL 公式ブログでも combine < dispatch の傾向は共通して見られます。

#### low-latency モード (`test_low_latency.py`, 128 tokens / hidden 7168 / top-8 / 256 experts)

推論の decode のように少数トークンを低レイテンシで送る経路です。

| 指標 | 値 (rank 0) |
|------|-------------|
| dispatch (FP8) | 20.2 GB/s, avg 372 us |
| combine (BF16) | 30.8 GB/s, avg 472 us |
| dispatch + combine 合計 | 16.6 GB/s, avg 1325 us |

少数トークンなので帯域自体は normal モードより低くなりますが、これは想定通りで、low-latency モードは「1 回あたりの遅延を抑える」ことに主眼があります。なお low-latency モードでは **dispatch が FP8、combine が BF16** (デフォルト) で、combine のペイロードが 2 倍になるため、ここでは combine の帯域が dispatch より高く出ています (normal モードの BF16 同士とは dtype 条件が異なる点に注意)。

:::message
上の表はすべて **本記事の EP16 実測値**です。UCCL 公式ブログは p5en の **EP32** で dispatch 54 GB/s・combine 43 GB/s を報告していますが、EP 規模が異なる (通信相手数が違う) ため、本記事の EP16 の値と**直接比較はできません**。「EFA でこのオーダーの帯域が実際に出た」という規模感として捉えてください。
:::

## 検証の範囲と限界

本記事の実機検証には、以下の限界があります。誠実に列挙しておきます。

- **EP16 まで**。検証環境は p5en 2 ノード = 16 GPU なので、EP16 が上限です。UCCL 公式ブログの主要な数値は EP32 (32 GPU) で測られており、規模が異なるため本記事では**直接比較しません**。
- **マイクロベンチのみ**。dispatch/combine 単体の帯域・レイテンシを測っただけで、slime / SGLang / vLLM などの実ワークロードに組み込んだ end-to-end の性能は検証していません。
- **DeepEP との直接比較はしていない (できない)**。DeepEP は EFA では動かないため、同じハードウェア上に比較対象が存在しません。したがって「DeepEP より速い/遅い」という主張は本記事では行いません。示したいのは、あくまで「**EFA 上でトークン単位の dispatch/combine が正しく完走し、実用的な帯域・レイテンシが出る**」という一点です。
- NCCL の all-to-all を参考値として併記する場合も、これは「規模感の参考」であり、性能比較ではありません。

## まとめ

- MoE の expert parallelism では dispatch/combine の all-to-all が通信ボトルネックになり、DeepEP はこれを GPU 起点 RDMA で高速化しますが、**IBGDA 依存のため AWS EFA では動きません**。
- UCCL-EP は「GPU はコマンドを積むだけ、CPU プロキシが verbs で NIC を叩く」という関心の分離により、EFA を含む多様な NIC で dispatch/combine を動かせるようにしたものです。EFA の SRD が順序保証しない問題は、RDMA immediate data のシーケンス番号で解決しています。
- DeepEP / NVSHMEM / UCCL-EP の関係は「UCCL-EP は NVSHMEM の汎用代替ではなく、DeepEP の下回り (NVSHMEM/IBGDA) を差し替えて MoE 通信を portable にした EP 特化の実装」と捉えると整理できます。
- 実機では、**PyPI の prebuilt wheel は `ibv_reg_mr` 経路のみでビルドされており、GPU メモリを掴む peer-memory 機構が無いこの EFA ノードでは MR 登録に失敗します**。`USE_DMABUF=1` でソースビルドし直すことで、dma-buf 経路で GPUDirect を有効にできました。

「移植性 (portability)」は地味なテーマですが、DeepEP のような高性能実装を NVIDIA + ConnectX の枠の外 (パブリッククラウドの EFA、AMD GPU、他ベンダ NIC) に広げられる意義は大きく、UCCL-EP はそのための現実的なアプローチだと言えます。
