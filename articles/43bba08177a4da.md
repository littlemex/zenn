## はじめに

:::message
本記事は、強化学習ベースの LLM ポストトレーニング分野のフレームワーク 4 つを ML インフラストラクチャ視点で整理・比較したものです。
:::

:::message alert
日々アップデートされているため 2026/03 現時点の状況の整理です。違和感を感じた部分はチェックしてますが AI に調査させているのでミスの可能性があります。優劣をつける意図はなく（そもそもユースケースがないので優劣をつけられない）主に自分用のまとめなので参考程度にしてください。
:::

https://zenn.dev/tosshi/articles/251ab3faae2f47

LLM のポストトレーニング（RLHF、GRPO 等）を実施する際、フレームワーク選定は非常に重要です。以前の Verl のブログで書いた通り GRPO のようなポストトレーニングの Generation phase と Training phase の計算特性の違い（主に Generation Decode がメモリバウンド、Training が計算バウンド）をどう扱うか、対応アルゴリズム、大規模クラスタのスケールアウト戦略、そして使用するハードウェアへの対応状況、RDMA 等のデータ転送効率の実装状況、が選定の鍵だと思います。

本記事では以下の 4 フレームワークを比較します。

- **VeRL** (verl-project/verl): ByteDance が開発した HybridFlow アーキテクチャ採用の LLM ポストトレーニング専用フレームワーク
- **NeMo RL** (NVIDIA-NeMo/RL): NVIDIA が開発した Megatron-Core 統合の LLM ポストトレーニング専用フレームワーク
- **OpenRLHF** (OpenRLHF/OpenRLHF): コミュニティ主導の Ray + DeepSpeed ベースの LLM RLHF フレームワーク
- **OpenRL** (OpenRL-Lab/openrl): NLP タスクも対応するが、LLM 大規模ポストトレーニングに特化した設計ではなく汎用 RL フレームワーク

:::message alert
4 つのフレームワークは**用途・設計目標が異なります**。VeRL・NeMo RL・OpenRLHF は「大規模 LLM のポストトレーニング（RLHF/GRPO）」向けの設計ですが、OpenRL は単一エージェント・マルチエージェント・NLP タスクを含む**汎用 RL フレームワーク**です。
:::

VeRL と NeMo RL はともに Ray ベースですが、Kubernetes 対応の方式が異なります。**VeRL は SkyPilot 経由**で Kubernetes を使用します。SkyPilot は KubeRay Operator を使わず、通常の Pod を起動して `ray start --head` / `ray start --address` コマンドで Ray クラスタを手動構成します。SkyPilot は AWS/GCP/Azure/Kubernetes を単一の YAML で切り替えられるマルチクラウド抽象化レイヤー（`infra: k8s` を `infra: aws` に変えるだけで動く）、です。VeRL リポジトリに kuberay の記述はありません。一方 **NeMo RL は KubeRay Operator を直接使用**し、`RayCluster` CRD（カスタムリソース）で Ray クラスタのライフサイクルを宣言的に管理します。NeMo RL の k8s 対応は KubeRay Operator の事前インストールが前提のエンタープライズ構成です。

::::details 調査根拠: VeRL の Kubernetes 対応

**VeRL リポジトリ全体での kuberay 検索結果: 0件**

VeRL の `docs/start/multinode.rst` に記載されているマルチノード起動方法は以下の 4 種類のみ

1. 手動 Ray クラスター起動（`ray start --head`）
2. SkyPilot 経由（Kubernetes またはクラウド）
3. Slurm
4. dstack

**SkyPilot が k8s 上で Ray クラスタを構成する仕組み（`examples/skypilot/verl-ppo.yaml` より）**

```yaml
infra:
  cloud: kubernetes  # ← k8s を指定

run: |
  # SkyPilot が Pod を起動した後、ユーザーが ray start を手動実行
  ray start --head --port=6379
  # 他ノードは:
  ray start --address=$HEAD_IP:6379
```

KubeRay Operator（`RayCluster` CRD）は一切使用しない。SkyPilot は各ノードを通常の Kubernetes Pod としてプロビジョニングし、Ray クラスタの管理は `run:` セクションのシェルスクリプトで手動構成する。

**NeMo RL の KubeRay Operator 対応（`docs/cluster.md` より）**

```yaml
apiVersion: ray.io/v1
kind: RayCluster
metadata:
  name: nemo-rl-cluster
spec:
  headGroupSpec:
    ...
  workerGroupSpecs:
    ...
```

`RayCluster` CRD を用いた宣言的管理。KubeRay Operator の事前インストールが必須。

## 選択指針

### ユースケース例

あくまで例です。実際のワークロード要求に応じて選択すべきです。

| ユースケース | 推奨フレームワーク | 主な理由 |
|------------|----------------|------|
| 大規模クラスタ（100 GPU 以上）での本番 LLM ポストトレーニング | VeRL | Fully Async による高スループット、RDMA 対応 |
| 超大規模モデル（70B 以上）の LLM ポストトレーニング | NeMo RL | Megatron-Core 6D 並列化、GB200 対応 |
| MoE モデル（DeepSeek-V3、Qwen3-MoE 等）のポストトレーニング | NeMo RL（第一選択）/ VeRL（EP は WIP） | NeMo RL は Megatron EP 対応済み。VeRL の Megatron EP は開発中だが FSDP バックエンドで MoE モデル訓練自体は可能 |
| NVIDIA GB200 / Blackwell 最大性能での LLM ポストトレーニング | NeMo RL | GB200 コンテナサポート確認済み（README に "Available now" の記載あり） |
| NVIDIA 環境での SFT→RLHF 全工程 | NeMo RL | SFT/DPO/RL を統一フレームワークで |
| 研究・プロトタイピング（手軽に LLM RLHF を始めたい） | OpenRLHF | 導入コストが低く情報が豊富 |
| AWS 環境での LLM RLHF（P2P RDMA 最適化） | VeRL | CheckpointEngine + NIXL による P2P RDMA 転送（EFA はインフラ層で全フレームワーク共通） |
| NVIDIA 以外のハードウェア（AMD/Ascend）での LLM ポストトレーニング | VeRL | 最も広いハードウェア対応 |
| 新しい LLM RL アルゴリズムの研究実装 | VeRL | 豊富なアルゴリズム実装と拡張性 |
| PPO を含む完全な RLHF パイプライン | OpenRLHF | PPO（Critic 付き）の完全実装 |
| マルチエージェント RL 研究 | OpenRL | MAPPO、MAT、VDN 等の豊富な実装 |
| 汎用 RL のプロトタイピング（DQN、SAC 等） | OpenRL | 汎用 RL アルゴリズムの幅広いカバレッジ |
| NLP 対話タスクの RL（小規模） | OpenRL | HuggingFace 統合、導入コスト低 |

:::message
Generation phase のコスト最適化と VeRL のアーキテクチャ詳細については [Verl の概要調査と v0.7 リリースノート解説](/articles/251ab3faae2f47) も参考にしてください。
:::

## まとめ

- **VeRL**: Gen/Train 分離の洗練度・CheckpointEngine+NIXL による P2P RDMA 転送・アルゴリズム多様性・ハードウェア幅広さで最上位。大規模クラスタ向け
- **NeMo RL**: Megatron-Core 6D 並列化と NVIDIA ハードウェア最適化で最上位。超大規模モデル・統一フレームワーク（SFT/DPO/RL）志向向け
- **OpenRLHF**: 使いやすさ・PPO 完全実装・コミュニティサポートで優位。研究・小中規模 LLM ポストトレーニング向け
- **OpenRL**: 汎用 RL フレームワーク。LLM ポストトレーニングには直接対応しないが、マルチエージェント RL や汎用 RL 研究に優れる
