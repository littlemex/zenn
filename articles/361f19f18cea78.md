## はじめに

本記事では、画像 + テキストのマルチモーダルモデル **Qwen2.5-VL** を AWS Trainium2 上の **NxD Inference** で動かすサンプルを解説します。対象は次の 2 モデルです。

- `Qwen/Qwen2.5-VL-7B-Instruct`
- `stockmark/Stockmark-DocReasoner-Qwen2.5-VL-32B`

サンプル一式は [`littlemex/aws-neuron-samples` の `samples/models/qwen2.5-vl-nxd/`](https://github.com/littlemex/aws-neuron-samples/tree/main/samples/models/qwen2.5-vl-nxd) に配置済みで、`trn2.3xlarge` (Trainium2 を 1 device 搭載した最小インスタンス。 96 GB HBM + 4 物理コア) で動作確認しています。

本サンプルは、**NxDI を直接叩き、HuggingFace の `GenerationMixin` 互換アダプタ (`HuggingFaceGenerationAdapter`、 NxDI 同梱。 NxDI モデルを HF の `generate()` API でラップする) で `generate()` する**経路を採用します。

![](/images/361f19f18cea78-dataflow.png)

*図1: CPU → Neuron デバイス間のデータフロー。画像前処理 (AutoProcessor) と generate ループ (HuggingFaceGenerationAdapter) は CPU 側に留まり、3 NEFF は Neuron 上で実行される。*

NxDI は `compile()` 1 回でこの 3 NEFF（**NEFF = Neuron Executable File Format。 Neuron compiler が出力する実行可能バイナリで、 1 NEFF が 1 つの静的計算グラフに対応する**）をまとめて生成し、 `load()` で Neuron デバイスに載せます。 3 NEFF の中身は、

- **vision encoder NEFF**: 画像を hidden vector 列に変換 (Vision Transformer)
- **text CTE NEFF**: プロンプト全体を一括処理する **prefill 用** (CTE = Context Encoding)
- **text TKG NEFF**: 自己回帰デコードの 1 step を担当する **decode 用** (TKG = Token Generation)

### 動作確認済み構成

下の表で Tensor Parallelism(TP) と **LNC** (Logical NeuronCore Configuration = 物理 NeuronCore を論理コアとして束ねる設定。 LNC=2 は 2 物理 → 1 論理にまとめるモード) を使っています。

| モデル | HW | TP | LNC | 結果 |
|---|---|---|---|---|
| `Qwen/Qwen2.5-VL-7B-Instruct` | trn2.3xlarge | 2 | 2 | dummy gray + real 3 image いずれも coherent |
| `stockmark/Stockmark-DocReasoner-Qwen2.5-VL-32B` | trn2.3xlarge | 8 | 1 | cos=0.999938 / 日英 sanity 6/6 / avg greedy 93.75% |

7B Instruct は `compile 239 秒 + load 19 秒` で 3 NEFF が揃い、一例としてダミーのグレー 448x448 画像 + `What do you see in this image?` プロンプトに対し coherent な英語応答が得られました。

> "The image appears to be a blank, solid gray color with no discernible features, objects, or text. It looks like a uniform gray background without any variation or content."

## 1 全体アーキテクチャ

### 1.1 まず登場人物 (クラス) を 1 つずつ整理する

**VLM (Vision-Language Model)** は「画像を理解する vision encoder (画像を意味ベクトル列に変換するモデル)」+「自己回帰でテキスト生成する text decoder (Llama や Qwen と同型の Transformer)」の合成です。 NxDI でこれを動かす場合、 触るクラスは大きく 3 階層に分かれます。 図2 では上段 (Application 層)・左下 (Vision 層)・右下 (Text 層) として並べました。

#### 上位 (Application 層): 全体の取りまとめ役

| クラス | 出自 | 役割 |
|---|---|---|
| `NeuronApplicationBase` | NxDI の基底クラス | NEFF を作る (`compile()`)、 Neuron に載せる (`load()`)、 重みを HF (Hugging Face、 オープンソースのモデル配布プラットフォーム + Transformers ライブラリ) checkpoint から変換する (`convert_hf_to_neuron_state_dict()`) などの「フレームワーク共通部」 |
| `NeuronQwen2VLForCausalLM` | NxDI の Qwen2-VL 用既製クラス。 `NeuronApplicationBase` を継承 | Qwen2-VL という具体モデル向けに、 vision encoder + text の 3 NEFF をどう配線して 1 step の `forward()` にまとめるか、 を実装した「VLM オーケストレータ」 |
| `NeuronStockmarkVLForCausalLM` | 本サンプルが追加するクラス。 上記 `NeuronQwen2VLForCausalLM` を継承 | Qwen2.5-VL 用に、 「中で使う text モジュール」「中で使う vision モジュール」「重み変換手順」を Qwen2.5-VL 仕様に差し替える。 全体フローは親クラスのものを そのまま借りる |

Qwen2-VL と Qwen2.5-VL は配線・形状ともほぼ同じで、 後述する 8 箇所のローカルな差分を直せば動きます。

#### Text 層: 文字列を logits に変換する 28 (7B) / 64 (32B) 層 Transformer

| クラス | 役割 |
|---|---|
| `NeuronStockmarkTextModel` | Embedding → Transformer block × N → 出力 hidden、 という素の text decoder。 Qwen2.5-VL の **M-RoPE (Multimodal RoPE、 通常の 1 軸 position encoding を時間/高さ/幅の 3 軸に拡張したもの。 動画や 2D 画像 patch を一意に位置付けるために使う)** を扱えるよう、 NxDI 同梱の Qwen2-VL text backbone を fork して per-step rotary 再計算を入れたもの |
| `NeuronStockmarkTextForCausalLM` | 上記 `NeuronStockmarkTextModel` を持ち、 **lm_head** (Transformer 末尾の hidden を vocab 次元の logits に射影する全結合層) と HF 重みの変換を担う「text-only LM」のラッパー |

端的には、すでに Qwen2-VL 用の M-RoPE 対応 text backbone がすでに同梱されており、それをパクるだけです。それだけだとデコード途中で位置がずれてループ生成する不具合が出たので、デコード 1 step ごとに rotary を計算し直す処理を差し込んでいます。matmul に比べると圧倒的に小さい計算量なので correctness を重視してこのような実装にしました。

#### Vision 層: 画像を hidden vector 列に変換する 32 層 ViT

| クラス | 役割 |
|---|---|
| `NeuronQwen25VisionModel` | Qwen2.5-VL の **vision tower** (= 画像を意味ベクトル列に変換する VLM の画像側ブロック全体。 Qwen2.5-VL では ViT 32 層 + PatchMerger をひとまとめにしたもの。 公式実装でも HF Transformers でもこの呼び方) 本体。 Qwen2-VL からは **LayerNorm → RMSNorm** 、 **GELU 2-matrix MLP → SwiGLU 3-matrix MLP** に変わっており、 そのままでは重みのキー名が合わないので Qwen2.5-VL 用に書き換えたもの。 32 層の VisionBlock + 末尾の **PatchMerger** (vision tower 末尾の層。 patch hidden 列を圧縮して text 側の hidden_size に揃える役割) を持つ |
| `NeuronQwen25VLForImageEncoding` | 上記を `NeuronApplicationBase` で包んだ「vision-only Application」。 vision NEFF の compile / load / 重み変換のエントリポイント |

### 1.2 起動から 1 step 推論までの処理フロー

親クラス `NeuronQwen2VLForCausalLM` を継承するだけでほぼ動きますが、 何が裏で起きているのか把握しないとデバッグできません。 ここでは「Python スクリプトを起動してから、 1 トークン生成して返ってくるまで」を 3 ステップに分けて追います。

**Step A: `compile()` (1 度だけ)**

1. `text_model_cls` (= `NeuronStockmarkTextModel`) のインスタンスを作り、 prefill 用と decode 用の **2 種類の入力形状** で **trace** (PyTorch モデルを実行しながら計算グラフを記録する操作) → **text CTE NEFF** と **text TKG NEFF** を生成
2. `vision_model_cls` (= `NeuronQwen25VisionModel`) のインスタンスを作り、 dummy 画像で trace → **vision NEFF** を生成
3. 計 3 個の NEFF がディスクに出力される

:::message
実際には入力形状はさらに種類が増える可能性があり、**bucket** = 入力長別の静的形状単位、 例えば prefill bucket = `[256, 1024]` なら 2 NEFF 出る
:::

**Step B: `load()` (1 度だけ)**

1. HF checkpoint (`config.json` + 重みファイル `*.safetensors`) を読む
2. `convert_hf_to_neuron_state_dict()` で **text 用に重み変換 → vision 用に重み変換** を順に走らせる。 変換が必要な理由は、 NxDI が **TP** (Tensor Parallelism、 重みを複数 NeuronCore に分割) で重みを分割保持するための形式と、 HF 標準形式が違うため
3. 変換後の重みを 3 NEFF それぞれに流し込み、 Neuron デバイスに転送する

**Step C: 推論 1 step (`forward()`、 generate ループから毎回呼ばれる)**

![](/images/361f19f18cea78-step-c-flow.png)

`NeuronStockmarkVLForCausalLM` 自体はこの一連の流れを書き換えず、 step C の中で **「どの text モジュールを呼ぶか」「どの vision モジュールを呼ぶか」だけを差し替える**形をとります。

### 1.3 図と紐付ける

![](/images/361f19f18cea78-image-to-text.png)

*図2: 上段 (Application 層) で `NeuronStockmarkVLForCausalLM` が 3 つの属性 (`text_model_cls` / `vision_model_cls` / `convert_hf_to_neuron_state_dict()`) を選択し、 黒帯の `compile()` を経て下段の 3 NEFF (vision encoder / text CTE / text TKG) が生成される。 推論時は左下の vision encoder の出力 (vision embeds) が右下 text CTE に scatter され、 続く decode ステップは右下 text TKG が CTE 由来の KV cache を再利用する。*

| 図中の要素 | 対応する処理 |
|---|---|
| 上段 `text_model_cls = NeuronStockmarkTextModel` | Step A で text NEFF を作る時に instantiate される text モジュール選択 |
| 上段 `vision_model_cls = NeuronQwen25VisionModel` | Step A で vision NEFF を作る時に instantiate される vision モジュール選択 |
| 上段 `convert_hf_to_neuron_state_dict()` | Step B の重み変換。 vision 側は `NeuronQwen25VLForImageEncoding`、 text 側は `NeuronStockmarkTextForCausalLM` のものをチェーンして 1 つの dict を返す |
| 黒帯 `compile() 1 回で 3 NEFF を生成` | Step A 全体 |
| 左下 `Qwen2.5-VL ViT — 32 layers` | vision NEFF の中身: 32 層の VisionBlock |
| 左下 `PatchMerger` | vision NEFF 末尾。 `out_hidden_size = text hidden_size` なので、 vision 出力次元が text 側にそのまま scatter できる形に揃う |
| `vision embeds (scatter to image_token_id)` 矢印 | Step C の prefill 時、 vision 出力を |image_pad| 位置に挿入する処理 |
| 右下 `text CTE — prefill (bucket 別)` | Step C の `if step == 0` 側。 input_ids + scatter 済 vision_embeds を一括処理 |
| 右下 `text TKG — decode (1 step / 呼び出し)` | Step C の `else` 側。 next_token 1 個と KV cache を入力して 1 step 進める |
| `KV cache` 矢印 | CTE が prefill 中に書き出した KV cache を、 TKG が decode の各 step で読み続ける関係 |

クラス名の `Stockmark` プレフィックスは 32B (Stockmark-DocReasoner) を題材に作り始めた都合で命名したもので、 7B でもそのまま同一クラスを使います。 ファイル名 (`modeling_qwen25vl*.py`) とプレフィックスが揃っていないのはそのためで、 機能上の意味はありません。

## 2 実装時にハマった経緯

最初からこの 3 NEFF + 親クラス継承構成だったわけではありません。 32B → 7B の順に Qwen2.5-VL 系列を詰めた過程で、 5 回ほど大きく方針転換しました。

**経緯 1: 「自前 VLM クラスを作る」**

最初、 vision encoder と text decoder を `NeuronApplicationBase` 直接継承で別々の Application として書き、 上位の VLM 制御は自前ループで書こうとしましたが prefill bucket 切り替えや CTE/TKG 切替の制御が NxDI 内部実装と二重管理になりました。 NxDI 同梱の `NeuronQwen2VLForCausalLM` が **その制御を全部やってくれている**ことに気づき、 これを継承して `text_model_cls` / `vision_model_cls` だけ差し替える今の形に倒しました。

**経緯 2: 「Qwen2-VL クラスを vision でもそのまま使う」→**

NxDI 同梱の `NeuronQwen2VLVisionModel` をそのまま `vision_model_cls` に指定して checkpoint をロードしたら、 大量の missing/unexpected key warning と共に **vision embedding が完全に壊れました**。 原因は Qwen2-VL → Qwen2.5-VL で vision tower(画像エンコーダ部分) が置き換わっていたことです。ちゃんと実装を確認しましょう。。。

| 場所 | Qwen2-VL | Qwen2.5-VL |
|---|---|---|
| `norm1` / `norm2` / `merger.ln_q` | LayerNorm (weight + bias) | RMSNorm (weight only) |
| VisionBlock MLP | `fc1 + GELU + fc2` | `gate_proj + up_proj + down_proj` (SwiGLU) |

**経緯 3: 「`padding_side="left"` で組んだら日本語生成が壊れた」**

まず前提から。 HuggingFace で言語モデルに **可変長のプロンプトをバッチで流す**とき、 短いプロンプトは `[PAD]` トークンで埋めて長さを揃えます。 この埋め方には 2 通りあって、

```
prompt = "猫の鳴き声は"  (6 token、 seq_len=10 にパディングする例)

right-pad: [猫, の, 鳴, き, 声, は, PAD, PAD, PAD, PAD]   ← 末尾を PAD で埋める
left-pad : [PAD, PAD, PAD, PAD, 猫, の, 鳴, き, 声, は]   ← 先頭を PAD で埋める
```

decoder 系モデルの推論では、 `generate()` の最終トークンの logits を見て「次のトークン」を決めるため、 **left-pad のほうが「実トークンの末尾 == テンソルの末尾」になって扱いやすい** です。そこでNxDI に渡す前に自前で left-pad してから `padding_side="left"` を NeuronConfig に渡す実装にしました。 加えて、 left-pad 部分を attention で無視させるため `forward()` を override して masked_fill 補正も入れました。1-layer smoke (簡略モデルでの動作確認) では cos 類似度 = 0.9998 と HF CPU と一致。 64-layer フルモデルでも英語 prompt は greedy match 14/16 でそれっぽく見える出力でした。

ところが日本語に切り替えた瞬間、 **3 step で生成が破綻**しました。

- 日本語: `"質問われてきょ The question..."`、 `"猫のん〜inkink Asia Japan's..."` のように意味不明な文字列 + 英語混入
- 英語も実は深掘ると怪しく、 greedy match は実 prompt で 2/16 (例: `"machine learning in-house-trained neural network"` のような不自然な接続)

NxDI のソースを読んで判明したのは、 **NxDI 内部の `ModelWrapper.pad_inputs()` (バッチ整形ヘルパ) が hardcoded で right-pad を行い、 NeuronConfig の `padding_side` パラメータを一切参照していませんでした**。

つまり実際に NEFF に流れていたテンソルは、私が追加した left-pad と自動で NxDI が追加した **両端から PAD で挟まれた状態**が NEFF に届きます。 さらに NxDI が内部で計算する `position_ids` と `attention_mask` は **すべて right-pad だけ前提**で組まれていました。

結果として attention の有効領域 ↔ 実 token 位置の対応が完全にズレ、 masked_fill 補正は **正しい配置を歪める方向** に働いていました。

- `padding_side="right"` に書き換え (NxDI のデフォルト挙動と一致させる)

これだけで英語 greedy match が 14-16/16、 日本語が HF CPU と同等まで一気に回復しました。

**経緯 4: 「CTE で計算した `cos_cache` を TKG に引き継ぐ (NxDI 既定)」→**

`padding_side` を直しても **decode 数 step 後に `Paris.Paris.Paris.` 的な無限ループや、 日本語ひらがな繰り返し**が出続ける症状がありました。 NxDI の `NeuronAttentionBase` は CTE で計算した **`cos_cache` / `sin_cache`** (RoPE が使う回転行列を、 cos/sin の lookup table として事前計算したもの) を内部で保持し、 TKG 各 step で再利用します。 Llama 系の通常 RoPE はこれで問題ありません。しかし Qwen2.5-VL の M-RoPE は **時間/高さ/幅の 3 軸 position に依存**するので、 prefill 末尾の position で計算した cos/sin を TKG が使い回すと、 全ての decode step が「同じ位置」で rotary をかけることになります。 結果、 直前 token に強く引き寄せられ、 同じ token を吐き続けます。

`apply_rotary_embedding` を override して **TKG step ごとに `rotary_emb(V, position_ids)` を呼び直す**ようにしたところループは消えました。

**経緯 5: 「`<|im_end|>` だけを `eos_token_id` に」→**

ループ問題が直った後、 今度は **`max_new_tokens` まで走り切ってから止まる** 症状に当たりました。 ここで言う **`eos_token_id`** は HuggingFace の `generate()` が「この token id が出たら停止しろ」と判断する EOS (End-Of-Sequence) token のリストです。 `<|im_end|>=151645` (Qwen 系の chat 終端マーカー) だけを `eos_token_id` にしていたのが原因で、 Qwen2.5-VL の chat template は終端に `<|endoftext|>=151643` (より一般的な文書終端マーカー) も挟むパターンがあり、 こちらを EOS 扱いしないと止まりません。 `eos_token_id=[151645, 151643]` の 2 値指定で解決しました。

**経緯 6: 「32B でしか動かさない前提」→**

最初に 32B (`Stockmark-DocReasoner-Qwen2.5-VL-32B`) で動かして固めたコードをそのまま 7B (`Qwen/Qwen2.5-VL-7B-Instruct`) に向けたら、 `StockmarkVLInferenceConfig.from_pretrained` の中で `cfg_dict["text_config"]["hidden_size"]` を読もうとして `KeyError` で死にました。 32B Stockmark-DocReasoner は HF config が `text_config` 配下にネストされているのに対し、 公式の 7B Instruct は **flat layout** (`text_config` が無く top-level に直接 `hidden_size` などを置く) を使っていたためです。

`text_config` が空なら top-level から拾うフォールバックを 1 関数に押し込んで両対応にしました 。 加えて `pad_token_id` が 7B では `None` で、 NxDI の `validate_config()` が落ちる事象も修正しました。

## 3 実機での動作確認手順

:::

### 3.3 venv の有効化と依存確認

NxDI 用の venv を有効化します。**vLLM 用 venv ではなく `_nxd_inference` の方を使う**点に注意してください（torch のバージョンが違うため両者は併用できません）。

### 3.4 Hugging Face checkpoint のダウンロード

`Qwen/Qwen2.5-VL-7B-Instruct` は約 16 GB です。本ステップ以降を含めて **合計 40 GB 程度のディスク空き容量**が必要です（HF ckpt 16 GB + `compile_qwen25vl.py` が保存する safetensors 16 GB + NEFF 約 5 GB + compile cache）。事前に `df -h` で確認してください。

公開モデル (`Qwen/Qwen2.5-VL-7B-Instruct`) には HF_TOKEN は不要ですが、 後続の `compile_qwen25vl.py` がローカル checkpoint 不在時に runtime download を試みます。 念のため事前に export しておくと安全です。

```bash
export HF_TOKEN=<your_hf_token>   # 7B Instruct は public なので空でも可。32B など gated model では必須。
export HF_HUB_DISABLE_XET=1       # 一部リージョンで HF Hub の xet が遅いため一律無効化推奨
```

ダウンロード本体。 `$HOME` を Python 側で正しく展開するため `os.path.expanduser` を使います（**python -c の単引用符内で `$HOME` を書くと shell 展開されないので注意**）。

```bash
mkdir -p $HOME/qwen25vl-nxd/hf-ckpt-28l

python - <<"PYEOF"
import os
from huggingface_hub import snapshot_download
snapshot_download(
    repo_id='Qwen/Qwen2.5-VL-7B-Instruct',
    local_dir=os.path.expanduser('~/qwen25vl-nxd/hf-ckpt-28l'),
    ignore_patterns=['*.gguf','*.ggml','*.bin'],
    max_workers=8,
)
PYEOF

du -sh $HOME/qwen25vl-nxd/hf-ckpt-28l
# 16G  /home/ubuntu/qwen25vl-nxd/hf-ckpt-28l
```

### 3.8 GQA 警告について

ログ中に次のような警告が大量に出ますが、本サンプルの設定では実害はありません。

```text
WARNING:Neuron:TP degree (2) and KV heads (4) are not divisible.
Overriding attention sharding strategy to GQA.CONVERT_TO_MHA!
```

この WARNING は TP=2, kv_heads=4 という設定でも出力されました（NxDI の内部 head-sharding 判定は単純な mod 演算とは異なる可能性があります）。verdict が A かつ coherent な出力が得られているため correctness への影響は確認されていません。TP=8 では GQA→MHA 変換が発生し correctness drift が生じます（README 参照）。

TP=4（LNC=1）での throughput 比較は未取得のため今後の課題とします。

## §4 32B (Stockmark-DocReasoner-Qwen2.5-VL-32B) を動かす場合

7B と同じコードベースで 32B も動かせます。設定だけ変えれば OK ですが、 ディスクと RAM の要件が増えるので事前確認が必要です。

:::message alert
32B モデルのダウンロード + 保存には約 60 GB のディスク空き容量、`from_pretrained` 時の bf16 ロードに 64 GB 以上のシステム RAM が必要です。事前に `df -h` / `free -h` で確認してください。
:::

```bash
# 1. 事前ダウンロード
mkdir -p $HOME/qwen25vl-nxd-32b/hf-ckpt-64l
export HF_TOKEN=<your_hf_token>   # 32B は gated なので必須

python - <<"PYEOF"
import os
from huggingface_hub import snapshot_download
snapshot_download(
    repo_id='stockmark/Stockmark-DocReasoner-Qwen2.5-VL-32B',
    local_dir=os.path.expanduser('~/qwen25vl-nxd-32b/hf-ckpt-64l'),
    ignore_patterns=['*.gguf','*.ggml','*.bin'],
    max_workers=8,
)
PYEOF

# 2. 環境変数 (LNC=1)
mkdir -p $HOME/qwen25vl-nxd-32b/.compile-cache
export NEURON_COMPILE_CACHE_URL=$HOME/qwen25vl-nxd-32b/.compile-cache
export NEURON_LOGICAL_NC_CONFIG=1
export NEURON_RT_VISIBLE_CORES=0-7
export NEURON_RT_NUM_CORES=8
export NEURON_CC_FLAGS="--target=trn2 --auto-cast=none --lnc=1"

# 3. compile + smoke
WORK_DIR=$HOME/qwen25vl-nxd-32b \
HF_CKPT_DIR=$HOME/qwen25vl-nxd-32b/hf-ckpt-64l \
TP_DEGREE=8 NUM_LAYERS=64 \
MAX_CONTEXT_LEN=512 MAX_NEW_TOKENS=64 \
MODEL_ID=stockmark/Stockmark-DocReasoner-Qwen2.5-VL-32B \
python compile_qwen25vl.py
```

`num_kv_heads=8` なので TP=8 で割り切れます。 32B での参考値として、 同等ロジックの実装で full 64-layer の compile に約 4.6 分、 HF CPU との probe cos=`0.999938`、 日英 6 prompt sanity が `6/6 non-degenerate / avg greedy 93.75%` という結果を得ています。 **なお `compile_qwen25vl.py` での 32B 直接実行は本記事執筆時点で未検証なので、 試す場合は HBM とディスクの容量を確認したうえで実行してください。**

### 7B と 32B の差分は本当に小さい

32B → 7B にコードを移植する際の差分は実質 4 箇所だけです。

| 差分 | 32B (Stockmark-DocReasoner) | 7B (Instruct) |
|---|---|---|
| HF config layout | `text_config` 配下にネスト | top-level に flat |
| `num_kv_heads` | 8 | 4 |
| `num_hidden_layers` | 64 | 28 |
| Module rename | `modeling_stockmark_*.py` | `modeling_qwen25vl_*.py` |

vision tower（RMSNorm + SwiGLU）と M-RoPE 関連の実装はどちらも完全に共通です。

## §5 6 prompt sanity (text NEFF だけを叩く)

`sanity_qwen25vl.py` は text 専用 NEFF を再ロードして 6 prompt（en/ja 各 3）を投げ、HF CPU と比較するスクリプトです。VLM 全体ではなく text 経路だけのリグレッション確認に使います。

:::message alert
`sanity_qwen25vl.py` は **VLM compile (`compile_qwen25vl.py`) が出力する `traces/vl-{N}l/` をそのまま流用できません**（VLM の `text_model` は VLM Application に組み込まれていて単体ロードできないため）。 sanity 用には **別途 text-only な NEFF を `traces/text-{N}l/` などに用意**する必要があります。 現状のサンプルは text-only NEFF を別経路で用意してある状態を想定しており、 7B 単体で text-only NEFF を生成する compile スクリプトは未同梱です。
:::

VLM 経路の動作確認だけが目的であれば §3 で十分です。 §5 は text-only NEFF を別途持っている場合の参考としてご覧ください。

```bash
WORK_DIR=$HOME/qwen25vl-nxd \
NEFF_DIR=$HOME/qwen25vl-nxd/traces/text-28l \
NUM_LAYERS=28 TP_DEGREE=2 \
MAX_CONTEXT_LEN=128 MAX_NEW_TOKENS=16 \
MODEL_ID=Qwen/Qwen2.5-VL-7B-Instruct \
HF_TOKEN=${HF_TOKEN:-} \
python sanity_qwen25vl.py
```

`MAX_CONTEXT_LEN` / `MAX_NEW_TOKENS` は text-only compile 時と同じ値にしてください。NEFF を流用する以上、これを変えると再 compile が必要です。 また `sanity_qwen25vl.py` は HF Hub からトークナイザーを取得するためネットワークアクセスが必要です。 gated モデル (32B) では `HF_TOKEN` が必須です。

参考: 32B (Stockmark-DocReasoner) での実行結果（7B での同等評価は未実施。 7B では一部 token が異なる場合があります）:

## §6 トラブルシューティング

実機で踏みやすい順に並べます。

### 6.1 `libneuronpjrt-path` not found

```text
FileNotFoundError: [Errno 2] No such file or directory: 'libneuronpjrt-path'
```

vLLM 用ではなく NxDI 用の venv を activate してください: `source /opt/aws_neuronx_venv_pytorch_2_9_nxd_inference/bin/activate`。 すでに別の venv を activate している場合は `deactivate` してから再実行してください（§3.3 参照）。

### 6.2 `PermissionError: /var/tmp/neuron-compile-cache/...`

DLAMI で `/var/tmp/neuron-compile-cache` の所有者が別ユーザーになっているケースです。`NEURON_COMPILE_CACHE_URL` をユーザー所有のパスに切り替えます（§3.5 参照）。

### 6.3 vLLM 経由で起動して `Model Qwen2_5_VLForConditionalGeneration is not supported`

```text
ValueError: Model Qwen2_5_VLForConditionalGeneration is not supported on Neuron for now.
Supported models: ['gpt_oss', 'llama', ..., 'qwen2_vl', ..., 'qwen3_vl']
```

vLLM Neuron plugin v0.16 は `qwen2_5_vl` 未対応です。本サンプルの NxDI 直叩き経路を使ってください。

### 6.4 日本語が 3 step で破綻する

`padding_side="left"` 系の設定が残っています。Point 1 を再確認してください。

### 6.5 Decode が `Paris.` / ひらがな繰り返しでループする

cos_cache stale の典型症状です。`apply_rotary_embedding` の override が外れているか、`rotary_emb` を毎ステップ呼んでいるかを確認してください（Point 4）。

### 6.6 `python -c` 内で `$HOME` などのパス変数が展開されない

`python -c '...'` の単引用符は shell 展開を抑止するため、 `'$HOME/...'` は文字列 `$HOME/...` のままで Python に渡されます。 §3.4 の例のように `python - <<"PYEOF"` ヒアドキュメント + `os.path.expanduser('~/...')` を使うか、 単引用符を二重引用符に変えてください。

### 6.7 32B が CPU メモリで OOM する

`Qwen2_5_VLForConditionalGeneration.from_pretrained` で bf16 model 全体を一旦 CPU にロードします。 32B の場合 64 GB RAM が必要で、 trn2.3xlarge は 64 GB RAM のため余裕がありません。 `low_cpu_mem_usage=True`（サンプル既定）でも OOM したら、 swap を有効化するか、 RAM の大きいインスタンスで checkpoint 変換だけ先に行ってから `HF_CKPT_DIR` 経由で持ち込むのが確実です。
