## はじめに

本記事では、画像 + テキストのマルチモーダルモデル **Qwen2.5-VL** を AWS Trainium2 上の **NxD Inference (NxDI)** で動かすサンプルを解説します。対象は次の 2 モデルです。

- `Qwen/Qwen2.5-VL-7B-Instruct`
- `stockmark/Stockmark-DocReasoner-Qwen2.5-VL-32B`

サンプル一式は [`littlemex/aws-neuron-samples` の `samples/models/qwen2.5-vl-nxd/`](https://github.com/littlemex/aws-neuron-samples/tree/main/samples/models/qwen2.5-vl-nxd) に配置済みで、`trn2.3xlarge` で動作確認しています。

:::message alert
NxDI 0.10 / NxD 0.19 / torch 2.9.1 / transformers 4.57.6 時点のバージョンに基づいています。具体的なパッケージバージョンは §3.3 を参照してください。AWS Neuron SDK のアップデートや HuggingFace Transformers の更新で挙動が変わる可能性があるため、最新の実装を必ず確認してください。
:::

本サンプルは、**NxDI を直接叩き、HuggingFace の `GenerationMixin` 互換アダプタ (`HuggingFaceGenerationAdapter`) で `generate()` する**経路を採用します。

![](/images/361f19f18cea78-dataflow.png)

*図1: CPU → Neuron デバイス間のデータフロー。画像前処理 (AutoProcessor) と generate ループ (HuggingFaceGenerationAdapter) は CPU 側に留まり、3 NEFF は Neuron 上で実行される。*

NxDI は `compile()` 1 回でこの 3 NEFF（vision encoder / text CTE / text TKG）をまとめて生成し、`load()` で全てを Neuron デバイスに載せます。

### 動作確認済み構成

| モデル | HW | TP | LNC | 結果 |
|---|---|---|---|---|
| `Qwen/Qwen2.5-VL-7B-Instruct` | trn2.3xlarge | 2 | 2 | dummy gray + real 3 image いずれも coherent |
| `stockmark/Stockmark-DocReasoner-Qwen2.5-VL-32B` | trn2.3xlarge | 8 | 1 | cos=0.999938 / 日英 sanity 6/6 / avg greedy 93.75% |

7B Instruct は `compile 239 秒 + load 19 秒` で 3 NEFF が揃い、一例としてダミーのグレー 448x448 画像 + `What do you see in this image?` プロンプトに対し coherent な英語応答 (degenerate=False, verdict=A) が得られました。

> "The image appears to be a blank, solid gray color with no discernible features, objects, or text. It looks like a uniform gray background without any variation or content."

## §1 全体アーキテクチャ

### 1.1 まず登場人物 (クラス) を 1 つずつ整理する

VLM (Vision-Language Model) は「画像を理解する vision encoder」+「自己回帰でテキスト生成する text decoder」の合成です。 NxDI でこれを動かす場合、 我々が触るクラスは大きく 3 階層に分かれます。 図2 では上段 (Application 層)・左下 (Vision 層)・右下 (Text 層) として並べました。

#### 上位 (Application 層): 全体の取りまとめ役

| クラス | 出自 | 役割 |
|---|---|---|
| `NeuronApplicationBase` | NxDI の基底クラス | NEFF を作る (`compile()`)、 Neuron に載せる (`load()`)、 重みを HF checkpoint から変換する (`convert_hf_to_neuron_state_dict()`) などの「フレームワーク共通部」 |
| `NeuronQwen2VLForCausalLM` | NxDI の Qwen2-VL 用既製クラス。 `NeuronApplicationBase` を継承 | Qwen2-VL という具体モデル向けに、 vision encoder + text の 3 NEFF をどう配線して 1 step の `forward()` にまとめるか、 を実装した「VLM オーケストレータ」 |
| `NeuronStockmarkVLForCausalLM` | 本サンプルが追加するクラス。 上記 `NeuronQwen2VLForCausalLM` を継承 | Qwen2.5-VL 用に、 「中で使う text モジュール」「中で使う vision モジュール」「重み変換手順」を Qwen2.5-VL 仕様に差し替える。 全体フローは親クラスのものを そのまま借りる |

ここで言う「Qwen2-VL の既存コンパイルパスを再利用」とは、 親クラス (`NeuronQwen2VLForCausalLM`) が持っている **「3 NEFF を作って配線して `forward()` する一連の手順」を書き直さずに継承する**、 という意味です。 Qwen2-VL と Qwen2.5-VL は配線・形状ともほぼ同じで、 後述する 8 箇所のローカルな差分を直せば動きます。

#### Text 層: 文字列を logits に変換する 28 (7B) / 64 (32B) 層 Transformer

| クラス | 役割 |
|---|---|
| `NeuronStockmarkTextModel` | Embedding → Transformer block × N → 出力 hidden、 という素の text decoder。 Qwen2.5-VL の M-RoPE (時間/高さ/幅 の 3 軸 RoPE) を扱えるよう、 NxDI 同梱の Qwen2-VL text backbone を fork して per-step rotary 再計算を入れたもの |
| `NeuronStockmarkTextForCausalLM` | 上記 `NeuronStockmarkTextModel` を持ち、 lm_head (vocab logit 化) と HF 重みの変換を担う「text-only LM」のラッパー |

#### Vision 層: 画像を hidden vector 列に変換する 32 層 ViT

| クラス | 役割 |
|---|---|
| `NeuronQwen25VisionModel` | Qwen2.5-VL の vision tower 本体。 Qwen2-VL からは LayerNorm → RMSNorm、 GELU 2-matrix MLP → SwiGLU 3-matrix MLP に変わっており、 そのままでは重みのキー名が合わないので Qwen2.5-VL 用に書き換えたもの。 32 層の VisionBlock + 末尾の PatchMerger を持つ |
| `NeuronQwen25VLForImageEncoding` | 上記を `NeuronApplicationBase` で包んだ「vision-only Application」。 vision NEFF の compile / load / 重み変換のエントリポイント |

### 1.2 「Qwen2-VL の仕組みに乗せる」の中身

`NeuronQwen2VLForCausalLM` (親クラス) が裏で何をしているかを、 起動から 1 step 推論までステップで追います。

**Step A: `compile()` (1 度だけ)**

1. `text_model_cls` (= `NeuronStockmarkTextModel`) のインスタンスを作り、 prefill 用と decode 用の **2 種類の入力形状** で trace → **text CTE NEFF** と **text TKG NEFF** を生成
2. `vision_model_cls` (= `NeuronQwen25VisionModel`) のインスタンスを作り、 dummy 画像で trace → **vision NEFF** を生成
3. 計 3 個の NEFF (Neuron Executable File Format) がディスクに出力される

**Step B: `load()` (1 度だけ)**

1. HF checkpoint (`config.json` + `*.safetensors`) を読む
2. `convert_hf_to_neuron_state_dict()` で **text 用に重み変換 → vision 用に重み変換** を順に走らせる (本サンプルが分割している部分)
3. 変換後の重みを 3 NEFF それぞれに流し込み、 Neuron デバイスに転送する

**Step C: 推論 1 step (`forward()`、 generate ループから毎回呼ばれる)**

```text
  if step == 0 and 画像あり:
      vision NEFF を呼ぶ  → 画像 patch 列 → vision_embeds [n_image_tokens, hidden_size]
      input_ids 中の <|image_pad|> 位置に vision_embeds を scatter
      text CTE NEFF を呼ぶ → 全 prefill token をまとめて処理 → KV cache + 末尾 hidden
  else:
      text TKG NEFF を呼ぶ → 直前の next_token を 1 つだけ流す → KV cache を更新 → hidden
  hidden に lm_head を当てて logits を返す
  CPU 側 (HuggingFaceGenerationAdapter) が argmax で next_token を決める
```

`NeuronStockmarkVLForCausalLM` 自体はこの一連の流れを書き換えず、 step C の中で **「どの text モジュールを呼ぶか」「どの vision モジュールを呼ぶか」だけを差し替える**形をとります。

### 1.3 図と紐付ける

![](/images/361f19f18cea78-image-to-text.png)

*図2: 上段 (Application 層) で `NeuronStockmarkVLForCausalLM` が 3 つの属性 (`text_model_cls` / `vision_model_cls` / `convert_hf_to_neuron_state_dict()`) を選択し、 黒帯の `compile()` を経て下段の 3 NEFF (vision encoder / text CTE / text TKG) が生成される。 推論時は左下の vision encoder の出力 (vision embeds) が右下 text CTE に scatter され、 続く decode ステップは右下 text TKG が CTE 由来の KV cache を再利用する。*

| 図中の要素 | 対応する処理 |
|---|---|
| 上段 `text_model_cls = NeuronStockmarkTextModel` | Step A で text NEFF を作る時に instantiate される text モジュール選択 |
| 上段 `vision_model_cls = NeuronQwen25VisionModel` | Step A で vision NEFF を作る時に instantiate される vision モジュール選択 |
| 上段 `convert_hf_to_neuron_state_dict()` | Step B の重み変換。 vision 側は `NeuronQwen25VLForImageEncoding`、 text 側は `NeuronStockmarkTextForCausalLM` のものをチェーンして 1 つの dict を返す |
| 黒帯 `compile() 1 回で 3 NEFF を生成` | Step A 全体 |
| 左下 `Qwen2.5-VL ViT — 32 layers` | vision NEFF の中身: 32 層の VisionBlock |
| 左下 `PatchMerger` | vision NEFF 末尾。 `out_hidden_size = text hidden_size` なので、 vision 出力次元が text 側にそのまま scatter できる形に揃う |
| `vision embeds (scatter to image_token_id)` 矢印 | Step C の prefill 時、 vision 出力を `<|image_pad|>` 位置に挿入する処理 |
| 右下 `text CTE — prefill (bucket 別)` | Step C の `if step == 0` 側。 input_ids + scatter 済 vision_embeds を一括処理 |
| 右下 `text TKG — decode (1 step / 呼び出し)` | Step C の `else` 側。 next_token 1 個と KV cache を入力して 1 step 進める |
| `KV cache` 矢印 | CTE が prefill 中に書き出した KV cache を、 TKG が decode の各 step で読み続ける関係 |

### 1.4 クラス名と継承関係を 1 行で

```text
NeuronApplicationBase           (NxDI 共通基底)

NeuronQwen2VLForCausalLM        (NxDI 同梱の Qwen2-VL 用 VLM オーケストレータ)
```

### 1.6 実装時にハマった経緯 (この設計に行き着くまで)

最初からこの 3 NEFF + 親クラス継承構成だったわけではありません。 32B (EXP-1037) → 7B の順に Stockmark-DocReasoner で詰めた過程で、 5 回ほど大きく方針転換しました。

#### 経緯 1: 「自前 VLM クラスを作る」→「NxDI の `NeuronQwen2VLForCausalLM` を継承する」に倒した

最初、 vision encoder と text decoder を `NeuronApplicationBase` 直接継承で別々の Application として書き、 上位の VLM 制御は自前ループで書こうとしました。 しかし `HuggingFaceGenerationAdapter.generate()` が要求する `forward()` シグネチャ (画像 + テキスト + KV cache + 位置情報をまとめて受ける) を素直に実装すると、 prefill bucket 切り替えや CTE/TKG 切替の制御が NxDI 内部実装と二重管理になりました。 NxDI 同梱の `NeuronQwen2VLForCausalLM` が **その制御を全部やってくれている**ことに気づき、 これを継承して `text_model_cls` / `vision_model_cls` だけ差し替える今の形に倒しました。 自前 VLM ループは捨て、 同等処理は親クラスの `forward()` に任せる方針です。

#### 経緯 2: 「Qwen2-VL クラスを vision でもそのまま使う」→「Qwen2.5-VL 用に書き直す」

NxDI 同梱の `NeuronQwen2VLVisionModel` をそのまま `vision_model_cls` に指定して checkpoint をロードしたら、 大量の missing/unexpected key warning と共に **vision embedding が完全に garbage** になりました。 原因は Qwen2-VL → Qwen2.5-VL で vision tower が置き換わっていたことです。

| 場所 | Qwen2-VL | Qwen2.5-VL |
|---|---|---|
| `norm1` / `norm2` / `merger.ln_q` | LayerNorm (weight + bias) | RMSNorm (weight only) |
| VisionBlock MLP | `fc1 + GELU + fc2` | `gate_proj + up_proj + down_proj` (SwiGLU) |

これに気づくまで「画像入力時のみ生成が壊れる」現象を text 側のバグだと誤診して、 数日 RoPE と KV cache を疑い続けました。 結果、 `Qwen25RMSNorm` / `Qwen25VLVisionMlp` / `Qwen25VLVisionBlock` を実装して `vision_model_cls` を差し替える Point 5 の方針に至りました。

#### 経緯 3: 「`padding_side="left"` で組む」→「`padding_side="right"` が NxDI 唯一の正解」 (最も時間を溶かした)

最初は HF Transformers のデフォルト感覚で `padding_side="left"` + 事前 left-pad + masked_fill で CTE 判定を補正する実装にしました。 1-layer smoke では cos=0.9998 で OK、 64-layer でも英語 prompt は greedy match 14/16 と一見成功。 ところが日本語 prompt は **3 step で `"3776 → 37777777"` のような無限ループ**で破綻します。

- 英語: greedy match 2/16 (`"machine learning in-house-trained neural network"` のような意味逸脱)
- 日本語: `"質問われてきょ The question..."`、 `"猫のん〜inkink Asia Japan's..."` 等

5 名体制の詳細調査を 2 周回した結果、 NxDI の `ModelWrapper.pad_inputs()` が **hardcoded で right-pad しており、 NeuronConfig の `padding_side` をそもそも参照していない**ことが判明しました。 つまり「`left` を指定したつもりで実は混合状態のテンソル」が CTE に流れていたわけです。

`padding_side="right"` に書き換え、 事前 left-pad 処理を削除し、 `forward()` / `prepare_inputs_for_generation` の override も全削除した瞬間に英語 14-16/16、 日本語 HF と同等まで一気に回復しました。 「あの override は補正ではなく改悪だった」というオチです。

| 条件 | 英語 greedy match | 日本語 |
|---|---|---|
| left-pad + padding_side="left" (当初) | 2/16 | 3 step で破綻 |
| **right-pad + padding_side="right" (最終)** | **14-16/16** | HF と同等 |

詳細は §2 Point 1 で。

#### 経緯 4: 「CTE で計算した `cos_cache` を TKG に引き継ぐ (NxDI 既定)」→「TKG で毎 step 再計算」

`padding_side` を直しても **decode 数 step 後に `Paris.Paris.Paris.` 的な無限ループや、 日本語ひらがな繰り返し**が出続ける症状がありました。 NxDI の `NeuronAttentionBase` は CTE で計算した `cos_cache` / `sin_cache` を内部で保持し、 TKG 各 step で再利用します。 Llama 系の通常 RoPE はこれで問題ありません。

しかし Qwen2.5-VL の M-RoPE は **時間/高さ/幅の 3 軸 position に依存**するので、 prefill 末尾の position で計算した cos/sin を TKG が使い回すと、 全ての decode step が「同じ位置」で rotary をかけることになります。 結果、 直前 token に強く引き寄せられ、 同じ token を吐き続けます。

`apply_rotary_embedding` を override して **TKG step ごとに `rotary_emb(V, position_ids)` を呼び直す**ようにしたところループは消えました。 XLA fusion が NEFF 内静的形状なら吸収するため再計算コストは無視できます。 詳細は §2 Point 4 で。

#### 経緯 5: 「`<|im_end|>` だけを `eos_token_id` に」→「`[151645, 151643]` の 2 値必須」

ループ問題が直った後、 今度は **`max_new_tokens` まで走り切ってから止まる** 症状に当たりました。 `<|im_end|>=151645` だけを `eos_token_id` にしていたのが原因で、 Qwen2.5-VL の chat template は終端に `<|endoftext|>=151643` も挟むパターンがあり、 こちらを EOS 扱いしないと止まりません。 `eos_token_id=[151645, 151643]` の 2 値指定で解決しました (Point 6)。

#### 経緯 6: 「32B でしか動かさない前提」→「7B/32B 両対応の `from_pretrained` に」

EXP-1037 (32B) で動かしたコードをそのまま 7B (`Qwen/Qwen2.5-VL-7B-Instruct`) に向けたら、 `StockmarkVLInferenceConfig.from_pretrained` の中で `cfg_dict["text_config"]["hidden_size"]` を読もうとして `KeyError` で死にました。 32B Stockmark-DocReasoner は HF config が `text_config` 配下にネストされているのに対し、 公式の 7B Instruct は **flat layout** (`text_config` が無く top-level に直接 `hidden_size` などを置く) を使っていたためです。

`text_config` が空なら top-level から拾うフォールバックを 1 関数に押し込んで両対応にしました (Point 2)。 加えて `pad_token_id` が 7B では `None` で、 NxDI の `validate_config()` が落ちる別件もここで直しています (`eos_token_id` フォールバック)。

#### まとめ: 何度も方針転換した結果のいまの形

これら 6 件は ① 上位構造 (経緯 1) ② vision (経緯 2) ③ text 数値 (経緯 3, 4) ④ 生成制御 (経緯 5) ⑤ config 解析 (経緯 6) を網羅しており、 §2 の 8 Point は **全部この経緯 1-6 の最終形**を整理し直したものです。 8 Point を最初から知っていれば 5 名 × 2 周のデバッグは要らなかった、 という意味で、 §2 を先に通読しておくのが結局いちばん速い入り方です。

## §2 実装の重要ポイント (8 箇所、3 カテゴリ)

NxDI 上で Qwen2.5-VL を動かすには、大きく「実行環境設定」「config 解析」「アーキテクチャ差分」の 3 カテゴリ、合計 8 箇所に対処が必要です。順に解説します。

### 2A. 実行環境・設定

#### Point 1: `padding_side="right"` が NxDI の唯一の正解

NxDI の `ModelWrapper.pad_inputs()` は **hardcoded で right-pad を行い、`padding_side` を参照しません**。これに気づかず `padding_side="left"` + 事前 left-pad + masked_fill 補正で組むと、日本語 prompt の生成が 3 step で破綻します。

| 設定 | 英語 greedy match (16 tokens) | 日本語 |
|---|---|---|
| left-pad + padding_side=left（誤） | 2/16 | 3 step で破綻 |
| **right-pad + padding_side=right（正）** | **14-16/16** | HF と同等 |

サンプルでは `compile_qwen25vl.py` で次のように明示しています。

```python
text_nc = m_text.StockmarkTextNeuronConfig(
    tp_degree=TP_DEGREE,
    logical_nc_config=1,
    torch_dtype=torch.bfloat16,
    batch_size=BATCH,
    seq_len=SEQ_LEN,
    max_context_length=MAX_CONTEXT_LEN,
    n_positions=SEQ_LEN,
    max_new_tokens=MAX_NEW_TOKENS,
    max_length=SEQ_LEN,
    on_device_sampling_config=None,
    vocab_parallel=False,
    fused_qkv=False,
    padding_side="right",   # <-- これが必須
)
```

加えて `forward()` / `prepare_inputs_for_generation()` の override は **書かない**のが正解です。left-pad 用の masked_fill 補正は right-pad では不要かつ有害です。

#### Point 8: LNC2 は runtime と compiler の両方で指定する

trn2.3xlarge を LNC=2 で使う場合、**runtime 環境変数と compiler フラグの両方で同じ値を設定**します。片方だけだと load 時に LNC 不一致でエラーになります。

```bash
export NEURON_LOGICAL_NC_CONFIG=2          # runtime
export NEURON_RT_VISIBLE_CORES=0-1         # runtime
export NEURON_RT_NUM_CORES=2               # runtime
export NEURON_CC_FLAGS="--target=trn2 --auto-cast=none --lnc=2"   # compiler
```

### 2B. Config 解析・key remap

#### Point 2: `from_pretrained` 自前実装と 7B/32B の config layout 吸収

NxDI の `ImageToTextInferenceConfig` には `from_pretrained` がありません。HF の `config.json` を自分で読んで `text_config` / `vision_config` を組み立てる必要があります。

さらに Qwen2.5-VL は同じシリーズでも config layout が 2 通りあります。

- **32B (Stockmark-DocReasoner)**: `text_config` 配下に `hidden_size` などをネスト
- **7B (公式 Instruct)**: `text_config` が無く top-level に flat 配置

サンプルの `StockmarkVLInferenceConfig.from_pretrained` は両方を 1 関数で吸収します。

```python
text_dict = dict(cfg_dict.get("text_config", {}))

    # flat layout: extract text keys from the top level

        "hidden_size", "num_hidden_layers", "num_attention_heads",
        "num_key_value_heads", "head_dim", "intermediate_size",

```

`pad_token_id` は Qwen2.5-VL では `None` で出てくるので `eos_token_id` にフォールバックします（NxDI の `validate_config()` は `text_config.pad_token_id` の存在を要求するため）。

#### Point 3: vision_config の key remap 4 箇所

HF Qwen2.5-VL の `vision_config` と NxDI の Qwen2-VL ベースクラスで key 名が違います。

| HF 側 key | NxDI 側 key | 操作 |
|---|---|---|
| `in_chans` | `in_channels` | リネーム |
| `hidden_size` | `embed_dim` | 複製 |
| `intermediate_size / hidden_size` | `mlp_ratio` | 計算 |
| `out_hidden_size` | `hidden_size` (上書き) | merger 出力次元を text hidden に揃える |

特に最後の **`out_hidden_size` を `hidden_size` に上書き**しないと、PatchMerger の出力次元が `1280` のままになり、text 側の `3584`（7B）/`5120`（32B）への scatter で shape mismatch を起こします。

#### Point 7: `transformers >= 4.52` の `layer_types` 二重 truncate

`num_hidden_layers` を縮める場合、top-level と `text_config` の **両方の `layer_types` を truncate** する必要があります。片方だけだと validation error になります。

```python
def _truncate_layers(cfg, n):
    if hasattr(cfg, "num_hidden_layers"):
        cfg.num_hidden_layers = n
    if hasattr(cfg, "layer_types") and cfg.layer_types is not None:
        cfg.layer_types = list(cfg.layer_types)[:n]
    if hasattr(cfg, "text_config") and cfg.text_config is not None:
        tc = cfg.text_config
        if hasattr(tc, "num_hidden_layers"):
            tc.num_hidden_layers = n
        if hasattr(tc, "layer_types") and tc.layer_types is not None:
            tc.layer_types = list(tc.layer_types)[:n]
```

### 2C. アーキテクチャ差分

#### Point 4: M-RoPE の cos_cache stale 回避

NxDI の `NeuronAttentionBase` は CTE で計算した `cos_cache` / `sin_cache` を TKG ステップにそのまま引き継ぎます。Llama 系の RoPE では問題ありませんが、**M-RoPE は position-dependent なので毎ステップ再計算が必要**です。

引き継ぎが起きると、TKG ループが「prefill 末尾の position」での rotary を使い回し、`Paris.` の無限ループや日本語ひらがな繰り返しを生成します。

サンプルでは `apply_rotary_embedding` を以下のように override しています（コメントは要約。実ファイルには `Root cause of TKG divergence (Agent investigation 2026-05-07)` から始まる詳細な根本原因説明が付いています）。

```python
def apply_rotary_embedding(
    self, Q, K, V, position_ids, cos_cache, sin_cache, use_polar_compatible_rope
):
    if self.rotary_emb is not None:
        # Always recompute per step (XLA fusion absorbs the cost when
        # shapes are static within a NEFF).
        cos_cache, sin_cache = self.rotary_emb(V, position_ids)
        Q, K = apply_multimodal_rotary_pos_emb(
            Q, K, cos_cache, sin_cache, self.mrope_section
        )
    return Q, K, cos_cache, sin_cache
```

#### Point 5: Vision tower の RMSNorm + SwiGLU 刷新

Qwen2-VL → Qwen2.5-VL で vision encoder が以下のように変更されました。

| 項目 | Qwen2-VL | Qwen2.5-VL |
|---|---|---|
| `norm1`/`norm2`/`merger.ln_q` | LayerNorm（weight + bias） | **RMSNorm（weight only）** |
| VisionBlock MLP | `fc1 + GELU + fc2`（2-matrix） | **`gate_proj + up_proj + down_proj`（SwiGLU 3-matrix）** |

NxDI 公式の Qwen2-VL クラスをそのまま使うと、checkpoint ロード時に大量の `missing_keys`（`norm.bias`）と `unexpected_keys`（`gate_proj/up_proj/down_proj`）が出て、vision embedding が garbage になります。

サンプルでは `modeling_qwen25vl_vision.py` で Qwen2.5-VL 互換クラスを実装しています。SwiGLU MLP の Tensor Parallel は次のような形です。

```python
class Qwen25VLVisionMlp(nn.Module):
    def __init__(self, dim, hidden_dim, hidden_act, dtype=torch.bfloat16):
        super().__init__()
        self.gate_proj = ColumnParallelLinear(dim, hidden_dim, gather_output=False, dtype=dtype)
        self.up_proj   = ColumnParallelLinear(dim, hidden_dim, gather_output=False, dtype=dtype)
        self.down_proj = RowParallelLinear(
            hidden_dim, dim, input_is_parallel=True, dtype=dtype, reduce_dtype=dtype
        )
        self.act_fn = ACT2FN[hidden_act]

    def forward(self, x):
        return self.down_proj(self.act_fn(self.gate_proj(x)) * self.up_proj(x))
```

#### Point 6: `eos_token_id` は 2 値必要

`<|im_end|>=151645` だけだと `<|endoftext|>=151643` で止まらず、生成が `max_new_tokens` まで走り切ります。両方を `GenerationConfig` に渡します。

```python
gen_config = GenerationConfig(
    max_new_tokens=MAX_NEW_TOKENS,
    do_sample=False,
    repetition_penalty=1.05,
    pad_token_id=processor.tokenizer.pad_token_id,
    eos_token_id=[151645, 151643],
)
```

## §6 トラブルシューティング

実機で踏みやすい順に並べます。

### 6.1 `libneuronpjrt-path` not found

```text
FileNotFoundError: [Errno 2] No such file or directory: 'libneuronpjrt-path'
```

vLLM 用ではなく NxDI 用の venv を activate してください: `source /opt/aws_neuronx_venv_pytorch_2_9_nxd_inference/bin/activate`。 すでに別の venv を activate している場合は `deactivate` してから再実行してください（§3.3 参照）。

### 6.2 `PermissionError: /var/tmp/neuron-compile-cache/...`

DLAMI で `/var/tmp/neuron-compile-cache` の所有者が別ユーザーになっているケースです。`NEURON_COMPILE_CACHE_URL` をユーザー所有のパスに切り替えます（§3.5 参照）。

### 6.3 vLLM 経由で起動して `Model Qwen2_5_VLForConditionalGeneration is not supported`

```text
ValueError: Model Qwen2_5_VLForConditionalGeneration is not supported on Neuron for now.
Supported models: ['gpt_oss', 'llama', ..., 'qwen2_vl', ..., 'qwen3_vl']
```

vLLM Neuron plugin v0.16 は `qwen2_5_vl` 未対応です。本サンプルの NxDI 直叩き経路を使ってください。

### 6.4 日本語が 3 step で破綻する

`padding_side="left"` 系の設定が残っています。Point 1 を再確認してください。

### 6.5 Decode が `Paris.` / ひらがな繰り返しでループする

cos_cache stale の典型症状です。`apply_rotary_embedding` の override が外れているか、`rotary_emb` を毎ステップ呼んでいるかを確認してください（Point 4）。

### 6.6 `python -c` 内で `$HOME` などのパス変数が展開されない

`python -c '...'` の単引用符は shell 展開を抑止するため、 `'$HOME/...'` は文字列 `$HOME/...` のままで Python に渡されます。 §3.4 の例のように `python - <<"PYEOF"` ヒアドキュメント + `os.path.expanduser('~/...')` を使うか、 単引用符を二重引用符に変えてください。

### 6.7 32B が CPU メモリで OOM する

`Qwen2_5_VLForConditionalGeneration.from_pretrained` で bf16 model 全体を一旦 CPU にロードします。 32B の場合 64 GB RAM が必要で、 trn2.3xlarge は 64 GB RAM のため余裕がありません。 `low_cpu_mem_usage=True`（サンプル既定）でも OOM したら、 swap を有効化するか、 RAM の大きいインスタンスで checkpoint 変換だけ先に行ってから `HF_CKPT_DIR` 経由で持ち込むのが確実です。
