## はじめに

RL post-training のフレームワークを性能の観点で比較しようとすると、必ず「weight sync は CUDA IPC か NCCL か」「rollout データはどう運ばれるか」「advantage の正規化はノード間通信を伴うのか」といった通信レイヤの話に行き着きます。

本記事は、slime と verl という二つのフレームワークの通信プリミティブの違いについて整理することが目的です。今後性能検証を実施するため性能で差が出そうなポイントをあらかじめ実装から仮説を立てておきたい意図があります。

記述はすべて各プロジェクトの GitHub ソース (2026 年 6 月時点、slime `v0.3.0` / verl `v0.8.0`) を一次情報とし、確証が取れていない部分は「未確認」と明示します。リンクはリンク切れを避けるためこのリリースタグに固定しています。

## なぜ RL post-training で通信が重要になるのか

RL post-training のループは、大きく 3 つの計算フェーズに分かれます。まずそれぞれが何をするものかを整理します。

| フェーズ | 担当 | 何をするか | 出力 |
|---------|------|-----------|------|
| **① rollout** | 推論エンジン (vLLM/SGLang) | いまのモデル (ポリシー) にプロンプトを与えて応答を生成。 | sequences/log-prob |
| **② reward + advantage** | フレームワーク本体 | 応答を「どれくらい良かったか」スコア付けし、「平均と比べてどれくらい良い/悪い行動か」を表す **advantage** を計算。 | advantage |
| **③ training** | 訓練バックエンド (Megatron/FSDP) | advantage を使ってポリシー勾配を計算し、モデルの重みを更新。大規模モデルを TP/PP/DP で分割しながら backward と optimizer step を回す | updated weight |

この 3 つが「生成 → 評価 → 学習 → (更新後重みで) また生成」と 1 ステップずつ回るのが RL post-training のループです。

ここで教師ありの事前学習と決定的に異なるのが、**rollout を担う推論エンジンと training を担う訓練バックエンドが、別の重みコピーを持つ**点です。事前学習なら更新するモデルは 1 つですが、RL では「生成に使うモデル」と「勾配で更新されるモデル」が物理的に別インスタンス (別 GPU・別ノード) になります。そのため training フェーズで重みを更新したら、その新しい重みを rollout 側のエンジンへ**配り直さないと、次の生成が古いポリシーのまま**になってしまいます。この「更新後の重みを training 側から rollout 側へ配る」処理を **weight sync (重み同期)** と呼び、モデルが大きいほど転送量が膨らむため通信のホットスポットになります。

そして 1 ステップの中では、この weight sync 以外にも複数の通信が**別系統で**走ります。具体的には、(a) rollout が生成したデータを training 側へ運ぶ **rollout データの移動**、(b) advantage 計算の際にデータ並列の各ランク間で統計を揃える **advantage 正規化**、(c) training の backward で勾配をデータ並列ランク間で集約する **gradient 同期**、です。

本記事で繰り返し出てくる用語を先に押さえておきます。

- **colocated (コロケート)**: training と rollout を同じ GPU セット上に同居させる構成。重みの受け渡しがノード内 (NVLink / CUDA IPC) で閉じる。
- **disaggregated (分離)**: training クラスタと rollout クラスタを別の物理ノードに分ける構成。weight sync がノード間 (EFA) を流れる。

## RL 特有の通信と、そうでない通信

RL post-training の 1 ステップで走る通信は、「RL post-training に特有の通信」と「事前学習 (Megatron / FSDP) で使う通信」の 2 種類に分けると見通しが良くなります。slime と verl で実装が割れるのは前者だけで、後者は [Megatron-Core](https://github.com/NVIDIA/Megatron-LM?tab=readme-ov-file#setup) / NCCL がそのまま提供するため両者でほぼ共通です。

まず slime の 1 ステップ (後述の図 2 参照) を、RL 特有 or Not で表にまとめます。`source → dest` は「どのプロセスからどのプロセスへ流れるか」、プリミティブは実際に使う通信ライブラリです。

| 通信 | source → dest | プリミティブ (slime) | 分類 |
|------|--------------|---------------------|------|
| rollout データ収集 | rollout → train | Ray object store (既定) / NIXL (任意) | **RL 特有** |
| advantage 正規化 | train DP group 内 | NCCL AllReduce | **RL 特有** |
| weight sync | train → rollout | NCCL broadcast / disk / delta | **RL 特有** |
| DP gradient 同期 | train DP group 内 | NCCL AllReduce / reduce-scatter | 事前学習と共通|
| TP | train TP group 内 | NCCL (NVLink 中心、TP がドメイン超なら EFA) | 事前学習と共通 |
| PP | train PP group 内 | NCCL P2P send/recv (ノード間なら EFA) | 事前学習と共通 |
| MoE expert all-to-all | train EP group 内 | AWS EFA では UCCL-EP | 事前学習と共通 |

ポイントは 2 つです。

- **RL 特有の通信** (rollout データ収集・advantage 正規化・weight sync) は、推論エンジンと訓練バックエンドが別の重みコピーを持つ RL ならではのもので、**slime と verl の実装差が集中する**場所です。本記事が主に整理するのはここです。
- **事前学習と共通の通信** (gradient 同期・TP/PP・MoE all-to-all) は Megatron-Core / NCCL がそのまま担い、両フレームワークで本質的な差はありません。

この分類を前提として、まず slime をスタック (図1) とフロー (図2) で読み、続いて verl (図3・図4) を対比します。図1・図2 が slime、図3・図4 が verl です。

## slime の通信スタック (図1)

まず slime が実際に使う通信プリミティブを、スタック階層で並べたのが図 1 です。

![slime が使う通信プリミティブのスタック階層](/images/1f6589ff93dca7-slime-comm-stack.png)

図の読み方は色と形で決まっています。**色付きの実線ボックスが slime が実際に使うもの**、**グレーの破線ボックスは「要素技術としてどこに位置するかは示すが、AWS EFA では使わないもの」** です。slime の通信は、以下に集約されます。

### L3 - Comm abstraction / collective library

- **NCCL**: gradient の AllReduce と、disaggregated 構成での weight sync broadcast に使います。AWS では `aws-ofi-nccl` が NCCL のトランスポートとして libfabric (EFA provider) をぶら下げ、最終的に EFA NIC に到達します。
- **Gloo**: 重みなど **GPU 上の大きいテンソルの集団通信は速い NCCL が担い**、Gloo はその裏方として **CPU 側の軽い処理** (各フェーズの足並みを揃える barrier や、小さなメタデータの受け渡し) を担当します。「NCCL = GPU の本データ、Gloo = CPU 側の制御」と役割分担していると捉えてください (Gloo は NCCL が無い CPU 環境向けの汎用バックエンドでもありますが、本記事の GPU 前提ではこの裏方役です)。
- **Ray object store**: rollout で生成したデータ (sequences / log-prob) の収集します。
- **NIXL**: SGLang 経由で rollout データ転送を最適化する任意経路です。
- **UCCL-EP** : MoE モデルを扱う場合、expert の all-to-all 通信に使われます。UCCL 関連については別の記事で扱うのでこういうものがあるくらいの認識で大丈夫です。

### L2 — Transport / proxy・backend

L3 の collective ライブラリ (NCCL など) は、自分で直接 NIC を叩くわけではありません。「どのトランスポートに載せるか」を担うのが L2 です。AWS EFA でこの記事に関わるのは次の 2 つです。

- **`aws-ofi-nccl`**: NCCL の **NET プラグイン**です。ここでいう NET プラグインとは、NCCL が「ネットワーク送受信の部分だけを外部から差し替えられるように」用意した拡張の口のことです。NCCL は `connect` / `isend` / `irecv` / `regMr` (メモリ登録) といった**決まった形の関数群**を [net plugin インターフェース](https://github.com/NVIDIA/nccl/tree/master/plugins/net) として定義していて、それを実装した共有ライブラリを置いておくと、NCCL は起動時に読み込んで自分のネットワーク処理をそっくり差し替えます (家電のコンセントに各規格の変換アダプタを挿すイメージ)。`aws-ofi-nccl` はこの口にハマるアダプタで、**NCCL から来た送受信の依頼を libfabric の呼び出しに変換 (マップ) する**部品です ([README](https://github.com/aws/aws-ofi-nccl) に "implements a plug-in which maps NCCLs ... transport APIs to libfabric's ... interface" と明記)。これがあるおかげで NCCL 本体を一切改造せずに通信を EFA に流せます。図で NCCL から下に伸びる経路はまずここを通ります。
- **`libfabric` (+ EFA provider)**: ベンダ非依存の通信 API (OFI = OpenFabrics Interfaces) を提供するライブラリです。実際の NIC ごとの実装は「provider」というプラグインで差し替えられ、EFA 用の実装が **EFA provider** ([`ofiwg/libfabric` の `prov/efa/`](https://github.com/ofiwg/libfabric/tree/main/prov/efa)) です。`aws-ofi-nccl` から見た libfabric は「OFI API を叩けば、あとは EFA provider が EFA のお作法 (SRD など) に翻訳して NIC まで届けてくれる」共通の窓口にあたります。libfabric と SRD の詳細は前提記事「[AWS EFA と Nitro System の仕組みを理解する](https://zenn.dev/tosshi/articles/0eeb53ca63f8b2)」で解説済みなので、ここでは「NCCL → `aws-ofi-nccl` → libfabric (EFA provider)」という縦のつながりだけ押さえてください。

なお図の L2 には `UCX (NIXL SRD backend)` も置いていますが、UCX は PD 分離推論関係でこれまでの記事でも繰り返し触れてきたため、本記事では割愛します。

### L1 — Kernel driver / GPU memory direct

L2 の libfabric (EFA provider) は、最終的に **カーネル空間のドライバ**を通して NIC のハードウェアに触れます。その本体が **`efa.ko`** です。

- **`efa.ko` (EFA driver)**: AWS の Elastic Fabric Adapter 用の **Linux カーネルモジュール**です。役割は「EFA という NIC を OS に認識させ、RDMA (InfiniBand verbs) のインターフェースとしてユーザー空間に見せる」こと。具体的には、デバイスの初期化・認識 (probe)、**Queue Pair (QP) や Memory Region (MR) といった RDMA 資源の生成・管理**、ファームウェアとのやり取りを担います。重要なのは、`efa.ko` 自身がデータ 1 バイトごとを運ぶわけではない点です。一度 QP と MR をセットアップしてしまえば、以降の実データ送受信はユーザー空間から NIC を直接叩く (**カーネルバイパス**) ため、`efa.ko` は「RDMA を使える状態をお膳立てする管理役」と捉えると分かりやすいです。

AWS が配布する [`amzn/amzn-drivers` の `kernel/linux/efa/`](https://github.com/amzn/amzn-drivers/tree/master/kernel/linux/efa) に実装があり、GPUDirect RDMA 対応 ([`efa_p2p.c`](https://github.com/amzn/amzn-drivers/blob/master/kernel/linux/efa/src/efa_p2p.c) / `efa_nvmem_v2.c`) や Neuron 対応 (`efa_neuronmem.c`) を含む拡張です。利用者向けの導入手順は [AWS 公式ドキュメント](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/efa.html) を参照してください。

## slime の RL 1 ステップ (図2)

図2 は「RL の 1 ステップで、何が、どこを流れるか」の時系列です。

![slime の GRPO 1 イテレーションの通信フェーズ](/images/1f6589ff93dca7-slime-rl-comm-flow.png)

**ノード間 EFA を流れる通信を、黄色帯 (中央) の中にオレンジ Box**で配置しました。オレンジ Box が「実際に EFA を通るデータ」、灰箱が「ノード内 (NVLink・IPC) で完結する通信」を表します。

1. **① 応答生成**: rollout engine (SGLang) が sequences / tokens を生成します。
2. **② データ収集**: 生成された sequences と log-prob を training 側へ運びます。デフォルトは **Ray object store** (`ray.put` / `ray.get`)、オプションで **NIXL+UCX** の RDMA 経路です。disaggregated 構成ではこれがノード間を流れるのでオレンジ箱です。
3. **③ reward 計算 + advantage 正規化**: slime は `--normalize-advantages` 指定時に advantage を DP group 全体で正規化し、`distributed_masked_whiten` が **NCCL AllReduce** をかけます (`slime/utils/distributed_utils.py`、グループは `mpu.get_data_parallel_group()`)。DP group がノード間に分散していれば、この AllReduce も EFA を通ります (図2 では簡潔さのため ③ の EFA 箱は省略していますが、後述の図4 verl との対比ではこの ③ が分岐点になります)。reward 自体はルールベースならローカル、外部 reward service なら HTTP です。
4. **④ Train**: 一般的な分散学習の TP/PP/DP の通信の話なので割愛します。
5. **⑤ Weight Sync**: training で更新した重みを rollout に配ります。これが weight sync で、転送方式は構成で選択します。

### slime の weight sync は複数方式

「複数方式」とは、**weight sync をどの経路で運ぶかが構成によって変わり、その構成ごとに別々の実装クラスが用意されている**という意味です。具体的には「training と rollout が同じノードか別ノードか (colocated / disaggregated)」「全部の重みを送るか差分だけ送るか (full / delta)」の組み合わせで、使われるクラスが切り替わります。

slime はこの切り替えを [`actor.py`](https://github.com/THUDM/slime/blob/v0.3.0/slime/backends/megatron_utils/actor.py) の手書き分岐 (`colocate` か、`update_weight_mode` が `delta` か) で行います。実装クラスは次の 3 つです (v0.3.0)。

| 構成 | クラス (GitHub) | 運び方 | 経路 |
|------|----------------|--------|------|
| colocated | [`UpdateWeightFromTensor`](https://github.com/THUDM/slime/blob/v0.3.0/slime/backends/megatron_utils/update_weight/update_weight_from_tensor.py#L24) | Gloo gather + Ray IPC (CUDA IPC handle) | ノード内 |
| disaggregated / full | [`UpdateWeightFromDistributed`](https://github.com/THUDM/slime/blob/v0.3.0/slime/backends/megatron_utils/update_weight/update_weight_from_distributed.py#L23) | NCCL broadcast (`slime-pp_{pp_rank}` group) | ノード間 (EFA) |
| disaggregated / delta | [`UpdateWeightFromDistributedDelta`](https://github.com/THUDM/slime/blob/v0.3.0/slime/backends/megatron_utils/update_weight/update_weight_from_distributed_delta.py#L480) | 差分のみを送る。transport は NCCL broadcast (`nccl`) と safetensors 共有 FS (`disk`) を選べる | ノード間 (EFA) / 共有 FS |

delta クラスは、差分を「ネットワークで送る (`nccl`)」か「safetensors として共有ディスクに書く (`disk`)」かを内部で選べます。**delta sync 自体が slime の特徴的な実装**です。RL fine-tuning では step 間で BF16 の大半のバイトが変化しないため、前回との差分だけを送ることで転送量を大きく削れます。

## verl の通信スタック (図3) — slime との差は「抽象化」

同じ視点で verl のスタックを描いたのが図3 です。slime の図1 と見比べてください。

![verl が使う通信プリミティブのスタック階層](/images/1f6589ff93dca7-verl-comm-stack.png)

L0〜L2 (EFA NIC / efa.ko / libfabric / aws-ofi-nccl / GPUDirect RDMA) は、AWS 上で動かす以上、最終的に EFA に届く経路として共通の前提層になります。ただしこの層はフレームワークのコードではなく実行環境が提供するもので、リポジトリ成果物としては非対称です。verl は AWS EFA 向けの Dockerfile ([`docker/aws/Dockerfile.extention.awsefa`](https://github.com/volcengine/verl/blob/v0.8.0/docker/aws/Dockerfile.extention.awsefa)、aws-efa-installer + aws-ofi-nccl + libfabric + `FI_PROVIDER=efa`) を同梱しますが、slime はこの層を同梱せず実行環境任せです。**フレームワークのコードとして見たときの違いは L3 の weight sync の抽象化**にあります。

ここがこの記事の肝です。slime は weight sync を「用途別のクラスを手書きの `if/elif` で選ぶ」のに対し、verl は **[`CheckpointEngineRegistry`](https://github.com/volcengine/verl/blob/v0.8.0/verl/checkpoint_engine/base.py#L49) という pluggable な registry** で管理します。v0.8.0 時点で登録されているのは次の 5 クラスです。

| backend キー | クラス (GitHub) | 下層トランスポート |
|------------|----------------|------------------|
| `naive` | [`ColocatedCheckpointEngine`](https://github.com/volcengine/verl/blob/v0.8.0/verl/checkpoint_engine/base.py#L221) | 同一 GPU 上で weight generator を in-memory 受け渡し (torch.distributed / all_gather) |
| `nccl` | [`NCCLCheckpointEngine`](https://github.com/volcengine/verl/blob/v0.8.0/verl/checkpoint_engine/nccl_checkpoint_engine.py#L103) | NCCL broadcast + ZMQ PUB/SUB (メタデータ) |
| `nixl` | [`NIXLCheckpointEngine`](https://github.com/volcengine/verl/blob/v0.8.0/verl/checkpoint_engine/nixl_checkpoint_engine.py#L239) | UCX + IB RDMA の ring 転送 |
| `mooncake` | [`MooncakeCheckpointEngine`](https://github.com/volcengine/verl/blob/v0.8.0/verl/checkpoint_engine/mooncake_checkpoint_engine.py#L35) | Mooncake TransferEngine (RDMA P2P chain) |
| `kimi_ckpt_engine` | [`KIMICheckpointEngine`](https://github.com/volcengine/verl/blob/v0.8.0/verl/checkpoint_engine/kimi_checkpoint_engine.py#L223) | Mooncake P2P + NCCL broadcast |
| `nccl` キー上書き | [`HCCLCheckpointEngine`](https://github.com/volcengine/verl/blob/v0.8.0/verl/checkpoint_engine/hccl_checkpoint_engine.py#L97) | HCCL (Ascend NPU) |

図3 のポイントを整理すると、

- **verl だと、weight sync が「差し替え式」**。NCCL broadcast だけでなく NIXL ring や Mooncake P2P chain を config で選べます。slime は full sync が NCCL broadcast 一択です。
- **verl だと、NIXL は weight sync 用**。slime では NIXL は rollout データ転送に使われるのに対し、verl の `NIXLCheckpointEngine` は weight sync の ring 転送 (`trainer→r1→r2→…→rN`) に使われます。**同じ NIXL でも用途が逆**です。
- **verl だと、汎用の sparse delta sync がない**。任意パラメータの位置・値のみを送る方式は Draft PR #6794 が提案中で未マージです。
- **MoE の expert all-to-all (UCCL-EP) は slime と対称**。

## verl の RL 1 ステップ (図4) — advantage と KV cache が分岐点

verl の 1 ステップを、図2 と同じレイアウトで描いたのが図4 です。

![verl の GRPO/PPO 1 イテレーションの通信フェーズ](/images/1f6589ff93dca7-verl-rl-flow.png)

slime の図2 と「箱の色」を見比べると、verl の構造差が一目で分かります。

- **③ advantage 正規化が灰箱 (ローカル CPU のみ)**。slime は `--normalize-advantages` 時に `distributed_masked_whiten` で NCCL AllReduce をかけてグローバル統計で正規化しますが、**verl だと `masked_whiten` がローカル統計のみで完結し、distributed collective を使いません** (`verl/trainer/ppo/core_algos.py`、`verl/utils/torch_functional.py`)。だから図4 では ③ が EFA 帯に出ません。これは「同じ GRPO でも advantage の正規化スコープが違う」という設計差で、学習の安定性にも効きうる重要な分岐です。
- **①b KV cache PD 分離**。verl は SGLang の prefill-decode 分離をサポートし、prefill が生成した KV cache を `disaggregation_transfer_backend` (NIXL / Mooncake) で decode 側に転送します (`verl/workers/rollout/sglang_rollout/sglang_pd_replica.py`)。**slime も同様に PD 分離をサポートします** (`--prefill-num-servers`、専用ドキュメント、mooncake/nixl テスト)。どちらも KV の物理転送は SGLang に委譲しており、PD 分離の有無は両者の差別化点ではありません。図4 では rollout 側に ①b の箱を置き、KV cache 転送をオレンジ箱で帯に出しています。
- **② データ収集は Ray のみ**。verl の rollout データ収集は `DataProto` / `DataProtoFuture` (Ray object store) だけで、**slime のような NIXL オプションはありません**。

④ (gradient 同期) と ⑤ (weight sync) が別系統である点、内部分散の経路が軸ごとに割れる点 (TP は原則ノード内 NVLink、PP と DP はノード間 EFA に出やすい) は slime と共通です。verl では FSDP backend なら gradient 同期が `loss.backward()` 内の reduce-scatter で暗黙に走り、Megatron backend なら DDP の AllReduce で走ります。いずれも weight sync の CheckpointEngine とはコードパスが分離しています。

## まとめ

本記事では、slime をベースに RL post-training の 1 ステップで発生する通信を実装レベルで verl と対比しました。

- **共通の骨格**: 両者とも actor (Megatron/FSDP) と rollout (SGLang/vLLM) を分け、AWS では NCCL → aws-ofi-nccl/libfabric → EFA、MoE の expert all-to-all は UCCL-EP → libibverbs/efadv (SRD) → EFA という下層に到達する (この下層は実行環境が提供し、verl は AWS EFA Dockerfile を同梱・slime は同梱しない)。SGLang の KV cache PD 分離、MoE の expert all-to-all (EFA では UCCL-EP) も両者がサポートする。Ray object store と Gloo は EFA でなく TCP (eth0) を通る点も共通。
- **weight sync の抽象化**: slime は用途別の独立クラスを分岐で選ぶ (registry なし)、verl は `CheckpointEngineRegistry` の pluggable backend (NCCL/NIXL/Mooncake/Kimi/HCCL)。
- **slime にあって verl の main にないもの**: 汎用の sparse delta weight sync、NIXL を rollout データ転送に使う経路、advantage のグローバル分散正規化。
- **verl にあって slime にないもの**: weight sync の差し替え式抽象、NIXL/Mooncake を weight sync 本体に使う経路、elastic scaling。

これらの構造差から、weight sync のトポロジー (broadcast vs ring)、delta 転送の有無、advantage 正規化の通信コストという形で、性能差が出そうなポイントを 6 つ立てました。**次のステップは、これらを実際に B300 環境で計測して仮説を検証すること**です。実測結果は別記事で報告する予定です。
