## はじめに

:::message
この記事は、大規模 LLM の RL post-training (GRPO / PPO) で発生するノード間通信を、フレームワークの実装に踏み込んで整理する記事です。題材として **slime** (THUDM) をベースに「どの処理が、どの通信プリミティブで、ノード内なのかノード間 (EFA) なのか」を 1 ステップ分そろえて示し、それに対比する形で **verl** (volcengine) が「同じフェーズをどう実装しているか」を並べます。RDMA・GPUDirect RDMA・EFA・MoE の基礎は筆者の既存記事に委ね、本記事は通信プリミティブの棚卸しと 2 フレームワークの構造差に集中します。AWS EFA 環境を前提とし、InfiniBand 専用機構 (IBGDA など) は対象外です。
:::

RL post-training のフレームワークを性能の観点で比較しようとすると、必ず「weight sync は CUDA IPC か NCCL か」「rollout データはどう運ばれるか」「advantage の正規化はノード間通信を伴うのか」といった通信レイヤの話に行き着きます。ところがこの層は名前の似た略語が多く、どれがどのレイヤで、どれが何に依存するのかが見えにくいのが実情です。

本記事のスタンスは、**まず slime のコードを 1 本の縦串として読み切り、その対比軸で verl を読む**ことです。両者は「Megatron + SGLang/vLLM で actor と rollout を分離する」という共通骨格を持ちながら、weight sync の抽象化、rollout データ転送、advantage 正規化の置き場所が実装レベルで異なります。この差は、後半で述べる「性能で差が出そうなポイント」に直結します。

記述はすべて各プロジェクトの GitHub ソース (2026 年 6 月時点 main ブランチ) を一次情報とし、確証が取れていない部分は「未確認」と明示します。

## なぜ RL post-training で通信が支配的になるのか

RL post-training のループは、3 つの計算フェーズに分かれます。現在のポリシーで応答を生成する rollout (推論エンジンが担当)、報酬計算と advantage 推定、そしてポリシー勾配でモデルを更新する training (訓練バックエンドが担当) の 3 つです。

教師ありの事前学習と決定的に異なるのは、rollout を担う推論エンジン (vLLM / SGLang) と training を担う訓練バックエンド (Megatron / FSDP) が、別の重みコピーを持つ点です。1 ステップごとに、訓練側で更新した重みを推論側へ配り直す処理を **weight sync (重み同期)** と呼び、ここが通信のホットスポットになります。さらに gradient 同期、rollout データの移動、advantage の分散正規化など、複数の通信が 1 ステップの中で別系統で走ります。

本記事で繰り返し出てくる用語を先に押さえておきます。

- **colocated (コロケート)**: training と rollout を同じ GPU セット上に同居させる構成。重みの受け渡しがノード内 (NVLink / CUDA IPC) で閉じます。
- **disaggregated (分離)**: training クラスタと rollout クラスタを別の物理ノードに分ける構成。weight sync がノード間 (EFA) を流れます。
- **TP / PP / DP / EP**: モデル並列の軸で、それぞれ Tensor Parallel (層内の行列分割) / Pipeline Parallel (層をステージ分割) / Data Parallel (データ並列、勾配を集約) / Expert Parallel (MoE の expert をノードに分散) を指します。TP は通常ノード内、DP はノード間に分散するのが典型です。

以降は、この 1 ステップで発生する通信を slime のコードに沿って 4 枚の図で見ていきます。図1・図2 が slime、図3・図4 が verl です。

## slime の通信スタック (図1)

まず slime が実際に使う通信プリミティブを、スタックの上 (アプリ) から下 (ハードウェア) まで並べたのが図1 です。

![slime が使う通信プリミティブのスタック階層](/images/up114vtyv1obg-comm-stack.png)

図の読み方は形で決まっています。**角丸の実線ボックスが slime が実際に使うもの**、**六角形の破線グレーが slime が使わないもの**です。slime の通信は、以下に集約されます。

- **NCCL** (右): gradient の AllReduce と、disaggregated 構成での weight sync broadcast。AWS では `aws-ofi-nccl` が NCCL のトランスポートとして libfabric (EFA provider) をぶら下げ、最終的に EFA NIC に到達します。
- **Gloo** (中央): TP シャードの gather や barrier など CPU 側の集約・同期。
- **Ray object store** (左): rollout で生成したデータ (sequences / log-prob) の収集。
- **NIXL** (左): SGLang 経由で rollout データ転送を最適化する任意経路 (`ray.put(_tensor_transport="nixl")`)。`--rollout-data-transport nixl` で有効化します。
- **DeepEP / NVSHMEM** (中央): MoE モデルを扱う場合、expert の all-to-all 通信に使われます。slime は `--sglang-moe-a2a-backend deepep` (SGLang 側) / `--moe-enable-deepep` (Megatron 側) で DeepEP を有効化でき、DeepEP は下層で NVSHMEM を使います (`build_conda.sh` で依存をインストール)。ただし slime 自身が DeepEP の dispatch/combine カーネルを実装しているわけではなく、MoE expert parallel の実体は Megatron core / SGLang に委譲しています。

逆に、**UCCL は slime では使われません** (六角形破線)。UCCL は後述する verl の NIXL checkpoint engine のトランスポート選択肢としては実在します。

ここで強調したいのは、**「EFA は単独で何でも運ぶ層」ではない**ということです。L3 の NCCL や NIXL が、その下の L2 (aws-ofi-nccl / libfabric / UCX) を経由して初めて EFA に届きます。GPU メモリを直接転送する場合は、条件が揃えば GPUDirect RDMA が併用されます。なお、この L0〜L2 のスタック (EFA NIC / efa.ko / libfabric / aws-ofi-nccl / GPUDirect RDMA) は、フレームワークのコードではなく実行環境 (コンテナイメージやホストのドライバ) が提供する層です。slime はこの層を同梱せず実行環境に任せます。この「上から下への依存チェーン」が、後で性能を考えるときの土台になります。

## slime の RL 1 ステップ (図2)

図1 が「何が使えるか」の地図なら、図2 は「1 ステップで、何が、どこを流れるか」の時系列です。

![slime の GRPO 1 イテレーションの通信フェーズ](/images/up114vtyv1obg-rl-comm-flow.png)

この図の肝は、**ノード間 EFA を流れる通信を、黄色帯 (中央) の中にオレンジ箱として置いた**ことです。オレンジ箱が「実際に EFA を通るデータ」、灰箱が「ノード内 (NVLink・IPC) で完結する通信」を表します。形と位置だけで「どこに帯域がかかるか」が分かるようにしてあります。番号順に追います。

1. **① 応答生成**: rollout engine (SGLang) が sequences / tokens を生成します。KV cache は engine 内で完結し、ノード間転送はありません (灰箱)。
2. **② データ収集**: 生成された sequences と log-prob を training 側へ運びます。デフォルトは **Ray object store** (`ray.put` / `ray.get`)、オプションで **NIXL+UCX** の RDMA 経路です (`slime/ray/rollout.py` の `_split_train_data_by_dp`)。disaggregated 構成ではこれがノード間を流れるのでオレンジ箱です。**NCCL はこのフェーズに一切関与しません。**
3. **③ reward 計算 + advantage 正規化**: slime は `--normalize-advantages` 指定時に advantage を DP group 全体で正規化し、`distributed_masked_whiten` が **NCCL AllReduce** をかけます (`slime/utils/distributed_utils.py`、グループは `mpu.get_data_parallel_group()`)。DP group がノード間に分散していれば、この AllReduce も EFA を通ります (図2 では簡潔さのため ③ の EFA 箱は省略していますが、後述の図4 verl との対比ではこの ③ が分岐点になります)。reward 自体はルールベースならローカル、外部 reward service なら HTTP です。
4. **④ Train 内部分散**: TP/PP は通常ノード内 NVLink で閉じる (灰箱) のに対し、**DP の gradient 同期 (NCCL AllReduce / reduce-scatter) はノード間** を流れます (オレンジ箱)。同じ「内部分散」でも経路が割れる点を、点線枠の中で対比させています。
5. **⑤ Weight Sync**: training で更新した重みを rollout に配ります。これが weight sync で、転送方式は構成で選択します (後述)。

ここで重要なのは、**gradient 同期 (④ の DP) と weight sync (⑤) は別系統**だということです。前者は train クラスタ内 DP の勾配集約、後者は train→rollout の重み配布で、コードパスも通信グループも完全に分離しています (`slime/backends/megatron_utils/model.py` の DDP hook と `actor.py` の `weight_updater.update_weights()` は別の呼び出しサイト)。図でも ④ と ⑤ を別の箱・別の矢印にしているのはこのためです。

### slime の weight sync は複数方式

slime の weight sync (⑤) は、registry のような共通抽象を持たず、`slime/backends/megatron_utils/actor.py` の手書き分岐 (`update_weight_mode` (full/delta) × `update_weight_transport` (nccl/disk) × colocate) で実装クラスが選ばれます。クラスは以下の 4 つです (`UpdateWeightFromDistributedDelta` は `UpdateWeightFromDistributed` を継承し、`UpdateWeightFromTensor` も `UpdateWeightFromDistributed` の関数を再利用するため、完全独立ではありません)。

| 方式 | クラス | プリミティブ | ノード間/内 |
|------|--------|------------|------------|
| colocated | `UpdateWeightFromTensor` | Gloo gather + Ray IPC (CUDA IPC handle) | ノード内 (EFA 不使用) |
| disaggregated full / nccl | `UpdateWeightFromDistributed` | NCCL broadcast (`slime-pp_{pp_rank}` group) | ノード間 (EFA) |
| disaggregated full / disk | `UpdateWeightFromDisk` | safetensors を共有 FS 経由 | 共有 FS |
| disaggregated delta | `UpdateWeightFromDistributedDelta` | 差分のみ NCCL broadcast / disk (safetensors) | ノード間 (EFA) / 共有 FS |

とくに **delta sync は slime の特徴的な実装**です。RL fine-tuning では step 間で BF16 の大半のバイトが変化しないため、前回との差分だけを broadcast することで転送量を大きく削れます。汎用 delta weight sync が main に正式実装されている点が、後の対比の一つの軸になります。

## verl の通信スタック (図3) — slime との差は「抽象化」

同じ視点で verl のスタックを描いたのが図3 です。slime の図1 と見比べてください。

![verl が使う通信プリミティブのスタック階層](/images/up114vtyv1obg-verl-comm-stack.png)

L0〜L2 (EFA NIC / efa.ko / libfabric / aws-ofi-nccl / GPUDirect RDMA) は、AWS 上で動かす以上、最終的に EFA に届く経路として共通の前提層になります。ただしこの層はフレームワークのコードではなく実行環境が提供するもので、リポジトリ成果物としては非対称です。verl は AWS EFA 向けの Dockerfile (`docker/aws/Dockerfile.extention.awsefa`、aws-efa-installer + aws-ofi-nccl + libfabric + `FI_PROVIDER=efa`) を同梱しますが、slime はこの層を同梱せず実行環境任せです。**フレームワークのコードとして見たときの違いは L3 の weight sync の抽象化**にあります。

slime が weight sync を「用途別のクラスを手書き分岐で選ぶ」のに対し、verl は **`CheckpointEngineRegistry` という pluggable な registry** を持ちます (`verl/checkpoint_engine/base.py`)。`@CheckpointEngineRegistry.register(backend)` デコレータで以下が登録され、config で差し替えられます。

| backend キー | クラス | 下層トランスポート |
|------------|--------|------------------|
| `naive` | `ColocatedCheckpointEngine` | 同一 GPU 上で weight generator を in-memory 受け渡し (torch.distributed / all_gather) |
| `nccl` | `NCCLCheckpointEngine` | NCCL broadcast + ZMQ PUB/SUB (メタデータ) |
| `nixl` | `NIXLCheckpointEngine` | UCX + IB RDMA の ring 転送 |
| `mooncake` | `MooncakeCheckpointEngine` | Mooncake TransferEngine (RDMA P2P chain) |
| `kimi_ckpt_engine` | `KIMICheckpointEngine` | Mooncake P2P + NCCL broadcast |
| (`nccl` キー上書き) | `HCCLCheckpointEngine` | HCCL (Ascend NPU) |

図3 では、verl が使うこれらのバックエンドを角丸実線で並べ、**verl が使わない / slime 側にあるもの** (Gloo 専用 barrier、NIXL rollout transport) を右端に六角形破線で置いています。ポイントを整理すると、

- **verl だと、weight sync が「差し替え式」**。NCCL broadcast だけでなく NIXL ring や Mooncake P2P chain を config で選べます。slime は full sync が NCCL broadcast 一択です (registry を持たず、用途別クラスを分岐で選ぶ)。
- **verl だと、NIXL は weight sync 用**。slime では NIXL は rollout データ転送 (図2 の②) に使われるのに対し、verl の `NIXLCheckpointEngine` は weight sync の ring 転送 (`trainer→r1→r2→…→rN`) に使われます。**同じ NIXL でも用途が逆**である点は実装を読まないと取り違えます。
- **verl だと、汎用の sparse delta sync が main にない**。任意パラメータの位置・値のみを送る方式は Draft PR #6794 が提案中で未マージです (slime は同方式を main に正式実装)。ただし verl main にも LoRA adapter の delta 同期 (base 重みを GPU に残して adapter 差分のみ同期する別種の delta) は存在します。

## verl の RL 1 ステップ (図4) — advantage と KV cache が分岐点

verl の 1 ステップを、図2 と同じレイアウトで描いたのが図4 です。

![verl の GRPO/PPO 1 イテレーションの通信フェーズ](/images/up114vtyv1obg-verl-rl-flow.png)

slime の図2 と「箱の色」を見比べると、verl の構造差が一目で分かります。

- **③ advantage 正規化が灰箱 (ローカル CPU のみ)**。slime は `--normalize-advantages` 時に `distributed_masked_whiten` で NCCL AllReduce をかけてグローバル統計で正規化しますが、**verl だと `masked_whiten` がローカル統計のみで完結し、distributed collective を使いません** (`verl/trainer/ppo/core_algos.py`、`verl/utils/torch_functional.py`)。だから図4 では ③ が EFA 帯に出ません。これは「同じ GRPO でも advantage の正規化スコープが違う」という設計差で、学習の安定性にも効きうる重要な分岐です。
- **①b KV cache PD 分離**。verl は SGLang の prefill-decode 分離をサポートし、prefill が生成した KV cache を `disaggregation_transfer_backend` (NIXL / Mooncake) で decode 側に転送します (`verl/workers/rollout/sglang_rollout/sglang_pd_replica.py`)。**slime も同様に PD 分離をサポートします** (`--prefill-num-servers`、専用ドキュメント、mooncake/nixl テスト)。どちらも KV の物理転送は SGLang に委譲しており、PD 分離の有無は両者の差別化点ではありません。図4 では rollout 側に ①b の箱を置き、KV cache 転送をオレンジ箱で帯に出しています。
- **② データ収集は Ray のみ**。verl の rollout データ収集は `DataProto` / `DataProtoFuture` (Ray object store) だけで、**slime のような NIXL オプションはありません**。

④ (gradient 同期) と ⑤ (weight sync) が別系統である点、TP/PP がノード内・DP がノード間である点は slime と共通です。verl では FSDP backend なら gradient 同期が `loss.backward()` 内の reduce-scatter で暗黙に走り、Megatron backend なら DDP の AllReduce で走ります。いずれも weight sync の CheckpointEngine とはコードパスが分離しています。

## 性能で差が出そうなポイント

ここまでの実装差を踏まえると、「slime には xx が無いから原理的にこの処理は遅い / 速いはず」という形の仮説がいくつか立ちます。実測前の予想として、根拠 (実装) と検証方法をセットで表にまとめます。これらは今後の実験で検証していきたいポイントです。

| # | 仮説 | 実装上の根拠 | 検証方法 |
|---|------|------------|---------|
| A | **大規模 disagg では verl の NIXL ring weight sync が slime の NCCL broadcast より速い**。slime は full sync が NCCL broadcast 一択で、root ノードの NIC 帯域が O(N) ファンアウトのボトルネックになる。verl の `NIXLCheckpointEngine` は ring (trainer→r1→r2→…) で各ホップ O(1) 帯域。 | slime: `UpdateWeightFromDistributed` (broadcast)。verl: `nixl_checkpoint_engine.py` (ring one-sided read) | rollout ノード数を 4/8/16 と変え、NCCL backend vs NIXL backend で weight sync レイテンシ計測。NIXL ring がノード数にほぼ線形、NCCL broadcast が degradation する想定。 |
| B | **slime の汎用 delta sync が full sync を大幅に上回る (最大 ~100x ペイロード削減)**。step 間で BF16 の >99% が不変。verl main には同方式 (任意パラメータの sparse delta) がなく Draft PR #6794 のみ。 | slime: `UpdateWeightFromDistributedDelta`。verl: Draft PR #6794 (LoRA adapter delta は別途存在) | slime で `update_weight_mode=full` vs `delta` を切り替え、転送バイト数と所要時間を計測。差分計算とディスク I/O のオーバーヘッドも併せて計測。 |
| C | **verl の advantage 正規化はローカルで通信ゼロ、slime は (`--normalize-advantages` 時) DP size に比例した AllReduce コストが増える**。一方 slime はグローバル統計で正規化するため、大規模 DP では学習安定性が異なる可能性。 | slime: `distributed_masked_whiten` (NCCL AllReduce)。verl: `masked_whiten` (ローカルのみ) | DP size を 4/8/16/32 と変え、advantage フェーズの通信時間を nsys で計測 (verl は定数、slime は比例の想定)。同時に reward カーブを比較し収束への影響を見る。 |
| D | **colocated の weight sync は slime (Gloo gather + Ray IPC) と verl (in-memory generator 受け渡し) で経路が異なる**。slime は CPU バッファ経由の gather を挟むのに対し、verl の `naive` は同一 GPU 上で generator をそのまま渡す。 | slime: `UpdateWeightFromTensor` (Gloo gather + Ray IPC)。verl: `ColocatedCheckpointEngine` (in-memory) | 同一ノード colocated 構成でモデルサイズ別 (7B/30B/70B) に weight sync レイテンシと CPU 使用率を計測。 |
| E | **長入力タスクでは KV cache PD 分離が rollout throughput を上げる**。prefill (compute bound) と decode (memory bound) を別 GPU で並列化。slime・verl とも SGLang の PD 分離を利用可能なので、両者で同条件比較できる。 | verl: `sglang_pd_replica.py`。slime: `--prefill-num-servers` ほか | 入力長 512/2048/4096 で rollout throughput を計測。PD 分離 on/off を slime・verl 双方で比較。KV 転送オーバーヘッドのクロスオーバー点も特定。 |
| F | **訓練中の負荷変動に対し verl は elastic scaling で rollout を動的増減できる**。slime は静的構成。ただし replica 追加時の process group 再構築コストが問題になりうる。 | verl: `CheckpointEngineManager.add/remove_replicas()`。slime: 該当なし | 訓練中に rollout replica を 1→2→4 と追加し、追加完了までの待ち時間 (process group 再構築 + weight sync 1 回) と throughput 改善を計測。 |

これらの仮説は、いずれも「slime / verl のどちらかにしかない実装」に由来します。逆に言えば、**実装の有無を地図にしておけば、性能差が出る場所はかなり絞り込める**ということです。本記事の 4 図と対比表は、その地図にあたります。

### 現時点の暫定的な使い分け (構造ベース・未検証)

以下は実測前の、あくまで「実装構造から言える」目安です (数値検証は次記事)。

- **大規模 disaggregated + 頻繁な weight sync**: rollout ノードが多いほど NCCL broadcast の root 帯域がボトルネックになりうるため、ring 転送や Mooncake を選べる **verl** に分があると予想されます。
- **step 間の重み変化が小さい (継続的な RL fine-tuning)**: 差分のみ送る delta sync を本番実装している **slime** が有利と予想されます。
- **colocated の単一ノード構成**: weight sync がノード内で閉じるため両者の差は小さく、機能要件 (delta が欲しいか、Ascend/Mooncake が要るか、elastic scaling が要るか) で選ぶのが現実的です。

繰り返しになりますが、これらは構造から立てた仮説であり、優劣を断定するものではありません。

## まとめ

本記事では、slime をベースに RL post-training の 1 ステップで発生する通信を実装レベルで読み、verl と対比しました。要点を再掲します。

- **共通の骨格**: 両者とも actor (Megatron/FSDP) と rollout (SGLang/vLLM) を分け、AWS では NCCL/NIXL → aws-ofi-nccl/UCX → libfabric → EFA という同じ下層に到達する (この下層は実行環境が提供し、verl は AWS EFA Dockerfile を同梱・slime は同梱しない)。SGLang の KV cache PD 分離も両者がサポートする。
- **weight sync の抽象化**: slime は用途別の独立クラスを分岐で選ぶ (registry なし)、verl は `CheckpointEngineRegistry` の pluggable backend (NCCL/NIXL/Mooncake/Kimi/HCCL)。
- **slime にあって verl の main にないもの**: 汎用の sparse delta weight sync、NIXL を rollout データ転送に使う経路、advantage のグローバル分散正規化。
- **verl にあって slime にないもの**: weight sync の差し替え式抽象、NIXL/Mooncake を weight sync 本体に使う経路、elastic scaling。

これらの構造差から、weight sync のトポロジー (broadcast vs ring)、delta 転送の有無、advantage 正規化の通信コストという形で、性能差が出そうなポイントを 6 つ立てました。**次のステップは、これらを実際に B300 環境で計測して仮説を検証すること**です。実測結果は別記事で報告する予定です。
