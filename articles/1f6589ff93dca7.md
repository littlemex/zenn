## はじめに

RL post-training のフレームワークを性能の観点で比較しようとすると、必ず「weight sync は CUDA IPC か NCCL か」「rollout データはどう運ばれるか」「advantage の正規化はノード間通信を伴うのか」といった通信レイヤの話に行き着きます。

本記事は、slime と verl という二つのフレームワークの通信プリミティブの違いについて整理することが目的です。今後性能検証を実施するため性能で差が出そうなポイントをあらかじめ実装から仮説を立てておきたい意図があります。

記述はすべて各プロジェクトの GitHub ソース (2026 年 6 月時点 main ブランチ) を一次情報とし、確証が取れていない部分は「未確認」と明示します。

## なぜ RL post-training で通信が重要になるのか

RL post-training のループは、大きく 3 つの計算フェーズに分かれます。まずそれぞれが何をするものかを整理します。

| フェーズ | 担当 | 何をするか | 出力 |
|---------|------|-----------|------|
| **① rollout** | 推論エンジン (vLLM/SGLang) | いまのモデル (ポリシー) にプロンプトを与えて応答を生成。 | sequences/log-prob |
| **② reward + advantage** | フレームワーク本体 | 応答を「どれくらい良かったか」スコア付けし、「平均と比べてどれくらい良い/悪い行動か」を表す **advantage** を計算。 | advantage |
| **③ training** | 訓練バックエンド (Megatron/FSDP) | advantage を使ってポリシー勾配を計算し、モデルの重みを更新。大規模モデルを TP/PP/DP で分割しながら backward と optimizer step を回す | updated weight |

この 3 つが「生成 → 評価 → 学習 → (更新後重みで) また生成」と 1 ステップずつ回るのが RL post-training のループです。

ここで教師ありの事前学習と決定的に異なるのが、**rollout を担う推論エンジンと training を担う訓練バックエンドが、別の重みコピーを持つ**点です。事前学習なら更新するモデルは 1 つですが、RL では「生成に使うモデル」と「勾配で更新されるモデル」が物理的に別インスタンス (別 GPU・別ノード) になります。そのため training フェーズで重みを更新したら、その新しい重みを rollout 側のエンジンへ**配り直さないと、次の生成が古いポリシーのまま**になってしまいます。この「更新後の重みを training 側から rollout 側へ配る」処理を **weight sync (重み同期)** と呼び、モデルが大きいほど転送量が膨らむため通信のホットスポットになります。

そして 1 ステップの中では、この weight sync 以外にも複数の通信が**別系統で**走ります。具体的には、(a) rollout が生成したデータを training 側へ運ぶ **rollout データの移動**、(b) advantage 計算の際にデータ並列の各ランク間で統計を揃える **advantage 正規化**、(c) training の backward で勾配をデータ並列ランク間で集約する **gradient 同期**、です。

本記事で繰り返し出てくる用語を先に押さえておきます。

- **colocated (コロケート)**: training と rollout を同じ GPU セット上に同居させる構成。重みの受け渡しがノード内 (NVLink / CUDA IPC) で閉じる。
- **disaggregated (分離)**: training クラスタと rollout クラスタを別の物理ノードに分ける構成。weight sync がノード間 (EFA) を流れる。

## RL 特有の通信と、そうでない通信

RL post-training の 1 ステップで走る通信は、「RL post-training に特有の通信」と「事前学習 (Megatron / FSDP) で使う通信」の 2 種類に分けると見通しが良くなります。slime と verl で実装が割れるのは前者だけで、後者は [Megatron-Core](https://github.com/NVIDIA/Megatron-LM?tab=readme-ov-file#setup) / NCCL がそのまま提供するため両者でほぼ共通です。

まず slime の 1 ステップ (後述の図 2 参照) を、RL 特有 or Not で表にまとめます。`source → dest` は「どのプロセスからどのプロセスへ流れるか」、プリミティブは実際に使う通信ライブラリです。

| 通信 | source → dest | プリミティブ (slime) | 分類 |
|------|--------------|---------------------|------|
| rollout データ収集 | rollout → train | Ray object store (既定) / NIXL (任意) | **RL 特有** |
| advantage 正規化 | train DP group 内 | NCCL AllReduce | **RL 特有** |
| weight sync | train → rollout | NCCL broadcast / disk / delta | **RL 特有** |
| DP gradient 同期 | train DP group 内 | NCCL AllReduce / reduce-scatter | 事前学習と共通|
| TP | train TP group 内 | NCCL (NVLink 中心、TP がドメイン超なら EFA) | 事前学習と共通 |
| PP | train PP group 内 | NCCL P2P send/recv (ノード間なら EFA) | 事前学習と共通 |
| MoE expert all-to-all | train EP group 内 | AWS EFA では UCCL-EP | 事前学習と共通 |

ポイントは 2 つです。

- **RL 特有の通信** (rollout データ収集・advantage 正規化・weight sync) は、推論エンジンと訓練バックエンドが別の重みコピーを持つ RL ならではのもので、**slime と verl の実装差が集中する**場所です。本記事が主に整理するのはここです。
- **事前学習と共通の通信** (gradient 同期・TP/PP・MoE all-to-all) は Megatron-Core / NCCL がそのまま担い、両フレームワークで本質的な差はありません。

この分類を前提として、まず slime をスタック (図1) とフロー (図2) で読み、続いて verl (図3・図4) を対比します。図1・図2 が slime、図3・図4 が verl です。

## slime の通信スタック (図1)

まず slime が実際に使う通信プリミティブを、スタック階層で並べたのが図 1 です。

![slime が使う通信プリミティブのスタック階層](/images/1f6589ff93dca7-slime-comm-stack.png)

図の読み方は色と形で決まっています。**色付きの実線ボックスが slime が実際に使うもの**、**グレーの破線ボックスは「要素技術としてどこに位置するかは示すが、AWS EFA では使わないもの」** です。slime の通信は、以下に集約されます。

### L3

- **NCCL**: gradient の AllReduce と、disaggregated 構成での weight sync broadcast に使います。AWS では `aws-ofi-nccl` が NCCL のトランスポートとして libfabric (EFA provider) をぶら下げ、最終的に EFA NIC に到達します。
- **Gloo**: 重みなど **GPU 上の大きいテンソルの集団通信は速い NCCL が担い**、Gloo はその裏方として **CPU 側の軽い処理** (各フェーズの足並みを揃える barrier や、小さなメタデータの受け渡し) を担当します。「NCCL = GPU の本データ、Gloo = CPU 側の制御」と役割分担していると捉えてください (Gloo は NCCL が無い CPU 環境向けの汎用バックエンドでもありますが、本記事の GPU 前提ではこの裏方役です)。
- **Ray object store**: rollout で生成したデータ (sequences / log-prob) の収集します。
- **NIXL**: SGLang 経由で rollout データ転送を最適化する任意経路です。
- **UCCL-EP** : MoE モデルを扱う場合、expert の all-to-all 通信に使われます。UCCL 関連については別の記事で扱うのでこういうものがあるくらいの認識で大丈夫です。

### L2 — Transport / proxy・backend

L3 の collective ライブラリ (NCCL など) は、自分で直接 NIC を叩くわけではありません。「どのトランスポートに載せるか」を担うのが L2 です。AWS EFA でこの記事に関わるのは次の 2 つです。

- **`aws-ofi-nccl`**: NCCL の **NET プラグイン**です。NCCL は「ネットワークをこう抽象化する」というプラグイン API を持っていて、`aws-ofi-nccl` はその API を実装し、**NCCL の送受信を libfabric の呼び出しに変換 (マップ) する**アダプタとして動きます ([README](https://github.com/aws/aws-ofi-nccl) に "implements a plug-in which maps NCCLs ... transport APIs to libfabric's ... interface" と明記)。これがないと NCCL は EFA を使えず、図で NCCL から下に伸びる経路はまずここを通ります。
- **`libfabric` (+ EFA provider)**: ベンダ非依存の通信 API (OFI = OpenFabrics Interfaces) を提供するライブラリです。実際の NIC ごとの実装は「provider」というプラグインで差し替えられ、EFA 用の実装が **EFA provider** ([`ofiwg/libfabric` の `prov/efa/`](https://github.com/ofiwg/libfabric/tree/main/prov/efa)) です。`aws-ofi-nccl` から見た libfabric は「OFI API を叩けば、あとは EFA provider が EFA のお作法 (SRD など) に翻訳して NIC まで届けてくれる」共通の窓口にあたります。libfabric と SRD の詳細は前提記事「[AWS EFA と Nitro System の仕組みを理解する](https://zenn.dev/tosshi/articles/0eeb53ca63f8b2)」で解説済みなので、ここでは「NCCL → `aws-ofi-nccl` → libfabric (EFA provider)」という縦のつながりだけ押さえてください。

なお図の L2 には `UCX (NIXL SRD backend)` も置いていますが、UCX は vLLM/SGLang まわりでこれまでの記事でも繰り返し触れてきたため、本記事では割愛します (NIXL の RDMA バックエンドとして同じ EFA の下層に到達する、という位置づけだけ図に残しています)。

### L1 — Kernel driver / GPU memory direct

L2 の libfabric (EFA provider) は、最終的に **カーネル空間のドライバ**を通して NIC のハードウェアに触れます。その本体が **`efa.ko`** です。

- **`efa.ko` (EFA driver)**: AWS の Elastic Fabric Adapter 用の **Linux カーネルモジュール**です。役割は「EFA という NIC を OS に認識させ、RDMA (InfiniBand verbs) のインターフェースとしてユーザー空間に見せる」こと。具体的には、デバイスの初期化・認識 (probe)、**Queue Pair (QP) や Memory Region (MR) といった RDMA 資源の生成・管理**、ファームウェアとのやり取りを担います。重要なのは、`efa.ko` 自身がデータ 1 バイトごとを運ぶわけではない点です。一度 QP と MR をセットアップしてしまえば、以降の実データ送受信はユーザー空間から NIC を直接叩く (**カーネルバイパス**) ため、`efa.ko` は「RDMA を使える状態をお膳立てする管理役」と捉えると分かりやすいです。
  - **実装の所在 (証拠)**: 2 系統あります。(1) Linux カーネル本体に upstream 済みの [`drivers/infiniband/hw/efa/`](https://github.com/torvalds/linux/tree/master/drivers/infiniband/hw/efa) (`efa_main.c` = デバイス probe、`efa_verbs.c` = QP/MR などの verbs 実装)。(2) AWS が配布する [`amzn/amzn-drivers` の `kernel/linux/efa/`](https://github.com/amzn/amzn-drivers/tree/master/kernel/linux/efa) で、こちらは GPUDirect RDMA 対応 (`efa_p2p.c` / `efa_nvmem_v2.c`) や Neuron 対応 (`efa_neuronmem.c`) を含む拡張版です。README に EFA は "a network device that provides reliable userspace communication and kernel bypass capabilities" とあり、ビルド成果物が `efa.ko`、ユーザー空間からは rdma-core (libibverbs) 経由で使う、と書かれています。
- ユーザー空間からこの `efa.ko` を SRD で叩く入口が、図 L2 の **`libibverbs + efadv SRD QP (efadv_create_qp_ex)`** です。`efadv` は rdma-core に含まれる **EFA 専用の verbs 拡張 (EFA direct verbs)** で、[`providers/efa/efadv.h`](https://github.com/linux-rdma/rdma-core/blob/master/providers/efa/efadv.h) に `efadv_create_qp_ex` が宣言されています。これが `EFADV_QP_DRIVER_TYPE_SRD` を指定して **SRD (Scalable Reliable Datagram) の QP** を作る関数で、libfabric の EFA provider はこの API を通じて `efa.ko` に QP 生成を依頼します。
- 同じ L1 にある **`GDRCopy` / `GPUDirect RDMA (nv_peer_mem)`** は、NIC が GPU の VRAM を直接読み書きできるようにする仕組みで、こちらは前提記事「[GPUDirect RDMA 入門](https://zenn.dev/tosshi/articles/42f0ee03b328a4)」で解説済みです。

(L0 はハードウェア層 — EFA NIC・GPU・通常 NIC (eth0) — なので説明は省略します。)

## slime の RL 1 ステップ (図2)

図1 が「何が使えるか」の地図なら、図2 は「1 ステップで、何が、どこを流れるか」の時系列です。

![slime の GRPO 1 イテレーションの通信フェーズ](/images/1f6589ff93dca7-slime-rl-comm-flow.png)

この図の肝は、**ノード間 EFA を流れる通信を、黄色帯 (中央) の中にオレンジ箱として置いた**ことです。オレンジ箱が「実際に EFA を通るデータ」、灰箱が「ノード内 (NVLink・IPC) で完結する通信」を表します。形と位置だけで「どこに帯域がかかるか」が分かるようにしてあります。番号順に追います。

1. **① 応答生成**: rollout engine (SGLang) が sequences / tokens を生成します。KV cache は engine 内で完結し、ノード間転送はありません (灰箱)。
2. **② データ収集**: 生成された sequences と log-prob を training 側へ運びます。デフォルトは **Ray object store** (`ray.put` / `ray.get`)、オプションで **NIXL+UCX** の RDMA 経路です (`slime/ray/rollout.py` の `_split_train_data_by_dp`)。disaggregated 構成ではこれがノード間を流れるのでオレンジ箱です。**NCCL はこのフェーズに一切関与しません。**
3. **③ reward 計算 + advantage 正規化**: slime は `--normalize-advantages` 指定時に advantage を DP group 全体で正規化し、`distributed_masked_whiten` が **NCCL AllReduce** をかけます (`slime/utils/distributed_utils.py`、グループは `mpu.get_data_parallel_group()`)。DP group がノード間に分散していれば、この AllReduce も EFA を通ります (図2 では簡潔さのため ③ の EFA 箱は省略していますが、後述の図4 verl との対比ではこの ③ が分岐点になります)。reward 自体はルールベースならローカル、外部 reward service なら HTTP です。
4. **④ Train 内部分散**: ここは並列化軸ごとに経路が割れます。**TP** は per-layer に AllReduce を繰り返し通信量が大きいため、原則 TP group を NVLink ドメイン内に収めてノード内で閉じます (灰箱。ただし TP サイズが単一ノードの GPU 数を超えればノード間 EFA に出ます)。一方 **PP** は stage 境界の activation を P2P send/recv で渡すだけで通信量が小さく、**ノード境界をまたいで配置するのがむしろ定石**で、その場合はノード間 EFA を流れます (オレンジ箱)。そして **DP の gradient 同期 (NCCL AllReduce / reduce-scatter) はノード間** を流れます (オレンジ箱)。同じ「内部分散」でも軸ごとに経路が割れる点を、点線枠の中で対比させています。
5. **⑤ Weight Sync**: training で更新した重みを rollout に配ります。これが weight sync で、転送方式は構成で選択します (後述)。

ここで重要なのは、**gradient 同期 (④ の DP) と weight sync (⑤) は別系統**だということです。前者は train クラスタ内 DP の勾配集約、後者は train→rollout の重み配布で、コードパスも通信グループも完全に分離しています (`slime/backends/megatron_utils/model.py` の DDP hook と `actor.py` の `weight_updater.update_weights()` は別の呼び出しサイト)。図でも ④ と ⑤ を別の箱・別の矢印にしているのはこのためです。

### slime の weight sync は複数方式

slime の weight sync (⑤) は、registry のような共通抽象を持たず、`slime/backends/megatron_utils/actor.py` の手書き分岐 (`update_weight_mode` (full/delta) × `update_weight_transport` (nccl/disk) × colocate) で実装クラスが選ばれます。クラスは以下の 4 つです (`UpdateWeightFromDistributedDelta` は `UpdateWeightFromDistributed` を継承し、`UpdateWeightFromTensor` も `UpdateWeightFromDistributed` の関数を再利用するため、完全独立ではありません)。

| 方式 | クラス | プリミティブ | ノード間/内 |
|------|--------|------------|------------|
| colocated | `UpdateWeightFromTensor` | Gloo gather + Ray IPC (CUDA IPC handle) | ノード内 (EFA 不使用) |
| disaggregated full / nccl | `UpdateWeightFromDistributed` | NCCL broadcast (`slime-pp_{pp_rank}` group) | ノード間 (EFA) |
| disaggregated full / disk | `UpdateWeightFromDisk` | safetensors を共有 FS 経由 | 共有 FS |
| disaggregated delta | `UpdateWeightFromDistributedDelta` | 差分のみ NCCL broadcast / disk (safetensors) | ノード間 (EFA) / 共有 FS |

とくに **delta sync は slime の特徴的な実装**です。RL fine-tuning では step 間で BF16 の大半のバイトが変化しないため、前回との差分だけを broadcast することで転送量を大きく削れます。汎用 delta weight sync が main に正式実装されている点が、後の対比の一つの軸になります。

## verl の通信スタック (図3) — slime との差は「抽象化」

同じ視点で verl のスタックを描いたのが図3 です。slime の図1 と見比べてください。

![verl が使う通信プリミティブのスタック階層](/images/1f6589ff93dca7-verl-comm-stack.png)

L0〜L2 (EFA NIC / efa.ko / libfabric / aws-ofi-nccl / GPUDirect RDMA) は、AWS 上で動かす以上、最終的に EFA に届く経路として共通の前提層になります。ただしこの層はフレームワークのコードではなく実行環境が提供するもので、リポジトリ成果物としては非対称です。verl は AWS EFA 向けの Dockerfile (`docker/aws/Dockerfile.extention.awsefa`、aws-efa-installer + aws-ofi-nccl + libfabric + `FI_PROVIDER=efa`) を同梱しますが、slime はこの層を同梱せず実行環境任せです。**フレームワークのコードとして見たときの違いは L3 の weight sync の抽象化**にあります。

slime が weight sync を「用途別のクラスを手書き分岐で選ぶ」のに対し、verl は **`CheckpointEngineRegistry` という pluggable な registry** を持ちます (`verl/checkpoint_engine/base.py`)。`@CheckpointEngineRegistry.register(backend)` デコレータで以下が登録され、config で差し替えられます。

| backend キー | クラス | 下層トランスポート |
|------------|--------|------------------|
| `naive` | `ColocatedCheckpointEngine` | 同一 GPU 上で weight generator を in-memory 受け渡し (torch.distributed / all_gather) |
| `nccl` | `NCCLCheckpointEngine` | NCCL broadcast + ZMQ PUB/SUB (メタデータ) |
| `nixl` | `NIXLCheckpointEngine` | UCX + IB RDMA の ring 転送 |
| `mooncake` | `MooncakeCheckpointEngine` | Mooncake TransferEngine (RDMA P2P chain) |
| `kimi_ckpt_engine` | `KIMICheckpointEngine` | Mooncake P2P + NCCL broadcast |
| (`nccl` キー上書き) | `HCCLCheckpointEngine` | HCCL (Ascend NPU) |

図3 では、verl が使うこれらのバックエンドを角丸実線で並べ、**verl が使わない / slime 側にあるもの** (Gloo 専用 barrier、NIXL rollout transport) を右端に六角形破線で置いています。ポイントを整理すると、

- **verl だと、weight sync が「差し替え式」**。NCCL broadcast だけでなく NIXL ring や Mooncake P2P chain を config で選べます。slime は full sync が NCCL broadcast 一択です (registry を持たず、用途別クラスを分岐で選ぶ)。
- **verl だと、NIXL は weight sync 用**。slime では NIXL は rollout データ転送 (図2 の②) に使われるのに対し、verl の `NIXLCheckpointEngine` は weight sync の ring 転送 (`trainer→r1→r2→…→rN`) に使われます。**同じ NIXL でも用途が逆**である点は実装を読まないと取り違えます。
- **verl だと、汎用の sparse delta sync が main にない**。任意パラメータの位置・値のみを送る方式は Draft PR #6794 が提案中で未マージです (slime は同方式を main に正式実装)。ただし verl main にも LoRA adapter の delta 同期 (base 重みを GPU に残して adapter 差分のみ同期する別種の delta) は存在します。
- **MoE の expert all-to-all (UCCL-EP) は slime と対称**。verl も large MoE 学習で DeepEP API を使い、Megatron-Core の flex dispatcher 経由で有効化します (公式サンプル `run_deepseek_v3_671b_megatron.sh` 等が `moe_enable_deepep=True` / `moe_token_dispatcher_type=flex` を指定)。AWS EFA 環境では slime と同様に `deep_ep` の解決先を UCCL-EP に差し替えて動かします。slime が `import deep_ep` の monkey-patch で到達するのに対し、**verl は config 経由で間接的に到達する**点だけが違いで、verl 本体に `import deep_ep` はありません。図3 でも UCCL-EP の縦経路 (NVSHMEM/IBGDA はグレー) を図1 と同形に描いています。

## verl の RL 1 ステップ (図4) — advantage と KV cache が分岐点

verl の 1 ステップを、図2 と同じレイアウトで描いたのが図4 です。

![verl の GRPO/PPO 1 イテレーションの通信フェーズ](/images/1f6589ff93dca7-verl-rl-flow.png)

slime の図2 と「箱の色」を見比べると、verl の構造差が一目で分かります。

- **③ advantage 正規化が灰箱 (ローカル CPU のみ)**。slime は `--normalize-advantages` 時に `distributed_masked_whiten` で NCCL AllReduce をかけてグローバル統計で正規化しますが、**verl だと `masked_whiten` がローカル統計のみで完結し、distributed collective を使いません** (`verl/trainer/ppo/core_algos.py`、`verl/utils/torch_functional.py`)。だから図4 では ③ が EFA 帯に出ません。これは「同じ GRPO でも advantage の正規化スコープが違う」という設計差で、学習の安定性にも効きうる重要な分岐です。
- **①b KV cache PD 分離**。verl は SGLang の prefill-decode 分離をサポートし、prefill が生成した KV cache を `disaggregation_transfer_backend` (NIXL / Mooncake) で decode 側に転送します (`verl/workers/rollout/sglang_rollout/sglang_pd_replica.py`)。**slime も同様に PD 分離をサポートします** (`--prefill-num-servers`、専用ドキュメント、mooncake/nixl テスト)。どちらも KV の物理転送は SGLang に委譲しており、PD 分離の有無は両者の差別化点ではありません。図4 では rollout 側に ①b の箱を置き、KV cache 転送をオレンジ箱で帯に出しています。
- **② データ収集は Ray のみ**。verl の rollout データ収集は `DataProto` / `DataProtoFuture` (Ray object store) だけで、**slime のような NIXL オプションはありません**。

④ (gradient 同期) と ⑤ (weight sync) が別系統である点、内部分散の経路が軸ごとに割れる点 (TP は原則ノード内 NVLink、PP と DP はノード間 EFA に出やすい) は slime と共通です。verl では FSDP backend なら gradient 同期が `loss.backward()` 内の reduce-scatter で暗黙に走り、Megatron backend なら DDP の AllReduce で走ります。いずれも weight sync の CheckpointEngine とはコードパスが分離しています。

## 補足: MoE を EFA で訓練する場合の UCCL-EP

ここまでの図は **dense モデルの RL** ならそのまま読めます。dense では expert routing がないため、図1・図3 中央の UCCL-EP の縦経路は登場せず、通信の主役 (weight sync・rollout・gradient) はすべて NCCL / Ray で、AWS EFA 対応は `aws-ofi-nccl` 経由で確立済みです。

**MoE モデル (DeepSeek-V3, Qwen3-MoE など)** を訓練するときだけ、expert の all-to-all (dispatch / combine) が加わります。slime も verl もこれを **DeepEP API (`import deep_ep`)** に委ねますが、本記事が前提とする AWS EFA では本家 DeepEP がそのままでは動きません。これを埋めるのが DeepEP 互換 fork の **UCCL-EP** で、`deep_ep` という import 名を共有する drop-in 置換です。Megatron 経由なら UCCL-EP をインストールするだけでフレームワーク改変ゼロで EFA 上の all-to-all が動きます (AWS 公式の [awslabs/awsome-distributed-ai megatron-bridge](https://github.com/awslabs/awsome-distributed-ai/tree/main/3.test_cases/megatron/megatron-bridge) が p6-b300 + EFA で実証)。

:::message
本家 DeepEP が EFA で動かない理由 (低レイテンシ経路が InfiniBand 前提のバックエンドに依存) や UCCL-EP の EFA 経路 (GPU kernel → CPU proxy → libibverbs/efadv) の詳細は本記事の主題から外れるため、別記事で改めて整理します。ここでは「**MoE では expert all-to-all が加わり、EFA では本家 DeepEP でなく UCCL-EP を使う**」という点だけ押さえれば十分です。
:::

## 性能で差が出そうなポイント

ここまでの実装差を踏まえると、「slime には xx が無いから原理的にこの処理は遅い / 速いはず」という形の仮説がいくつか立ちます。実測前の予想として、根拠 (実装) と検証方法をセットで表にまとめます。これらは今後の実験で検証していきたいポイントです。

| # | 仮説 | 実装上の根拠 | 検証方法 |
|---|------|------------|---------|
| A | **大規模 disagg では verl の NIXL ring weight sync が slime の NCCL broadcast より速い**。slime は full sync が NCCL broadcast 一択で、root ノードの NIC 帯域が O(N) ファンアウトのボトルネックになる。verl の `NIXLCheckpointEngine` は ring (trainer→r1→r2→…) で各ホップ O(1) 帯域。 | slime: `UpdateWeightFromDistributed` (broadcast)。verl: `nixl_checkpoint_engine.py` (ring one-sided read) | rollout ノード数を 4/8/16 と変え、NCCL backend vs NIXL backend で weight sync レイテンシ計測。NIXL ring がノード数にほぼ線形、NCCL broadcast が degradation する想定。 |
| B | **slime の汎用 delta sync が full sync を大幅に上回る (最大 ~100x ペイロード削減)**。step 間で BF16 の >99% が不変。verl main には同方式 (任意パラメータの sparse delta) がなく Draft PR #6794 のみ。 | slime: `UpdateWeightFromDistributedDelta`。verl: Draft PR #6794 (LoRA adapter delta は別途存在) | slime で `update_weight_mode=full` vs `delta` を切り替え、転送バイト数と所要時間を計測。差分計算とディスク I/O のオーバーヘッドも併せて計測。 |
| C | **verl の advantage 正規化はローカルで通信ゼロ、slime は (`--normalize-advantages` 時) DP size に比例した AllReduce コストが増える**。一方 slime はグローバル統計で正規化するため、大規模 DP では学習安定性が異なる可能性。 | slime: `distributed_masked_whiten` (NCCL AllReduce)。verl: `masked_whiten` (ローカルのみ) | DP size を 4/8/16/32 と変え、advantage フェーズの通信時間を nsys で計測 (verl は定数、slime は比例の想定)。同時に reward カーブを比較し収束への影響を見る。 |
| D | **colocated の weight sync は slime (Gloo gather + Ray IPC) と verl (in-memory generator 受け渡し) で経路が異なる**。slime は CPU バッファ経由の gather を挟むのに対し、verl の `naive` は同一 GPU 上で generator をそのまま渡す。 | slime: `UpdateWeightFromTensor` (Gloo gather + Ray IPC)。verl: `ColocatedCheckpointEngine` (in-memory) | 同一ノード colocated 構成でモデルサイズ別 (7B/30B/70B) に weight sync レイテンシと CPU 使用率を計測。 |
| E | **長入力タスクでは KV cache PD 分離が rollout throughput を上げる**。prefill (compute bound) と decode (memory bound) を別 GPU で並列化。slime・verl とも SGLang の PD 分離を利用可能なので、両者で同条件比較できる。 | verl: `sglang_pd_replica.py`。slime: `--prefill-num-servers` ほか | 入力長 512/2048/4096 で rollout throughput を計測。PD 分離 on/off を slime・verl 双方で比較。KV 転送オーバーヘッドのクロスオーバー点も特定。 |
| F | **訓練中の負荷変動に対し verl は elastic scaling で rollout を動的増減できる**。slime は静的構成。ただし replica 追加時の process group 再構築コストが問題になりうる。 | verl: `CheckpointEngineManager.add/remove_replicas()`。slime: 該当なし | 訓練中に rollout replica を 1→2→4 と追加し、追加完了までの待ち時間 (process group 再構築 + weight sync 1 回) と throughput 改善を計測。 |
| G | **MoE 訓練では UCCL-EP の expert all-to-all が NCCL alltoall より速い (AWS 報告 ~36%)**。slime・verl とも EFA 上では `deep_ep` を UCCL-EP に差し替えるだけで効く (フレームワーク非依存)。CPU proxy + SRD のオーバーヘッドが dispatch/combine の頻度でどう効くかが論点。 | 両者とも Megatron flex dispatcher (`moe_token_dispatcher_type=flex`) に委譲、実体は UCCL-EP (`deep_ep` import 解決先)。実証は awslabs megatron-bridge | DeepSeek-V3/Qwen3-MoE クラスで `moe_flex_dispatcher_backend` を NCCL alltoall vs UCCL deepep で切替え、step 時間と all-to-all 区間を nsys で計測。slime・verl 双方で同条件比較。 |

これらの仮説は、いずれも「slime / verl のどちらかにしかない実装」に由来します。逆に言えば、**実装の有無を地図にしておけば、性能差が出る場所はかなり絞り込める**ということです。本記事の 4 図と対比表は、その地図にあたります。

### 現時点の暫定的な使い分け (構造ベース・未検証)

以下は実測前の、あくまで「実装構造から言える」目安です (数値検証は次記事)。

- **大規模 disaggregated + 頻繁な weight sync**: rollout ノードが多いほど NCCL broadcast の root 帯域がボトルネックになりうるため、ring 転送や Mooncake を選べる **verl** に分があると予想されます。
- **step 間の重み変化が小さい (継続的な RL fine-tuning)**: 差分のみ送る delta sync を本番実装している **slime** が有利と予想されます。
- **colocated の単一ノード構成**: weight sync がノード内で閉じるため両者の差は小さく、機能要件 (delta が欲しいか、Ascend/Mooncake が要るか、elastic scaling が要るか) で選ぶのが現実的です。

繰り返しになりますが、これらは構造から立てた仮説であり、優劣を断定するものではありません。

## まとめ

本記事では、slime をベースに RL post-training の 1 ステップで発生する通信を実装レベルで読み、verl と対比しました。要点を再掲します。

- **共通の骨格**: 両者とも actor (Megatron/FSDP) と rollout (SGLang/vLLM) を分け、AWS では NCCL → aws-ofi-nccl/libfabric → EFA、MoE の expert all-to-all は UCCL-EP → libibverbs/efadv (SRD) → EFA という下層に到達する (この下層は実行環境が提供し、verl は AWS EFA Dockerfile を同梱・slime は同梱しない)。SGLang の KV cache PD 分離、MoE の expert all-to-all (EFA では UCCL-EP) も両者がサポートする。Ray object store と Gloo は EFA でなく TCP (eth0) を通る点も共通。
- **weight sync の抽象化**: slime は用途別の独立クラスを分岐で選ぶ (registry なし)、verl は `CheckpointEngineRegistry` の pluggable backend (NCCL/NIXL/Mooncake/Kimi/HCCL)。
- **slime にあって verl の main にないもの**: 汎用の sparse delta weight sync、NIXL を rollout データ転送に使う経路、advantage のグローバル分散正規化。
- **verl にあって slime にないもの**: weight sync の差し替え式抽象、NIXL/Mooncake を weight sync 本体に使う経路、elastic scaling。

これらの構造差から、weight sync のトポロジー (broadcast vs ring)、delta 転送の有無、advantage 正規化の通信コストという形で、性能差が出そうなポイントを 6 つ立てました。**次のステップは、これらを実際に B300 環境で計測して仮説を検証すること**です。実測結果は別記事で報告する予定です。

| train ノード内: advantage 正規化はローカルのみ (矢印が EFA 帯に出ない) | [`verl/trainer/ppo/core_algos.py`](https://github.com/volcengine/verl/blob/main/verl/trainer/ppo/core_algos.py) の `compute_gae_advantage_return`、[`verl/utils/torch_functional.py`](https://github.com/volcengine/verl/blob/main/verl/utils/torch_functional.py) の `masked_whiten` がローカル統計のみで完結し distributed collective を使わない。*(slime 差分: slime は NCCL AllReduce でグローバル正規化)* |
| train → rollout: weight sync (registry で差し替え) | [`verl/checkpoint_engine/base.py`](https://github.com/volcengine/verl/blob/main/verl/checkpoint_engine/base.py) の `@CheckpointEngineRegistry.register(backend)` デコレータ + dict registry で backend を config 選択。実装は [`verl/checkpoint_engine/`](https://github.com/volcengine/verl/tree/main/verl/checkpoint_engine) の `{nccl,nixl,mooncake,kimi,hccl}_checkpoint_engine.py` (HCCL は `nccl` キーで Ascend 向けに上書き)。*(slime 差分: slime は registry なしの手書き分岐)* |
| train → rollout (colocated): in-memory generator 受け渡し | [`verl/checkpoint_engine/base.py`](https://github.com/volcengine/verl/blob/main/verl/checkpoint_engine/base.py) の `ColocatedCheckpointEngine` (`send_weights` でインスタンス保持、`receive_weights` で `yield from`)。torch.distributed / all_gather 分類。*(slime 差分: slime は Gloo gather + Ray IPC)* |
| train → r1 → r2 → … → rN: NIXL は weight sync 用 ring 転送 | [`verl/checkpoint_engine/nixl_checkpoint_engine.py`](https://github.com/volcengine/verl/blob/main/verl/checkpoint_engine/nixl_checkpoint_engine.py) の ring (各ホップ one-sided read、backend は UCX/UCCL/Mooncake)。*(slime 差分: slime の NIXL は rollout データ転送用で用途が逆)* |
| train DP group 内: gradient 同期 (weight sync と別系統) | FSDP backend は `loss.backward()` 内の reduce-scatter で暗黙に走り、Megatron backend は DDP の AllReduce。いずれも CheckpointEngine とはコードパス分離 |
| trainer → UCCL-EP → … → EFA NIC: MoE expert all-to-all | 公式サンプル [`examples/grpo_trainer/run_deepseek_v3_671b_megatron.sh`](https://github.com/volcengine/verl/blob/main/examples/grpo_trainer/run_deepseek_v3_671b_megatron.sh) が `moe_enable_deepep=True` / `moe_token_dispatcher_type=flex` を指定。実体は Megatron-Core flex dispatcher に委譲 (verl 本体に `import deep_ep` は無し)。*(slime 差分: slime は `import deep_ep` の monkey-patch、verl は config 経由の間接到達)*。verl の `uccl` は NIXL p2p backend 候補の言及で UCCL-EP (MoE 用) ではない |
| UCCL-EP → GPU kernel → CPU proxy → libibverbs/efadv → EFA NIC: EFA 上の a2a 実体 | slime と同一の段差。本家 DeepEP は mlx5 WQE 直構築で IBGDA (IB/RoCE 専用) 依存、`provider=efa` とは別系統で EFA 不可。EFA では UCCL-EP が GPU kernel → D2H ring → CPU proxy → libibverbs/efadv (SRD) → EFA NIC で実現 (GDRCopy 併用、libfabric 不経由)。AWS 公式実証は [awslabs/awsome-distributed-ai megatron-bridge](https://github.com/awslabs/awsome-distributed-ai/tree/main/3.test_cases/megatron/megatron-bridge) (Dockerfile で UCCL-EP をビルド、`moe_flex_dispatcher_backend=deepep`)。自前 TransferEngine 実装例も ([uccl-project/uccl](https://github.com/uccl-project/uccl)、[deepseek-ai/DeepEP](https://github.com/deepseek-ai/DeepEP)、[Perplexity: trillion-param on EFA](https://research.perplexity.ai/articles/enabling-trillion-parameter-models-on-aws-efa)) |
| rollout 内 (prefill → decode): KV cache PD 分離 | [`verl/workers/rollout/sglang_rollout/sglang_pd_replica.py`](https://github.com/volcengine/verl/blob/main/verl/workers/rollout/sglang_rollout/sglang_pd_replica.py)、[`async_sglang_server.py`](https://github.com/volcengine/verl/blob/main/verl/workers/rollout/sglang_rollout/async_sglang_server.py) の `disaggregation_transfer_backend` (NIXL/Mooncake) に委譲。slime も `--prefill-num-servers` で同様サポート (差別化点ではない) |
| (機能) elastic scaling: rollout replica 動的増減 | [`verl/checkpoint_engine/base.py`](https://github.com/volcengine/verl/blob/main/verl/checkpoint_engine/base.py) の `CheckpointEngineManager.add_replicas()` / `remove_replicas()`。*(slime 差分: slime は該当機能なし)* |
| (主張) 汎用 sparse delta weight sync は未マージ | Draft PR [#6794](https://github.com/volcengine/verl/pull/6794) (2026-06 時点 open)。LoRA adapter delta 同期は main に別途存在。*(slime 差分: slime は汎用 delta を main 実装)* |
