## TL;DR

- 大規模 MoE のデコード推論では、**Attention はメモリ帯域に張り付き、FFN/Expert は演算器を遊ばせます**。計算特性が反対の 2 つを同じ GPU に同居させると、どちらも最適化しきれません。
- **AFD (Attention-FFN Disaggregation)** は、この 2 つを物理的に別々の GPU プールへ分離する推論アーキテクチャです。各プールを別々のバッチ・並列度・ハードウェアで独立に最適化できます。
- 用語と設計は StepFun の **Step-3** 論文が体系化し、先行する ByteDance の **MegaScale-Infer**、Huawei の **xDeepServe** が同じ思想を実装しています。Step-3 は Hopper 上で **4,039 tokens/s/GPU**(DSv3 比 +74%)を報告しています。
- **vLLM でも実装が進行中**です(PR #29772)。`vllm fserver` で FFN を別プロセス起動し、Connector 越しに隠れ状態を送受信する基本フレームワークが入る予定です。

## なぜ Attention と FFN を分けたいのか

自己回帰デコードは 1 トークンずつ生成します。このとき GPU の中で起きていることを Roofline の視点で見ると、Attention と FFN はまったく違う場所で詰まります。

**Attention はメモリバウンドです。** トークンを 1 個生成するたびに、これまでの系列ぶんの KV キャッシュをまるごと読み出します。読み出すバイト数に対して実際の演算(クエリとの内積)はごくわずかなので、演算強度(FLOP/byte)が極端に低くなります。しかも各リクエストの KV キャッシュは別物なので、バッチを増やしても読み出し量がバッチ数ぶん膨らむだけで演算強度は上がりません。つまり **GPU の演算器ではなく HBM 帯域がボトルネック**になります。系列が長くなれば KV キャッシュは線形に膨らみ、今度は HBM 容量を食い潰します。

**FFN は、複数リクエストをまとめてバッチを太らせれば演算バウンドにできます。** 重み行列はバッチ全体で共有されるので、バッチを増やすほど GEMM の演算強度が上がり、演算器を埋められます。ところが **MoE の疎活性化がこれを壊します**。各トークンは多数の Expert のうち上位 k 個しか使わないため、1 個の Expert に届くトークンが薄くなります。MegaScale-Infer の例では、Mixtral 8×22B(top-2 / 8 Expert)をバッチ 156 で回しても、各 Expert に届くのは平均 `156 × 2 / 8 = 39` トークンに過ぎません。GEMM が小さいままなので、せっかくの演算器が遊んでしまいます。

**Attention は「メモリ帯域と容量」が欲しく、FFN は「演算性能と大きなバッチ」が欲しい**のです。性格の違う 2 つを 1 台の GPU に押し込めると、デコード効率を上げようとバッチを増やすほど KV キャッシュが HBM を圧迫し、メモリ特化と演算特化という相反する要求を 1 台で満たそうとして、結局どちらも中途半端になります。

AFD のアイデアはシンプルです ── **同居をやめて、別々のマシンに引っ越す**。Attention はメモリ帯域・KV 容量に特化した GPU プールへ、FFN/Expert は演算性能に特化した GPU プールへ振り分け、隠れ状態テンソルだけをネットワーク越しに受け渡します。こうすれば各プールを独立に最適化でき、複数の Attention プールからトークンを集約して FFN 側の GEMM を太らせ、演算器を埋めることもできます。

> Disaggregation という言葉自体は、Prefill と Decode を分ける **PD 分離**で既に知られています。AFD はこれとは独立に適用できる別軸の最適化で、**1 つの層の中の Attention と FFN を分ける**ものです。両者は組み合わせて使えます(実際に後述の vLLM では PD 分離との統合がロードマップに入っています)。

## ping-pong パイプラインで通信を隠す

分離の素朴な懸念は「ノード間でテンソルを往復させたら通信コストで損するのでは?」というものです。AFD はこれを **マイクロバッチのパイプライン化**で隠します。MegaScale-Infer はこれを *ping-pong pipeline parallelism* と呼びました。

バッチを複数のマイクロバッチに分割し、Attention プールと FFN プールの間を交互にシャトルさせます。マイクロバッチ #0 が通信・FFN へ進んでいる間に、Attention プールは次のマイクロバッチ #1 を処理します。両プールが常に動き続け、転送レイテンシが計算の裏に隠れます。

![ping-pong パイプライン](https://arxiv.org/html/2504.02263v4/x4.png)

これが効くにはいくつか条件があります。Attention と FFN の計算時間がだいたい釣り合っていること、通信時間が計算時間より短いこと、そしてパイプラインを埋めるのに十分なマイクロバッチ数があることです。

## 3 つの論文が示した同じ方向

AFD は単一の論文の発明ではなく、複数チームが近い時期に独立に到達した設計です。整理しておきます。

### MegaScale-Infer (ByteDance, arXiv:2504.02263)

https://arxiv.org/html/2504.02263v4

タイトルは *Serving Mixture-of-Experts at Scale with Disaggregated Expert Parallelism* です。Attention と FFN をノードレベルで分離し、ping-pong パイプラインで通信を隠し、**M:N 構成**(Attention M 台、FFN N 台を独立にスケール)を自動探索します。NCCL を置き換える専用の **M2N 通信ライブラリ**(M 台の Attention ノードと N 台の FFN ノードの間の通信に特化)を実装し、GPU↔CPU コピー排除などで NCCL 比 P99 レイテンシ 92.9% 削減を報告しています。

性能は Scaled-MoE(317B)で TensorRT-LLM 比 **1.90 倍**、vLLM 比 7.11 倍の per-GPU スループットです。さらに **異種 GPU 構成**(Attention に高 HBM の H20、FFN に高 TFLOPS の L40S)で vLLM 比 3.24 倍の **コスト正規化**スループットを示し、「ワークロードに合わせて違う GPU を充てる」という AFD ならではの旨味を定量化しています。

### Step-3 (StepFun, arXiv:2507.19427)

https://arxiv.org/html/2507.19427v1

**AFD (Attention-FFN Disaggregation) という呼び名を与え、model-system co-design として整理した論文**です(分離アイデア自体は MegaScale-Infer が先行しており、Step-3 もそれを先行研究として引用しています)。321B パラメータ(活性 38B/token)の MoE 型 VLM を、推論コストを最初から見据えて設計しています。

2 つの柱があります。1 つは **AFD** そのものです。もう 1 つは **MFA (Multi-matrix Factorization Attention)** で、QK 回路を低ランク分解して KV キャッシュと演算を同時に削減し、Attention 側の演算強度を「弱めの GPU でも回せる」値に調整します。

報告値は Hopper(H800)・FP8・4K コンテキスト・50ms TPOT SLA・32 GPU(2A2F)構成で **4,039 tokens/s/GPU**。同条件で測った DeepSeek-V3(DSv3)の 2,324 tokens/s/GPU に対して約 74% 高い数字です。

### xDeepServe (Huawei Cloud, arXiv:2508.02520)

https://arxiv.org/html/2508.02520v3

Ascend 910(第2世代)/ CloudMatrix384 スーパーポッド(384 NPU)上の本番サービングシステムです。Transformer 推論を **Attention・FFN・MoE のモジュール単位に分解**して別々の NPU プールに割り当てる "Transformerless" 設計を掲げ、PD 分離と Attention-MoE 分離を同時に成立させています。マイクロ秒級の P2P と All-to-All を提供する **XCCL** を独自実装しています。

## vLLM での実装(PR #29772)

この AFD 設計が **vLLM 本体で実装が進んでいます**。設計議論は RFC として Issue #22799 に、最初の実装が PR #29772(*AFD basic implementation*)にあります。

![vLLM の AFD 構成](/images/8480f5be05871e-afd.png)

構成要素は次のとおりです。

- **FFN サーバの独立起動**: `vllm fserver` という CLI コマンドで FFN プロセスを別に立てます。役割は `AFDConfig`(`afd_role` が `attention` か `ffn` か、使う connector、ホスト/ポート等)で切り替えます。
- **Connector による抽象化**: `AFDConnector` が Attention↔FFN 間通信を抽象化します。`base.py` / `p2p_connector.py` / `dummy_connector.py` / `factory.py` の構成で、現状の基本実装は `send_attn_output` / `recv_ffn_output` といった**点対点の送受信**です。Factory への手動登録方式なので、後から **stepmesh のような高性能バックエンドに差し替え可能**です(All-gather 等の集合通信や entry-point プラグイン化は RFC 上の将来設計です)。
- **DBO (Dual-Batch Overlap)**: マイクロバッチを 2 つに割って計算と通信を重ねる、ping-pong の vLLM 版です。`--enable-dbo` と prefill/decode 別の閾値で制御します。
- **配置**: Attention 側は DP レプリカ(`--data-parallel-size 2`)、FFN 側は EP(`--enable-expert-parallel`)です。現状は Attention 台数が FFN 台数の整数倍、かつ両側 TP=1 という制約があります。

動作検証は DeepSeek-V2-Lite を **2A2F**(Attention 2 + FFN 2)で実行しています(これは Step-3 の比較相手だった DeepSeek-V3 とは別モデルです)。初期版は **eager mode 限定**(カスタムカーネル不要で素早く出す方針)で、CUDA Graph 対応・MoE ゲーティング・XBO(DBO を 3 つ以上のマイクロバッチに拡張)・PD 分離との統合・Elastic AFD などはロードマップに並んでいます。
