### 3. Elasticache Serverless の作成

:::message alert
**ElastiCache Serverless は TLS が常に有効**

ElastiCache Serverless は TLS を無効化できません。LMCache の `remote_url` には必ず `rediss://`（s が 2 つ）を使います。`redis://` では接続できません。

```yaml
# NG: TLS なし
remote_url: "redis://endpoint:6379"

# OK: TLS あり（s が 2 つ）
remote_url: "rediss://endpoint:6379"
```

`rediss://` スキームを使うと redis-py が自動的に SSL/TLS を有効化します。証明書検証は自動的に無効化されるため、追加設定は不要です。
:::

:::message
ElastiCache Serverless はサブネットが最低 2 つ必要なため AWS ParallelCluster 向けの CloudFormation スタックでは 1 つしか作成されないのでエラーします。そのためスクリプト内で 1 サブネット追加しました。好ましいやり方ではないかもしれませんが暫定対応です。
:::

## アプリケーション実装ポイント解説

:::message
まだ Elasticache Serverless を使って LMCache 経由で PD 分離推論を動かしてハマり散らかした人は少ないと思うのでハマり散らかしログを残しておきます。
:::

### サーバー構成

1台の g5.12xlarge に Prefill サーバーと Decode サーバーを同居させます。今回の実装はシングルノード前提で GPU が 1 つしかないインスタンスは Prefill しか起動しないので注意してください。

| サーバー | GPU | ポート | TP | kv_role | コンテナ名 |
|---------|-----|--------|-----|---------|-----------|
| Prefill | GPU 0 | 8100 | 2 | kv_producer | vllm-prefill |
| Decode | GPU 1 | 8200 | 2 | kv_consumer | vllm-decode |

### 失敗履歴

数々の失敗の履歴をシェアしておきます。

:::message alert
かなりの時間をかけて AI 君とともに苦労して解決しておりパッチを当ててみたり色々やった挙句、最終的には設定だけで全ての問題を回避できました。

## 症状

```
[LMCache ERROR] Failed during post_init: Remote backend requires
local CPU backend as a buffer. Please turn on local cpu backend
with max_local_cpu_size > 0.
```

## 変更箇所

**原因**: ElastiCache（リモートストレージ）を使う場合でも、ローカル CPU バッファが必須。

```yaml
# NG: local_cpu を無効にしていた
local_cpu: False
max_local_cpu_size: 0
remote_url: "redis://..."

# OK: ローカル CPU バッファを有効化
local_cpu: True
max_local_cpu_size: 5  # GB
remote_url: "redis://..."
```

::::details v03

## 症状

サーバーは正常起動、推論も成功。しかし KV Cache が全く保存されない。

```
[LMCache DEBUG] In connector.start_load_kv, but the attn_metadata is None
```

## 変更箇所

**原因**: `--enforce-eager`（CUDA graph 無効化）モードでは `attn_metadata` が None になり、KV Cache の保存処理がスキップされる。

`--enforce-eager` を抜く。
::::

::::details v04

## 症状

```
ValueError: Free memory on device cuda:0 (13.85/22.05 GiB) is less
than desired GPU memory utilization (0.9, 19.84 GiB).
```

## 変更箇所

**原因**: Prefill が GPU 0-1 を占有した状態で、Decode が同じ GPU を使おうとします。Docker の `--gpus` オプションと `NVIDIA_VISIBLE_DEVICES` の組み合わせでデバイスの分離が必要。

```bash
# Prefill: GPU 0-1
-e NVIDIA_VISIBLE_DEVICES=0,1

# Decode: GPU 2-3
-e NVIDIA_VISIBLE_DEVICES=2,3
```

::::details v05

## 症状

`--enforce-eager` を外しても `attn_metadata is None` 問題が継続した。ソースコードを読み込んだ結果、vLLM 0.19.0 の V1 Engine と LMCache の統合部分にバグを発見。

## 変更箇所（現在は不要）

**修正箇所**: `lmcache/integration/vllm/vllm_v1_adapter.py` の `start_load_kv` メソッド

ちなみに `vllm/distributed/kv_transfer/kv_connector/v1/lmcache_integration/vllm_v1_adapter.py` ここにも類似コードがあり混乱した。こちらは使われていない。

```python
# Before（バグ）:
attn_metadata = forward_context.attn_metadata
if attn_metadata is None:
    logger.debug("attn_metadata is None")
    return  # ← KV Cache 処理が全てスキップされる

# After（修正）:
# V1 Engine の kv_connector_no_forward パスでは attn_metadata は None だが、
# retrieve() 自体は attn_metadata を使用しない。
# requests が存在する限り KV Cache の処理を続行すべき。
if not metadata.requests:
    logger.debug("no requests to process")
    return
```

この修正を Docker コンテナ内の vLLM に直接パッチとして適用していたが v0.19.0 で解消された。

## 症状

SSL 問題は解消し、推論自体は成功し、ローカル CPU バッファへの保存も成功する。しかし、ElastiCache への保存が毎回このエラーで失敗し、ElastiCache のキー数は常に 0 になる。

```
[LMCache WARNING] batched put error: zip() argument 2 is shorter than argument 1
  (instrumented_connector.py:186:lmcache.v1.storage_backend.connector.instrumented_connector)
```

## 変更箇所

### 根本原因の特定

**[`BytesBufferMemoryObj`](https://github.com/LMCache/LMCache/blob/0b4437749906b17df74ff007155db2e920dffa8e/lmcache/v1/memory_management.py#L703) クラスのバグ** が根本原因だった。

```python
# lmcache/v1/memory_management.py
class BytesBufferMemoryObj(MemoryObj):
    def get_shapes(self) -> list[torch.Size]:
        return [torch.Size([self.length, 0, 0, 0])]  # 1要素のリスト

    def get_dtypes(self) -> list[torch.dtype]:
        return []  # 空のリスト ← ここが問題
```

`get_shapes()` は1要素のリストを返すのに、`get_dtypes()` は空のリストを返す。この不整合が `RemoteMetadata._prepare_params()` で `zip(shapes, dtypes, strict=True)` を実行した際に `ValueError` を引き起こす。

### エラーの伝播経路

エラーの伝搬経路から、`remote_serde` が `cachegen` の場合、かつ `RedisConnector` を使用している場合に Prefill ノードが KV Cache を保存すると発生する。

```
CacheEngine.store()
  └─ StorageManager.batched_put()
       └─ RemoteBackend.batched_submit_put_task()
            └─ CacheGenSerializer.serialize()  ← BytesBufferMemoryObj を生成
                 └─ RedisConnector._put()
                      └─ RemoteMetadata._prepare_params()
                           └─ zip(shapes, dtypes, strict=True)
                                └─ ValueError: zip() argument 2 is shorter than argument 1
```

### 変更

`naive` にすることで動作するが圧縮が効かないため、トレードオフとして、KV Cache の転送量は増加する。この変更でようやく ElastiCache に KV Cache が保存されるようになった。

```yaml
# cachegen → naive
remote_serde: "naive"
```

## 変更箇所

### `save_unfull_chunk: True`

最初の設定では、Prefill にのみ `save_unfull_chunk: True` を設定していた。

```yaml
# Prefill (lmcache-prefiller.yaml)
save_unfull_chunk: True  # 設定あり

# Decode (lmcache-decoder.yaml)
# save_unfull_chunk の指定なし (デフォルト: False)
```

Prefill で生成されるトークン数（40-50トークン程度）は `chunk_size: 256` より小さいため、「unfull chunk」（満たないチャンク）として扱われる。`save_unfull_chunk: False`（デフォルト）の場合、この unfull chunk は保存も取得もスキップされる。

### 変更

```yaml
# Decode の save_unfull_chunk を明示的に True
save_unfull_chunk: True
```

## まとめ

なんとか再現性のある形で PD 分離推論 with LMCache を紹介できました。今後、vLLM で PD 分離推論 with LMCache を構築する方のためにチェックリストを残します。

### vLLM 側

- [ ] `--kv-transfer-config` に `LMCacheConnectorV1` を指定
- [ ] `engine_id` を Prefill と Decode で統一
- [ ] `kv_role` を Prefill は `kv_producer`、Decode は `kv_consumer` に設定
- [ ] `--enforce-eager` を**使わない**
- [ ] GPU を Prefill と Decode で分離 (`NVIDIA_VISIBLE_DEVICES`)
- [ ] `PYTHONHASHSEED` を Prefill と Decode で統一

### LMCache 側

- [ ] `local_cpu: True` + `max_local_cpu_size > 0`
- [ ] `remote_url: "rediss://..."` (`s` が2つ)
- [ ] `remote_serde: "naive"` (CacheGen バグ回避)
- [ ] `save_unfull_chunk: True` (Prefill と Decode の**両方**)
- [ ] `chunk_size` を Prefill と Decode で統一
